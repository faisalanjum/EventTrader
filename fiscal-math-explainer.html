What's the Bug?

The Core Issue: Wrong Fiscal Year End (FYE) Month

build_guidance_period_id() converts fiscal labels (like "FY2025 Q1") to calendar date boundaries. It needs the company's FYE month. The FYE month is read from the 10-K filing's periodOfReport — but for 52-week calendar companies, this date can fall in the WRONG month.

Example: Five Below (FIVE) — fiscal year ends late January. But the 10-K's periodOfReport = 2025-02-01 (February 1, day ≤ 5). Raw month = 2 (February). Correct FYE = 1 (January). The fix: if day ≤ 5, use the previous month.

Why Does This Happen?

52-Week Fiscal Calendar

Many companies use a 52/53-week fiscal calendar instead of calendar months. Their fiscal year ends on "the Saturday nearest to January 31" — which lands on Jan 28, Jan 29, Feb 1, Feb 2, Feb 3, etc. depending on the year.

2023: periodOfReport = 2023-01-28 → month=1 ✓ 2024: periodOfReport = 2024-02-03 → month=2 ✗ (should be 1) 2025: periodOfReport = 2025-02-01 → month=2 ✗ (should be 1)
The Day ≤ 5 Rule

If the period ends on day 1-5 of a month, it's really the END of the previous month's fiscal period, not the start of a new one. The adjustment:

if periodOfReport.day <= 5: fye_month = periodOfReport.month - 1 # Feb→Jan else: fye_month = periodOfReport.month # Jan 28→Jan ✓

This rule is already in period_to_fiscal() for quarter assignment but is MISSING from get_derived_fye() and the guidance extraction path.

Not Just January → February

The bug is month-agnostic. Any 52-week company whose period crosses a month boundary with day ≤ 5 is affected:

The 3-Tier Cascade Fix

Each tier is tried in order. The first one that returns valid data is used. Higher tiers are more accurate but have less coverage.

PROVEN Tier 1 — v3 XBRL DEI-Anchored YTD Differencing

What it does: Reads exact fiscal quarter dates from XBRL Period nodes in the graph. Traverses the actual filing → XBRL → fact → context → period chain. Derives standalone quarters by differencing adjacent YTD endpoints.

Report → HAS_XBRL → XBRLNode ← REPORTS ← Fact(dei:DocumentFiscalYearFocus) → IN_CONTEXT → Context → HAS_PERIOD → Period Q1 = direct context period (Q1 filing's own period) Q2 = Q1.end → Q2_YTD.end (YTD differencing) Q3 = Q2_YTD.end → Q3_YTD.end Q4 = Q3_YTD.end → FY.end

SIMULATION Tier 2 — Previous Quarter End + Historical Length

What it does: When Tier 1 doesn't have the requested quarter (the 10-Q hasn't been filed yet), predict it from the previous quarter's exact end date + the company's historical quarter length pattern.

start = previous quarter's end_date (EXACT, from Tier 1) end = start + median(historical quarter length for this ticker + quarter number) Example: FIVE Q3 FY2025 Q2 end = 2024-08-04 (exact, from Tier 1 — Q2 10-Q filed months ago) FIVE Q3 historical length = 91 days (13 weeks, consistent across all FYs) Predicted Q3 end = 2024-08-04 + 91 = 2024-11-03 Actual Q3 end = 2024-11-03 ← EXACT MATCH

The 40-Day Gap: Why Transcripts & 8-Ks Can't Be Exact

The fundamental timing issue: earnings 8-Ks and transcripts fire 40 days before the 10-Q that contains the current quarter's XBRL data. At extraction time, exact dates for the current quarter don't exist anywhere.

The Unfixable Limit

For current and forward quarter guidance from transcripts/8-Ks: Exact dates are impossible at extraction time. The 10-Q for that quarter hasn't been filed — the data doesn't exist anywhere (not in our graph, not on SEC EDGAR, nowhere). Tier 2 predicts to ±0-1d for 98.4% of cases, but this is inherently a prediction, not a fact. The only way to get exact dates is to wait 40 days for the 10-Q.
