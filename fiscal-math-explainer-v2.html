What's the Bug?

The Core Issue: Wrong Fiscal Year End (FYE) Month

build_guidance_period_id() converts fiscal labels (like "FY2025 Q1") to calendar date boundaries. It needs the company's FYE month. The FYE month is derived from the 10-K filing's periodOfReport — but for 52-week calendar companies, this date can fall in the WRONG month.

Example: Five Below (FIVE) — fiscal year ends late January. But the 10-K's periodOfReport = 2025-02-01 (February 1, day ≤ 5). Raw month = 2 (February). Correct FYE = 1 (January).
v2 fix: SEC EDGAR's fiscalYearEnd field gives 0131 for FIVE — a single, fixed registration value. Apply day≤5 adjustment → month 1 (January). No mode voting, no latest-filing flip risk. Tested on 737/738 tickers (99.86%). And the SEC XBRL API gives exact quarter dates directly — bypassing the FYE math entirely.

Why Does This Happen?

52-Week Fiscal Calendar

Many companies use a 52/53-week fiscal calendar instead of calendar months. Their fiscal year ends on "the Saturday nearest to January 31" — which lands on Jan 28, Jan 29, Feb 1, Feb 2, etc. depending on the year.

2023: periodOfReport = 2023-01-28 → month=1 ✓ 2024: periodOfReport = 2024-02-03 → month=2 ✗ (should be 1) 2025: periodOfReport = 2025-02-01 → month=2 ✗ (should be 1)
v2: SEC EDGAR Bypasses the Problem

Instead of deriving the FYE from varying filing dates, we use two SEC sources:

// 1. SEC Submissions API — fixed FYE per company fiscalYearEnd: "0131" // FIVE = Jan 31, always // 2. SEC XBRL Company Concept API — exact per-quarter EarningsPerShareBasic: fy=2025 fp=Q1 start=2025-02-02 end=2025-05-03 fy=2025 fp=Q2 start=2025-05-04 end=2025-08-02 // No FYE math needed at all!

Not Just January → February

The bug is month-agnostic. Any 52-week company whose period crosses a month boundary with day ≤ 5 is affected:

The Fix (v2 — SEC EDGAR)

Each tier is tried in order. SEC XBRL data covers 100% of tickers for filed quarters — Tier 2 is only needed for current/forward quarters not yet filed. Tier 3 is a last resort for tickers without SEC CIK mapping (33 of 772).

PROVEN NEW in v2 Tier 1 — SEC EDGAR XBRL Company Concept API

What it does: One HTTP call per ticker to data.sec.gov/api/xbrl/companyconcept/. Returns every quarterly period with exact start/end dates and fiscal labels (fy, fp). No local XBRL processing. No graph traversal. No span anomalies.

GET data.sec.gov/api/xbrl/companyconcept/CIK{cik}/us-gaap/EarningsPerShareBasic.json Response includes per-fact period data: fy=2025 fp=Q1 start=2025-02-02 end=2025-05-03 (90d) form=10-Q fy=2025 fp=Q2 start=2025-05-04 end=2025-08-02 (90d) form=10-Q fy=2025 fp=Q3 start=2025-08-03 end=2025-11-01 (90d) form=10-Q No FYE derivation. No month-boundary math. Just exact dates from SEC.

The +1d end date difference is the XBRL inclusive vs exclusive convention — SEC facts use inclusive ends (last day of quarter), XBRL Period nodes use exclusive ends (first day of next quarter). Pick one and be consistent.

When it works: Any quarter where a 10-Q or 10-K has been filed with XBRL. SEC processes XBRL automatically — no dependency on our local pipeline.
When it doesn't cover: Current/forward quarters whose 10-Q hasn't been filed yet. Falls to Tier 2.
Key advantage over v1 Tier 1: No XBRL reprocessing prerequisite. No HAS_XBRL race condition. No 19 span anomalies from FY label collisions. SEC data is always up-to-date and free.
SIMULATION Tier 2 — Previous Quarter End + Historical Length

What it does: When Tier 1 doesn't have the requested quarter (10-Q not yet filed), predict it from the previous quarter's exact end date + the company's historical quarter length pattern. Now powered by SEC historical data (avg 112 quarters per ticker vs ~8 from local XBRL).

start = previous quarter's end_date (EXACT, from SEC Tier 1) end = start + median(historical quarter length for this ticker + quarter number) Example: FIVE Q3 FY2025 (8-K fires Nov 2024, Q3 10-Q not filed yet) Q2 end from SEC = 2025-08-02 (exact) FIVE Q3 median length from SEC = 90d (consistent across 15+ FYs) Predicted Q3 end = 2025-08-02 + 90 = 2025-10-31 Actual Q3 end (from later 10-Q) = 2025-11-01 ← off by 1 day

The 40-Day Gap: Why Current-Quarter Dates Are Predictions

The Remaining Limit

For current and forward quarter guidance from transcripts/8-Ks: The current quarter's exact end date isn't in the SEC XBRL API until the 10-Q is filed (~40 days later). Tier 2 predicts to ±0-1d for 98.4% of cases, but this is inherently a prediction. The only way to get exact dates is to wait for the 10-Q to be filed and processed by SEC.
