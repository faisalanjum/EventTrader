2 Who Does the Extraction?

Three layers: a routing skill, and two identical agent shells that differ only in which "working brief" they load.

extract/SKILL.md
Orchestrator Skill (runs in main Claude session)
Parses args → spawns agents → combines results
▼
Agent tool (foreground, wait)
extraction-primary-agent
Sub-agent (Opus, ultrathink, dontAsk)
Loads primary-pass.md as working brief
Extracts from PRIMARY source content
▼
Gate: do both files exist?
types/{TYPE}/enrichment-pass.md AND types/{TYPE}/assets/{ASSET}-enrichment.md
YES → spawn
extraction-enrichment-agent
Sub-agent (Opus, ultrathink, dontAsk)
Loads enrichment-pass.md as working brief
Extracts from SECONDARY content
NO → skip
Report primary-only results
Both agent shells are identical generic code. The only difference is slot #2: primary loads primary-pass.md, enrichment loads enrichment-pass.md. All specialization comes from the files they read.

5 The 3-Tier Query System

Queries are split across three files so that each tier is independently reusable. Every agent gets all three tiers loaded into its context.

Tier 1 — Common

queries-common.md (320 lines)
  • 1A-1D: Company/CIK/FYE/period resolution
  • 2A-2B: XBRL concept + member caches
  • 8A: Data inventory audit
  • 9A-9F: Fulltext / keyword search
Shared by every TYPE × ASSET combination

Tier 2 — Asset Queries

{ASSET}-queries.md
  • transcript: 3A-3G (PR, Q&A, fallbacks)
  • 8k: 4A-4G (items, exhibits, sections)
  • 10q: 5A-5E (MD&A, financials)
  • news: 6A-6E (content, batch, context)
"How to fetch data from this source type"

Tier 3 — Type Queries

{TYPE}-queries.md
  • guidance: 7A (existing tags)
  • guidance: 7B (latest per metric)
  • guidance: 7D (prior extractions)
  • guidance: 7E (full readback)
  • guidance: 7F (baseline frequency)
"How to look up existing items of this type"

10 Execution Timeline (Single Job)

What happens in order when trigger-extract.py AAPL --type guidance --asset transcript runs.

T0
trigger-extract.py queries Neo4j for AAPL transcripts with guidance_status=null
Sets guidance_status='in_progress', pushes JSON to Redis extract:pipeline
│
T1
extraction_worker.py BRPOPs job, spawns Claude Agent SDK with:
/extract AAPL transcript {SOURCE_ID} TYPE=guidance MODE=write SOURCE_TYPE=transcript
│
T2
SKILL.md parses args → spawns extraction-primary-agent via Agent tool
│
T3
Primary agent reads 8 files (slots 1-8) → executes primary-pass.md:
FETCH context → FETCH transcript PR → EXTRACT guidance items → VALIDATE via scripts → WRITE via guidance_writer.py
Writes result to /tmp/extract_pass_guidance_primary_{SRC}.json
│
T4
SKILL.md checks: do enrichment-pass.md AND transcript-enrichment.md exist? → YES
Spawns extraction-enrichment-agent
│
T5
Enrichment agent reads 8 files → executes enrichment-pass.md:
FETCH context → LOAD primary items (query 7E) → LOAD Q&A content → VERDICT each exchange → COMPLETENESS check (7F) → VALIDATE → WRITE changed/new only
│
T6
Worker reads result file → sets guidance_status='completed' on Transcript node in Neo4j

11 The Core Design Principle

Generic runtime (trigger → queue → worker → skill → 2 agent shells)
+
Specialized prompt packets (resolved at runtime via {TYPE} × {ASSET} path convention)
Zero code changes to add a type
Zero code changes to add an asset
Enrichment gated by file existence
Intersection rules are optional
