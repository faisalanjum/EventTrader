# Transcript Duplicate IDs — Root Cause Analysis

*2026-02-26 — Investigation only, no changes made. Root cause resolved 2026-02-26.*

---

## Problem

AAPL FY2025 Q3 has **two** Transcript nodes with different IDs pointing to the same earnings call:

| ID | Format |
|---|---|
| `AAPL_2025_3` | Legacy: `SYMBOL_YEAR_QUARTER` |
| `AAPL_2025-07-31T17.00.00-04.00` | Timestamp: `SYMBOL_DATETIME` |

Both have **identical** properties, content, and relationship graphs.

---

## Scope of the Problem

| Metric | Count |
|---|---|
| Total legacy-format transcripts (`SYMBOL_YEAR_QUARTER`) | **675** |
| Total timestamp-format transcripts (`SYMBOL_DATETIME`) | **3,722** |
| **Duplicates** (same symbol + fiscal year/quarter in BOTH formats) | **208** |
| **Legacy-only** (no timestamp counterpart — sole copy) | **469** |

| Most recent legacy-format created | Aug 2025 (e.g., `PINS_2025_2`, `TTD_2025_2`, etc.) |

**Legacy format is STILL being actively created.** This is not a historical artifact.

---

## Impact of 208 Duplicates

Each duplicate pair means:
- 2 Transcript nodes where 1 should exist (416 → 208)
- 2 PreparedRemark nodes with identical content
- ~40 QAExchange nodes with identical content (20 per transcript)
- 2 HAS_TRANSCRIPT edges from the same Company
- 2x4 INFLUENCES edges (Company, Sector, Industry, MarketIndex)
- **No Guidance nodes reference either** (checked — empty result)

Estimated waste: **~208 Transcript + 208 PreparedRemark + ~4,160 QAExchange = ~4,576 duplicate nodes** plus all their relationships.

## Root Cause Analysis

### How transcript IDs are generated

**Current code path** (scraper → Redis → Neo4j):

1. **Scraper** (`transcripts/EarningsCallTranscripts.py:228-242`): Creates transcript dict — **does NOT set an `id` field**
2. **Redis storage** (`store_transcript_in_redis`, line 747): Uses `RedisKeys.get_transcript_key_id(symbol, conference_datetime)` → produces timestamp format: `AAPL_2025-07-31T17.00.00-04.00`
3. **Neo4j ingestion** (`neograph/mixins/transcript.py:231`):
   ```python
   transcript_id = data.get("id") or key.split(":")[-1]
   ```
   - If JSON in Redis has `"id"` field → uses it (LEGACY format)
   - If no `"id"` field → falls back to Redis key suffix (TIMESTAMP format)

### The two paths

| Path | ID Source | Format | When Used |
|---|---|---|---|
| `data.get("id")` returns non-None | The `id` field in the stored JSON | `SYMBOL_YEAR_QUARTER` | When JSON data includes an `id` field |
| `key.split(":")[-1]` fallback | The Redis key, generated by `get_transcript_key_id()` | `SYMBOL_DATETIME` | When JSON data has no `id` field |

### Why both exist for the same transcript

The same earnings call gets ingested TWICE through different paths or at different times:
1. **First ingestion**: The JSON data has an `id` field set (possibly by an external process, API update, or pre-processing step) → `AAPL_2025_3`
2. **Second ingestion**: The JSON data does NOT have an `id` field → falls back to Redis key → `AAPL_2025-07-31T17.00.00-04.00`

### Resolved: Where does the `SYMBOL_YEAR_QUARTER` id originate?

**`redisDB/TranscriptProcessor.py:354`** — the `_standardize_content()` method explicitly sets it:

```python
standardized.update({
    'id': f"{content['symbol']}_{content['fiscal_year']}_{content['fiscal_quarter']}",
    ...
})
```

This is the TranscriptProcessor pipeline (BaseProcessor subclass). When transcripts flow through this path, the standardization step injects an `id` field in `SYMBOL_YEAR_QUARTER` format. That dict gets stored in Redis. When `transcript.py:231` later reads it, `data.get("id")` finds this field and uses it — producing the legacy format.

The scraper's `get_single_event()` (line 228-242) does NOT set `id`. But TranscriptProcessor's standardization step does. So the same transcript can produce different IDs depending on which code path processes it into Redis.

**Full chain**: scraper → `store_transcript_in_redis()` (key = timestamp format) → Redis JSON blob now contains `id` field (legacy format from TranscriptProcessor) → `transcript.py:231` reads `data.get("id")` → legacy format wins over key-derived timestamp format.

## Issues (No Changes Needed Yet — For Awareness)

### Issue 1: No deduplication guard in `merge_nodes`
The `_process_deduplicated_transcript()` method uses `self.manager.merge_nodes([transcript_node])` which MERGEs on `id`. Since the two formats produce different IDs for the same call, MERGE creates two separate nodes instead of updating one.

### Issue 2: `data.get("id") or key.split(":")[-1]` is fragile
The conditional ID source at transcript.py:231 means the same transcript data can produce different IDs depending on whether the JSON blob happens to contain an `id` field.

### Issue 3: 469 legacy-only transcripts cannot be safely deleted
These are the ONLY copy of those transcripts. They have valid data. Any cleanup must handle these as canonical entries.

### Issue 4: No uniqueness constraint on (symbol, fiscal_year, fiscal_quarter)
Neo4j has no constraint preventing two Transcript nodes with the same symbol+fiscal period but different IDs. The only uniqueness is on `id`, which is format-dependent.

### Issue 5: child node IDs are also duplicated
QAExchange and PreparedRemark IDs are derived from the transcript ID:
- `AAPL_2025_3_pr` vs `AAPL_2025-07-31T17.00.00-04.00_pr`
- `AAPL_2025_3_qa__0` vs `AAPL_2025-07-31T17.00.00-04.00_qa__0`

These contain identical content but different IDs, so both exist in the graph.

### Issue 6: Embeddings may be computed twice
If QAExchange embeddings are generated after ingestion, the 208 duplicate transcripts may have double the embedding storage and cost.

## Cleanup Considerations (Not Implemented)

### For the 208 duplicates:
1. Pick canonical format (timestamp is the current standard, 3,722 vs 675)
2. For each duplicate pair, verify content identity
3. Delete legacy node + all its child nodes + relationships
4. Re-point any references (Guidance, etc.) — currently none found

### For the 469 legacy-only transcripts:
1. These are NOT duplicates — they're the sole copy
2. Options: leave as-is, or re-key to timestamp format using `conference_datetime`
3. Re-keying requires updating all child node IDs and relationship references

### Prevention:
1. Normalize ID generation to a single function/format
2. Add a uniqueness constraint on `(symbol, fiscal_year, fiscal_quarter)` — but careful with non-standard fiscal calendars
3. Fix transcript.py:231 to always use the key-derived ID, never `data.get("id")`
