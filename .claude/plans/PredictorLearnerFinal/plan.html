G1 · Show-stopper

PIT fence — hook-enforced, env-marker-scoped. Final.md §1.1.2

Historical predictor runs in a forked Claude session that must be blind to anything beyond pit_cutoff. The fence is a project-level PreToolUse hook, gated by an environment marker the orchestrator sets only on the historical fork.

Mechanism

  • Hook config in .claude/settings.json PreToolUse block
  • Hook script: .claude/hooks/earnings_pit_fence.py
  • Env marker: EARNINGS_PIT_FENCE=1 (set only for historical fork, cleared on exit)
  • Marker present → hook DENIES listed tools; absent → pass through
  • Concurrency-safe per-process env scoping (proven empirically)

Denied in marked sessions

  • Bash, WebFetch, WebSearch
  • All mcp__* tools — class matcher mcp__.* + hook-side startswith("mcp__") (covers Neo4j, IBKR, any future MCP without enumeration)
  • Write/Edit unless tool_input.file_path ends in EXACTLY one of:
    • prediction/section_audit.json
    • prediction/result.json
    • prediction/reasoning.md
    • prediction/complete.json

Allowed

Read, Glob, and the path-scoped Write/Edit set above.

Empirical proof + coexistence with pit_gate.py
  • Validation: .claude/plans/Infrastructure.md — "Project PreToolUse hook fires INSIDE context: fork skills" (PROVEN 2026-05-19, v2.1.144). Concurrent marked/unmarked sessions, zero cross-contamination. Sessions X=0be4272b, Y=d2f75903.
  • Class matcher: proven across mcp__neo4j-cypher__* + mcp__yahoo-finance__* in one session (39140e04) — IBKR and any future MCP auto-covered.
  • Keep pit_gate.py: existing PostToolUse hook validates DataSubAgent tool outputs for PIT contamination (separate concern, separate event). The new PreToolUse hook does NOT replace it.
  • Path-scoped allowlist denies any learning/*, context_bundle*, prior-reports paths, result.md, and any other Write target. Hook-enforced, not prompt-only.

Live wrapper essentials §1.2.1 / §1.2.3

  • Live predictor embeds the SKILL body, then appends wrapper instructions, allowed research guidance, and SLA controls
  • Live wrapper minimum: PIT_MODE=live, report-release timestamp, SLA/env values, live allowed research set (§2.1), no runner-enforced tool allowlist, required output paths, fail-closed result.json rule
  • Env controls: LIVE_PREDICTOR_SLA_SECONDS (300), LIVE_PREDICTOR_NUDGE_INTERVAL_SECONDS (60), LIVE_PREDICTOR_GRACE_SECONDS (30), LIVE_PREDICTOR_SLA_MODE=hard|soft (hard)
  • Env home: set on launching process; orchestrator reads via os.environ; k8s manifests mirror extraction-worker env pattern
  • Clock sources: time_since_report_release uses release timestamp from trigger metadata (Redis if present, else Neo4j); elapsed_since_start_s uses predictor TMUX session spawn time
  • Nudge format: LIVE_TIME_UPDATE: time_since_report_release=<…>; price_change_since_report_release=<…>; elapsed_since_start_s=<…>; sla_seconds=<…> — sent by tmux send-keys text + Enter only; never Esc/Ctrl-C

  • Hard mode: missing/invalid result.json after SLA+grace = kill session, fail closed (no prediction, no learner input, no prior-report row); reasoning.md is non-blocking, result.json required
  • No timeout no_call fallback; timeouts are operational health only

Historical mode + common-to-both rules §1.1.3 / §1.3.2 / §1.3.3

  • Urgency inputs to both modes (§1.3.2): pass time_since_report_release and price_change_since_report_release. Historical uses PIT-safe values at launch; live receives queued LIVE_TIME_UPDATE nudges
  • Reasoning sidecar (§1.3.3): add REASONING_PATH=<component>/reasoning.md for historical prediction, live prediction, and learner. Audit-only: never canonical, never parsed for scoring/learning. Missing reasoning.md does NOT fail the run; result.json remains required
  • Predictor writes Phase 1-4 working analysis to REASONING_PATH BEFORE RESULT_PATH (key numbers/surprise math, what's-new/priced-in/material/counter-case, bull vs bear, final rationale, with bundle evidence)

  • Skill rule edits: predictor checklist rule 8 may write SECTION_AUDIT_PATH + REASONING_PATH + RESULT_PATH; learner must relax both single-file guards (Critical Rule 6 + step "ONLY file" rule) to allow REASONING_PATH + RESULT_PATH

Schema

prediction_result.v1 — what the LLM authors. Final.md §7

Canonical artifact: prediction/result.json. Schema: prediction_result.v1. Drop lesson_labels, cites_lesson_indices, and the entire lesson-label plumbing.

LLM-authored fields

  • direction: enum long | short | no_call
  • confidence_score
  • expected_move_range_pct [low, high] (unsigned magnitudes; sign from direction)
  • key_drivers[]: {driver, direction: long|short, evidence:[SRC ids]}
  • data_gaps[], evidence_ledger[], analysis
  • falsifier — single primary observable thesis-breaker; required for directional, null for no_call
  • opened_prior_reports[]: list[str] of report_id values matching §6 prior_report_row.report_id

Shape locks

  • evidence_ledger[] shape: {metric, value, source, source_id} (same shape reused by learner per §8)
  • no_call has two valid paths: Balanced = ≥1 grounded long + ≥1 grounded short driver; Missing-data = data_gaps[] non-empty + analysis explains
  • Invalid directional calls fail validation/retry — validator must NOT silently convert to no_call
Orchestrator-stamped fields + scoring relationship
  • Orchestrator-stamped: schema_version, ticker, quarter_label, predicted_at, model_version, prompt_version, sdk_session_id (nullable), pit_mode, prediction_delay_sec (live-only), confidence_bucket, magnitude_bucket
  • No LLM point estimate: scorer derives midpoint from expected_move_range_pct
  • analysis = concise canonical thesis: bull case, bear case, final rationale
Schema

learner_result.v1 — causal attribution, not lessons. Final.md §8

Canonical artifact: learning/result.json. The learner writes causal attribution, not managed lesson-lifecycle state.

LLM-authored fields

  • evidence_ledger[], primary_driver, contributing_factors[]
  • feedback{prediction_vs_actual, why, what_worked, what_failed}
  • key_takeaway — exactly one descriptive sentence for prior-report rows
  • future_checklist[] — prediction-time-checkable questions only
  • missing_inputs[], data_sources_used[]

Shape rules

  • primary_driver: {driver, evidence:[evidence_ledger.source_id]}
  • contributing_factors[]: {factor, evidence:[evidence_ledger.source_id]}
  • feedback.what_worked[] / what_failed[]: plain strings (cap ≤2 / ≤3)
  • feedback.prediction_vs_actual + feedback.why: plain LLM-authored prose strings; no scorer-owned numbers
  • future_checklist[]: list of plain strings; no fixed numeric cap
  • Internal grounding: every primary_driver / contributing_factors evidence ref must resolve in learner's own evidence_ledger[] — no source → no claim

Scorer-owned (NOT learner-authored)

Graded numbers such as direction_correct, magnitude_error, MSE, Brier; plus derived fields such as actual_direction (per §18).

Orchestrator-stamped + recovery
  • Orchestrator-stamped: schema_version, ticker, quarter_label, filed_8k, accession_8k, attributed_at, model_version, sdk_session_id (nullable), pit_mode, pit_cutoff, pit_boundary_source, pit_boundary_event, actual_return_pct, context_bundle_ref, prediction_result_ref
  • Recovery: existing valid target-schema learning/result.json is sufficient (post-cutover ongoing ops; §23 wipe is one-time)

Visibility + ordering

  • Visible: up to 12 own-ticker + up to 4 peer learner reports
  • Forced opens: latest 4 own + ≤2 most-recent own misses (independent set; doesn't reorder own_reports[])
  • Ordering: own_reports[] recency-desc; peer_reports[] mkt_cap-desc; forced misses may appear anywhere in recency-desc list
  • Peer universe: same Neo4j Industry via BELONGS_TO, mkt_cap-ranked; never LLM-selected, never widened to sector (except §21 D4 ≤2-company fallback approved only for §20)
  • Optional opens after forced: older own + up to 3 peer reports by predictor's own judgment of summary-row relevance
  • PIT selection: include only when learner_result.pit_cutoff <= predictor.pit_cutoff; always exclude the current event's own learner report

Per-row bullet rendering (§6) — Section 10 in the bundle

Render Section 10 as a bulleted list per report, NOT a table. Each report bullet shows:

  • Ticker + Quarter + Date
  • predicted_direction vs actual_direction + direction_correct/miss + actual_return_pct
  • Quoted key_takeaway from learner result
  • Link to allowlisted learning/result.md

Do NOT render tags, lesson labels, or audit history. Machine source is learning/result.json; result.md is read-only prose for the predictor — if missing, regenerate from JSON, never parse markdown as source truth.

Miss definition (excludes no_call)

Miss = directional call (long or short) with direction_correct == false. no_call is NOT a miss. Miss-priority forced opens are own-ticker only — peer reports never enter the miss-priority set.

Validator + allowlist contract
  • Validator enforces: every opened_prior_reports[i] ID maps to a row whose result_md_path is in _allowed_prior_report_paths; every forced_open_ids[i] in bundle MUST appear in opened_prior_reports[]
  • Peer opens ≤3, no duplicates, no current-event self-leak
  • _allowed_prior_report_paths is per §17 rename from _allowed_learner_paths
G6 · Show-stopper

Section audit + predictor methodology. Final.md §16

Bundle exposes a top-level allowlist of section names that ARE auditable. SKILL audits exactly that set — never hardcoded section numbers.

Section audit scope (G6)

  • Bundle exposes top-level section_audit_scope[] containing current-event factual section names
  • SKILL audits exactly that set, NOT hard-coded section numbers
  • Excludes prior_reports_context / checklist memory and any future memory-class section
  • Orchestrator/builder populates at bundle-assembly time

Predictor analytical spine (§16)

  1. Read the full bundle
  2. Write section audit scoped per section_audit_scope[]
  3. Open/read §6 prior reports per forced/optional rules; engage relevant §12 checklist questions; record opened_prior_reports[]
  4. Extract key numbers + surprise quality
  5. Answer existing Phase-2 questions (preserved verbatim from current SKILL)
  6. Stress-test long vs short
  7. Make the call (or no_call)

Market-grading lens (bounded loop)

  • Form a candidate lens → test against allowed bundle/live signals, prior own reports, peer reactions, analyst/management emphasis, price setup, surprise math
  • Revise until the lens stops changing materially, or live SLA forces a call
  • Then rank signals against the lens
  • Derive the lens per print; NEVER hardcode metric maps or examples

Stress-test elements

What's new · what's priced in · what's material · strongest counter-case · top drivers.

If evidence is thin, conflicting, or missing → lower confidence, widen range, use data_gaps, or choose no_call. Never invent a sector metric.

Mode + source discipline

  • Historical: must NOT use post-cutoff transcript/Q&A or other hindsight evidence
  • Live: may use live evidence within §1.2 / §2.1 urgency rules
  • Current-bundle peer reactions MAY inform the lens; prior learner reports provide context within §6/§9 only and NEVER replace current-event evidence
  • Output only through §7 schema + §1.3.3 reasoning sidecar — no new predictor fields or semantic validators
Why the data-driven section audit matters

The renderer's section numbering changes when sections are added/removed/reordered. Hardcoding "audit §2-§9" in the SKILL prompt would silently break the moment the bundle structure shifts. The allowlist approach is rename-resilient: when a new factual section is added to the bundle, the orchestrator includes its name in section_audit_scope[]; the SKILL automatically picks it up. Memory-class sections (Prior Reports, Standing Checklist) are absent from the list by design.

Methodology

Standing checklist — one rolling list, rewritten whole. Final.md §12

The learner's future_checklist[] is the one rolling own-ticker checklist; it lives only inside the latest eligible own learner result. The orchestrator surfaces it forward into the next predictor's bundle.

Loop contract

  • Stored only inside the latest eligible own learner result — orchestrator surfaces it forward
  • Learner rewrites future_checklist[] WHOLE each cycle, consolidating from the same PIT-visible own-ticker prior-report window §6/§11 already surfaces
  • Predictor engages relevant items inside analysis prose; dismisses irrelevant items in one line
  • No separate store, IDs, lineage, source links, scores, labels, keep/drop state, checklist_review[]
  • No peer/industry merging, no stored cross-report synthesis, no per-company "what works" profile

Anti-forgetting + length policy

  • Anti-forgetting: keep durable recurring-conditional questions even when the condition is absent this quarter; remove only items that are generic, stale, redundant, or one-off — NOT items whose recurring condition was simply absent this cycle
  • Length policy: no fixed numeric cap; interrogative questions only (not directives or answers); ticker-specific; concise; prediction-time-checkable from the predictor's allowed bundle
  • Never substitutes for §10's current-event/current-bundle evidence requirement — checklist items focus attention only
D8 output rule + learner edge-mining funnel
  • D8 output rule: a surviving edge enters future_checklist[] only as a prediction-time-checkable question answerable from the next predictor's allowed bundle. Post-event-only insight with no predictor-visible proxy is omitted, or kept as a non-evidence company-history caution — never as evidence.
  • Edge-mining funnel: (1) isolate from §15 the part the predictor got wrong; (2) knowable-test (was a signal derivable from info the predictor was allowed to use?); (3) attack the survivor (would it have changed/strengthened the call? repeatable? answerable next time?)
Methodology

Whole-move attribution — explain the move, not just the earnings story. Final.md §15

The learner must explain the entire stock move. Cause families are prompt lenses only — never schema enums or fields.

Cause lenses (prompt-only)

Fundamentals · guidance · one-time/cosmetic · positioning/expectations · macro/sector · peer read-through · analyst/news · flow/technical · unexplained residual.

Attribution loop (bounded)

  • Hypothesize drivers → test whether they explain the move's sign and magnitude
  • Return to evidence for contributors/offsets when a gap remains
  • Stop when the explanation stops changing materially
  • Only the genuine sourced remainder becomes unexplained residual (in feedback.why)
  • Never invent a cause or residual percentage
  • Residual → future_checklist only when the next predictor's bundle will contain data that can detect the same cause (§12 D8); otherwise leave as residual

Falsifier check

Explicitly verify whether the prediction's stated falsifier (§7) was triggered by the realized move. Record the answer in feedback.what_failed[] (fired) or feedback.why (did not fire or not applicable). No new schema field.

Workflow ordering (§12)

Read prediction, bundle, prior reports + carried standing checklist, and actual return; scan the bundle to fix the observability boundary (what the predictor could see); fetch post-event evidence; explain the whole stock move (§15); compare prediction thesis and falsifier vs actual outcome; write attribution, key_takeaway, and rewrite the rolling future_checklist.

Rule

Evidence grounding — anchor every call. Final.md §10 / §9

Directional calls require at least one supporting current-event / current-bundle evidence ID. Prior reports, checklist items, falsifier, and data_gaps never count.

Historical mode

  • Every evidence_ledger[].source_id must resolve in the PIT-safe bundle catalog
  • Every key_drivers[].evidence[] ID must resolve in the same catalog
  • Key driver prose stays free-form — no semantic validator
  • No numeric evidence floor beyond the current-event anchor

Live mode (currently weaker)

Defer SRC:LIVE:* runtime catalog. Until built, live-only facts in evidence_ledger[] or key_drivers[].evidence[] may carry provider/source detail but do not get strict runtime-ID validation. Per §21 D13, design a minimal grounding mechanism without restricting what live can fetch.

Future numeric floor + learner asymmetry
  • If a numeric floor is later added: count unit = distinct resolving SRC IDs across supporting key_drivers
  • Learner grounding is INTERNAL: every primary_driver/contributing_factors evidence ref must resolve in the learner's own evidence_ledger[] (which may include post-event sources). §10's bundle-catalog grounding is predictor-only.
Boundaries

Gap / failure fields are disjoint. Sub-agents fan out. Final.md §13 / §14

Three semantically distinct fields hold the three failure-modes; never copy across them. Live tools may run in parallel under TMUX with no batching trick required.

Predictor data_gaps[]

Material pre-outcome inputs unavailable to predictor. Shape [{gap}]. Required only for missing-data no_call. Never evidence.

Learner missing_inputs[]

Post-outcome evidence unavailable to learner. Shape [{input}]. NOT copied from predictor data_gaps. Never evidence.

Learner feedback.what_failed[]

Available-to-predictor information that predictor ignored, misread, or under/over-weighted. Cap ≤3. Evidence-grounded via learner ledger refs.

These three fields are disjoint.

DataSubAgent / Parallelism (§13)

  • DataSubAgents require top-level embed/TMUX context — forked predictor skill cannot use them
  • Spawn independent, self-contained tasks without waiting; one spawn per turn (~1s apart) is expected — still runs concurrently
  • Single-message batching is NOT the lever and is not achievable by instruction
  • Live allowed research set: existing DataSubAgent families + live direct tools when needed (WebFetch/WebSearch, Neo4j read, IBKR market data)
  • Live tools are read/query/fetch only; never mutate Neo4j, orders, or disk outside explicit output paths
  • No numeric live tool-call cap; the live SLA is the hard governor
  • Live mode: include a startup smoke check before DataSubAgent-dependent runs (per tmux-transport-handoff.md)

  • Trigger: just after learning/complete.json is written for that event (analogous to learner's just-in-time trigger per §22)
  • Inputs: prediction/result.json + learning/result.json::actual_return_pct + scorer version/config
  • Idempotency: byte-identical output for same inputs. If actual_return_pct is later corrected upstream (data revision), scorer recomputes — overwrites are safe.

Artifacts

  • Per-event: earnings-analysis/Companies/<T>/events/<Q>/scoring/result.json
  • Rolling aggregate: earnings-analysis/scoring/ev1_aggregate.json — recomputed deterministically from per-event scores
  • Writes are atomic and overwrite-safe; outputs are code-owned, never LLM-authored

Math

  • Direction score: compute direction_correct from predicted direction vs signed actual return
  • actual_direction derived: long if return >0, short if <0, flat if =0
  • direction_correct is null when direction == no_call; otherwise (direction == actual_direction)
  • Magnitude: signed predicted midpoint = sign(direction) × midpoint(expected_move_range_pct); compare to signed actual return for signed error + MSE
  • Calibration: Brier from raw confidence_score vs direction_correct
Aggregates + replay set
  • Aggregates: hit rate by direction, mean signed error/bias, MSE, Brier, calibration by bucket, ticker, quarter, replay suite
  • Buckets are grouping/display only — all math uses raw values, never bucket labels
  • no_call tracked separately from directional calls — operational timeouts excluded from accuracy/calibration/MSE/Brier
  • Use a frozen representative + edge replay set; do not re-pick after seeing results
  • Trade P&L / executioner outcomes are downstream analytics only — never flow back into predictor-readable context

Cross-cutting

Rate-limit gate — centralized + auto-failover. Final.md §5.1

One shared usage-gate module for every Claude-consuming process. Dedicated subscription with full capacity by default; tiered fallback only when the dedicated subscription is exhausted.

Centralize

  • ONE shared module (e.g., scripts/claude_rate_gate.py) used by guidance extraction, predictor, learner, earnings-worker, future workers
  • Trigger daemons enqueue only; the process about to start a Claude session (worker OR orchestrator) must call the shared gate

Subscription policy

  • Dedicated Claude subscription allocated to the project's automated jobs, separate from personal/interactive Claude use
  • Default for the dedicated subscription = FULL capacity (no daily reserve / no days-until-reset throttling)

Failover policy

  • If the dedicated subscription is exhausted, the shared gate automatically switches to another configured Claude subscription/account (no manual intervention)
  • On fallback, apply reserve/tier throttling to protect fallback capacity

Account profiles

  • Each configured Claude subscription/account is an isolated Claude home containing its own .claude/.credentials.json
  • The shared gate checks profile-scoped usage cache and launches both claude_usage_fetch.py and TMUX Claude sessions with HOME=<profile_home>
  • Never mutate or symlink global ~/.claude/.credentials.json at runtime; switching is per-process env only

Priority under constraint (tier order, highest → lowest)

  1. Live prediction / current-event blockers
  2. Just-in-time learner blocking the next prediction
  3. Historical prediction/learning catch-up
  4. Bulk guidance/backfill/future agents

Lower tiers pause first.

Gate mechanics
  • Shared gate owns: usage cache refresh, pre-flight wait/sleep, in-flight rate-limit detection
  • Jobs paused or requeued for rate limits do NOT count as failures and do NOT burn retry budget
  • Implementation: refactor scripts/extraction_worker.py to use the shared module; new predictor/learner/earnings workers MUST import it (no duplication)
  • Under the dedicated subscription, set DAILY_INTERACTIVE_PCT=0 and DAILY_INTERACTIVE_PCT_SONNET=0
