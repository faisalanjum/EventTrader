The goal

Read a filing like a person — then save the facts like a robot.

Companies bury their forward-looking numbers inside long documents. We want to pull those numbers out automatically, clean them up so every company looks the same, and store them in the graph so software can compare and trade on them.

The analogy

Imagine a tireless intern who reads every earnings call, press release, and SEC filing. For each one they fill in the same index card:

  • What metric? (Revenue) · How much? ($94–98B) · When? (next quarter) · On what basis? (non-GAAP) · Proof? (the exact quote).

The pipeline is that intern — but it never tires, never guesses a number, and files every card with a barcode so the same fact is never filed twice.

Why bother?

Once every company's guidance lives in one shape, you can ask: "who just cut their outlook?", "whose revenue guide is above consensus?" — across the whole market, instantly.

Why it's hard

The same number can be said five ways ("$1.13 billion", "1,130 million", "$1.13B"), tied to a fiscal quarter that isn't a calendar quarter, on GAAP or adjusted basis. Cleaning that up consistently is the whole game.

Guidance is the one fact-type built so far. The machine underneath is generic — it was designed so a future "analyst estimates" or "announcements" type drops in with no new plumbing.

The spine

One rule explains the whole system.

The AI reads and understands. Plain code computes IDs, units, dates, and does every database write. The AI never touches the database directly.

🧠 The AI (reader bots)

Anything needing understanding:

  • "Is this a real forward-looking number?"
  • "What metric / period / basis is this?"
  • "Copy the exact proof quote."

⚙️ The code (Python)

Anything mechanical:

  • Turn "$1.13B" into 1130 in millions.
  • Build the barcode ID, compute the date window.
  • Write to Neo4j and refuse anything malformed.

Why this split is the whole trick

LLMs are brilliant at reading but wobbly at arithmetic and consistency. If the AI built IDs or scaled numbers, two runs could disagree and you'd get duplicate or corrupt facts. So the AI only ever hands code a clean list of proposed facts; code does the exact, repeatable parts. The reader bots literally don't have the database "write" tool.

evidence-first No quote = no fact. recall over precision When unsure whether something is guidance, extract it — a spare fact is cheaper than a missed one.

Big picture

One job flows through five stages — every time.

A "job" is one tiny unit: one fact-type, one document, one company. It rides a queue to a worker, which runs an AI that reads the doc, and finishes with deterministic code writing the facts to Neo4j.

trigger ──→ Redis queue ──→ KEDA scales 1–7 worker pods ──→ extraction_worker.py (what to do) (extract:pipeline) (more jobs = more pods) (one job at a time) │ v Claude Agent SDK (runs the AI; limits on slide 45) │ v /extract (the orchestrator skill) ├─→ extraction-primary-agent (reads main text) └─→ extraction-enrichment-agent (only transcripts: Q&A) │ v guidance_write.sh ─→ guidance_write_cli.py ─→ guidance_writer.py │ v Neo4j: MERGE Guidance / GuidanceUpdate / GuidancePeriod

The generic half (never changes)

Trigger, queue, worker, orchestrator, the two bot shells. They don't know or care what they're extracting.

The specialized half (per type)

The instruction files + the writer scripts. Today only guidance exists. Add a folder and you've added a type.

We'll walk the generic half first (slides 6–12), then guidance in depth (13–36), then how this ran live (37–48).

The machine · part 1

Everything is decided by three independent dials.

Adding capability never means rewriting the engine — you just turn a dial. The three dials are Type × Asset × Pass.

TYPE what

What to pull out. Live: guidance. Designed-for: estimates, announcements, anything.

ASSET where

Which document to read: transcript · 8k · 10q · 10k · news.

PASS phase

A reading phase: primary (always) + optional enrichment (a 2nd look).

The grid this creates

1 type × 5 assets means 5 jobs you can run today — guidance from a transcript, guidance from an 8-K, and so on. Each combination gets its own tiny rule-sheet, so an 8-K's quirks never leak into how transcripts are read.

transcript 8k 10q 10k news guidance ✔(+Q&A) ✔ ✔ ✔ ✔ ← the 5 live jobs (estimates) … … … … … ← a future type: same grid, zero new plumbing
Why three dials and not two?

Two (type × asset) couldn't express "read the prepared remarks, then separately read the Q&A." That second look needs different rules from the first. So pass is its own dial — but it belongs under a type (each type defines its own passes). Today only guidance × transcript uses a second pass.

The machine · part 2

Before reading, each bot loads an 8-file "backpack."

A reader bot is a blank shell. What makes it a "guidance-from-an-8-K reader" is the exact 8 files it loads first — chosen automatically by the Type/Asset/Pass dials.

#FileWhat it gives the bot
1core-contract.mdthe schema bible (fields, IDs, rules)
2{pass}.mdthe working brief — do this, start to finish
3{asset}.mdhow this document is shaped in the database
4{asset}-{pass}.md ◄the key trick — what to extract from THIS asset
5queries-common.mdshared database queries
6{asset}-queries.mdqueries to fetch this document's text
7{type}-queries.mdtype-specific lookups (e.g. existing guidance)
8evidence-standards.md4 anti-hallucination rules

Slot 4 is the load-bearing wall

The "intersection file" holds rules like "in an 8-K, the press-release exhibit usually holds the guidance." It keeps type rules out of asset files. Without it, the 8-K profile would fill up with guidance-only rules and break the day you add a second type. Slot 4 is the reason the machine stays generic.

If a slot-4 file doesn't exist for a combo, it's silently skipped — that combo just has no special rules.

The machine · part 3

The orchestrator is a 3-step conductor.

/extract (the SKILL.md) doesn't read documents itself. It runs the reader bots in order and collects their results. Input: TICKER ASSET SOURCE_ID TYPE= MODE=.

1 · Primaryspawn primary bot, wait, read its result file
→
2 · Enrichment?only if both enrichment files exist → spawn 2nd bot
→
3 · Reportcombine counts, write result, clean up /tmp

The enrichment gate is just "do the files exist?"

Step 2 runs the second bot only if both enrichment-pass.md AND {asset}-enrichment.md are on disk. Today that's true for exactly one combo: guidance × transcript (to mine the Q&A). Everything else is single-pass. Want to add a second pass somewhere? Add the two files — no code change.

How the orchestrator picks the model per bot

When the worker launches /extract, it passes PRIMARY_MODEL= and ENRICHMENT_MODEL=. The orchestrator passes those straight to the Agent tool, overriding the bot's default. So one job can run the primary bot on Haiku and the enrichment bot on Sonnet — decided by config.yaml (slide 49). If no model is passed (a manual run), the bot's own default applies.

The machine · part 4

Two reader bots — same shell, different briefs.

Both are generic shells. They load their 8-file backpack, follow the brief, and write a small result file. Neither can write to Neo4j — they only read it and call the writer script.

🔵 extraction-primary-agent

Reads the main content (prepared remarks, the press release, the MD&A). FETCH → EXTRACT → VALIDATE → WRITE. This is where most facts come from.

🟣 extraction-enrichment-agent

Reads the secondary content (the Q&A). For each item: ENRICHES an existing fact, is a NEW fact, or NO GUIDANCE. Writes only what changed.

Three non-negotiable guardrails (in both bots)

  • Never use the Neo4j write tool — all writes go through the Python writer via Bash.
  • Never compute IDs, scale units, or build dates by hand — call the scripts.
  • Follow the brief (primary-pass.md / enrichment-pass.md) start to finish.

Tools each bot has: read-only Neo4j, Bash, Read/Write (for /tmp files), and the task tools. That's it. Default model: Sonnet.

The machine · part 5

Adding capability = adding files, (almost) never code.

The folder layout is the configuration. The worker scans the types/ folder on disk to discover what exists — so a new type needs zero changes to the engine.

➕ New TYPE (e.g. estimates)

Drop a folder types/analyst_estimates/ with a contract, a pass brief, queries, and per-asset rule files. Plus Python writer scripts. Zero runtime-code edits — it's auto-discovered.

➕ New ASSET (e.g. a proxy filing)

Add two files (proxy.md + proxy-queries.md) and register it in 2 Python dicts (the trigger's ASSET_QUERIES + the worker's ASSET_LABELS). Assets cost a tiny code edit; types don't.

The rule that keeps it clean: no contagion

Generic files (asset profiles, shared queries) contain zero guidance-specific words. All guidance rules live in the types/guidance/ folder. That's why a future type can reuse the exact same 8-K reader without inheriting guidance's habits.

Naming rule: asset names have no hyphens (the -{pass} suffix uses the hyphen as a separator). Use 10q, not 10-q.

The machine · part 6

Two safety switches must BOTH be on to write.

By default the pipeline practices — it computes every fact and ID but saves nothing. Writing to the real graph takes two independent "yes" votes.

MODE=dry_run default
Read, extract, validate, build IDs. Print results. No write.
MODE=shadow
Same as dry_run, but also logs the exact Cypher it would run.
MODE=write
Real write — but only if the second switch is also on.

The two-layer write gate

MODE = write AND ENABLE_GUIDANCE_WRITES = true → REAL WRITE (per job) (global env flag, default false)

Both must be permissive. MODE=write with the flag off still writes nothing. This is why a stray run can't accidentally mutate the graph.

Two footguns to know

  • The mode check is literal: the code is dry_run = (mode != '--write'). Any argument that isn't exactly --write silently becomes a dry-run.
  • The flag is read once at import: changing ENABLE_GUIDANCE_WRITES mid-process has no effect — it's frozen when the script loads.
The machine · part 7

Bots talk through files, not chat — and stamp each document.

Big AI outputs are unreliable to parse from a chat stream, so each bot writes its result to a small JSON file in /tmp, and the orchestrator reads it back.

primary bot → /tmp/extract_pass_guidance_primary_{SOURCE_ID}.json {status, items_extracted, items_written} enrichment → /tmp/extract_pass_guidance_enrichment_{SOURCE_ID}.json {status, items_enriched, new_secondary_items} the facts → /tmp/gu_{TICKER}_{SOURCE_ID}.json (the payload handed to the writer)

Status lives on the document

After a job, the worker sets guidance_status on the source node: in_progress → completed | failed. Each type gets its own status field, so guidance and a future type don't collide.

Why file-based?

A transcript can yield a 50KB+ result. Passing that through the model's text output corrupts easily. A file is robust — and lets the worker count exactly what was written.

Six error codes, one acceptable "failure"

SOURCE_NOT_FOUND · EMPTY_CONTENT · QUERY_FAILED · NO_GUIDANCE · WRITE_FAILED · VALIDATION_FAILED. Of these, NO_GUIDANCE is a success — plenty of documents simply contain no forward-looking numbers, and saying so is the correct answer.

Guidance · 1 of 24

Guidance = the company telling you its own future numbers.

Not analyst opinions. Not last quarter's results. Forward-looking statements the company itself makes about a period that hasn't happened yet.

✅ Is guidance

  • "We expect Q2 revenue of $94–98B."
  • "Gross margin around 47%."
  • "Services to grow double digits."
  • "CapEx of approximately $2B."
  • "We're withdrawing full-year guidance." (still a guidance event)

❌ Is NOT guidance

  • "Q1 revenue was $124B." (past = actuals)
  • "Analysts expect $3.50." (not the company)
  • "We see strength in iPhone." (no number)
  • "$10B buyback authorized." (an announcement, different type)

The two filters that draw the line

Forward-looking: the target period must be after the document's date. Specific: there must be a quantitative anchor — a number, a range, or a phrase like "low double digits". Vague words ("strong", "significant") with no magnitude are dropped. Dividend-per-share guidance counts; buyback/investment announcements belong to a separate (unbuilt) type. One exception to "must have a number": withdrawing or reaffirming guidance is itself a guidance event — the action is the signal, so it's kept even with no number.

Guidance · 2 of 24 · the schema

Three node types: a label, a data-point, and a time-window.

Everything guidance writes is one of just three shapes. Get these and the graph clicks.

1 · Guidance label

The reusable metric name, e.g. guidance:revenue. Created once. Just label + aliases.

2 · GuidanceUpdate data-point

One fact: this metric, this number, this period, from this document. 20 fields. Many pile up under one label.

3 · GuidancePeriod window

The calendar window the number is about, e.g. gp_2025-04-01_2025-06-30. Shared across companies.

The thermometer analogy

Guidance is a blank thermometer labelled "Revenue." A GuidanceUpdate is one reading taken from one filing ("$94–98B, expected, non-GAAP"). A GuidancePeriod is the dated window that reading is about ("next quarter"). Three separate things — never mashed together.

Two node types that an older design had were removed: no Context node (a direct FOR_COMPANY link replaces it) and no Unit node (the unit is just a property). Simpler graph, same information.

Guidance · 3 of 24 · the schema

Six links wire each fact into the graph.

A GuidanceUpdate always has the first four links. The last two are best-effort — they attach only on a confident match.

LinkFrom → ToWhen
UPDATESGuidanceUpdate → Guidancealways
FROM_SOURCEGuidanceUpdate → Report / Transcript / Newsalways (the proof)
FOR_COMPANYGuidanceUpdate → Companyalways
HAS_PERIODGuidanceUpdate → GuidancePeriodalways (exactly 1)

( Guidance: "Revenue" ) ^ | UPDATES FROM_SOURCE ( GuidanceUpdate ) HAS_PERIOD MAPS_TO_CONCEPT (XBRL) ┌──────────────┘ │ │ └──────────────→ ( GuidancePeriod ) ┌─→ ( Concept ) v | └─ FOR_COMPANY → ( Company ) | ( 8-K / Transcript / News ) ( GuidanceUpdate )──┘ MAPS_TO_MEMBER → ( Member: "iPhone" )

The last two links connect a fact to the company's official accounting concepts (XBRL = the standard tags companies file numbers under) and business segments — they "wire it to the books." If no good match, the link is simply skipped; the fact is still written. (Full detail on slides 28–29.)

Don't memorize — just see the four buckets. The next slides zoom into the tricky ones (period, units, derivation, basis, segment).

📊 The number

The iron rule

No quote = no fact. Every GuidanceUpdate must have a quote, a FROM_SOURCE link, and a given_date. If a field is genuinely unclear, leave it null — never invent it.

Plus system fields code adds at write time: the id (slide 17), the evhash16 (slide 18), label_slug/segment_slug (for fast filtering), and the V2 resolution fields (slide 24).

Guidance · 5 of 24 · identity

Every fact gets a barcode built from 5 ingredients.

The ID is the fact's slot — it answers "which guidance item is this?", never "what does it say?". No number lives in the ID. That's the secret to safe re-runs.

guidance_update_id = "gu:" + source_id + ":" + label_slug + ":" + period
                          + ":" + basis_norm + ":" + segment_slug

// worked example — Apple's next-quarter non-GAAP revenue from one transcript:
gu:AAPL_2025-01-30T17.00:revenue:gp_2025-04-01_2025-06-30:non_gaap:total

Why no value in the ID?

The ID is "Apple's Q3-2025 non-GAAP total revenue guidance, from this transcript." If Apple revises that number, it's the same slot → same ID → the row just updates. If the ID contained the value, a revision would make a confusing duplicate.

Same slot, same ID

Same document + metric + period + basis + segment → one row. Re-run safely overwrites it.

Different slot, different ID

Change the segment (iPhone vs Total), the basis (GAAP vs non-GAAP), or the period → a separate, correct row. No false duplicates.

Fine print: how the barcode string is built

"Barcode" just means the slot ID — it names the row, never the value. slug() = lowercase + turn spaces/punctuation into _. The Guidance label node's ID is simply guidance:{slug(label)}. A source_id can itself contain colons (e.g. a timestamp like 17.00.00-04.00) — those are swapped to _ first, so they can't break the five-part delimiter.

Guidance · 6 of 24 · identity

A separate fingerprint tracks when the VALUE changes.

The ID is the slot. The evhash16 is a fingerprint of the actual contents — so you can tell "did this number change?" without it ever affecting the ID.

evhash16 = sha256("{low}|{mid}|{high}|{unit}|{qualitative}|{conditions}")[:16]

// it is a stored PROPERTY, never part of the id
low=94000, mid=96000, high=98000, unit=m_usd   // AFTER billions → millions
   → "94000|96000|98000|m_usd|.|."  → sha256  → evhash16 = "a3f1c9d2e0b7..."

What it's for

Two documents (say an 8-K and the transcript) can carry the same slot. Comparing their evhash16 tells you instantly whether they agree or whether one revised the number — a change detector that's totally independent of identity. Nulls are encoded as ., numbers are canonicalized first, text is lowercased and trimmed.

Per-share metrics (EPS) are normalized to dollars and aggregates to millions before hashing, so "$1.13B" and "1,130 million" produce the same fingerprint.

Guidance · 7 of 24 · identity

Re-running is always safe — by construction.

Every write is a Neo4j MERGE on the barcode ID: "create if new, update if it exists." So you can re-extract the same document a hundred times and never get duplicates.

first run: MERGE gu:...:revenue:... → node created (was_created = true) second run: MERGE gu:...:revenue:... → same node updated (was_created = false) → ZERO new nodes

Latest write wins

A re-run with better/newer data overwrites ~25 properties on the existing node. The slot stays; the contents refresh.

One catch: run ONE model

Because latest-write-wins, running a second model after the first overwrites the first's work (incl. its XBRL links). Pick one model and stick to it.

What this buys you

No "delete before re-run." No idempotency bookkeeping. No pre-check queries. The math of the ID + MERGE makes correctness automatic. The enrichment pass relies on this too: it just re-MERGEs the items it touched.

Guidance · 8 of 24 · the hard parts

Periods are real calendar dates — not fiscal labels.

"Q2 FY2025" means different months at different companies. So instead of storing the fiscal label, we store the actual calendar window. Two companies guiding the same window share the same period node.

period_scopeExample windowFrom text like…
quartergp_2025-04-01_2025-06-30"Q3 revenue of $85B"
annualgp_2024-10-01_2025-09-30"FY2025 CapEx of $30B"
halfgp_2025-04-01_2025-09-30"second-half margin expansion"
long_rangegp_2026-01-01_2028-12-31"by 2028", "2026–2028 target"

Four "fuzzy" windows have no dates — they use sentinels

When the company gives no date anchor, the period is one of four fixed sentinel nodes (dates = null):

gp_ST short-term gp_MT medium-term gp_LT long-term gp_UNDEF undefined

"In the near term" → gp_ST. "Long-term margin model of 38–40%" → gp_LT. "Going forward" (no anchor) → gp_UNDEF. But "by 2028" → long_range (it has a year!).

For balance-sheet "instant" items (cash, debt, shares outstanding), the window collapses to a single day: start_date = end_date.

Guidance · 9 of 24 · the hard parts

To get exact dates, code tries four sources in order.

The AI only says "fiscal year 2025, quarter 2." Turning that into exact calendar dates is a deterministic 4-step cascade in the CLI — it stops at the first source that answers.

A · Reusesame quarter already in graph? inherit its dates
→
B · SEC cachereal filed dates from a Redis cache of SEC data
→
C · Predictunfiled quarter? estimate from the prior one
→
D · FYE mathlast resort: month-math from fiscal year-end

Why so many fallbacks?

Step B is the gold standard — exact dates the company actually filed with the SEC. But guidance often points at a quarter that hasn't been filed yet (that's the whole point of guidance!). So C predicts it, and D does pure calendar math as a safety net. Earlier the answer just used D; the cache + prediction were added to get the boundaries exactly right.

guidance_write_cli.py Step A is a real, working reuse/dedup step (it copies an existing period's dates) — it just rarely fires, because the quarter being guided usually isn't in the graph yet. The whole cascade lives in _ensure_period().

Where the exact SEC dates (Step B) come from

Step B reads a Redis cache keyed fiscal_quarter:{TICKER}:{FY}:Q{n}, built by sec_quarter_cache_loader.py — it polls the SEC EDGAR XBRL companyconcept API (EPS, falling back to net income), derives the real filed quarter / year boundaries with a 52/53-week adjustment, and the triggers call refresh_ticker() to pre-warm it. Step C predicts an unfiled quarter from the prior one's end + a median quarter length. Step D is pure month-math from the fiscal year-end — the only step that can fail.

Guidance · 10 of 24 · the hard parts

"Q1" isn't January for everyone — and that broke Opus.

A company's fiscal year-end (FYE) decides which calendar months its quarters cover. Apple's FYE is September, so Apple's "Q2" is Jan–Mar, not Apr–Jun.

FYE monthExampleQ1Q2Q3Q4
Dec (most)—Jan–MarApr–JunJul–SepOct–Dec
SepAppleOct–DecJan–MarApr–JunJul–Sep
JunMicrosoftJul–SepOct–DecJan–MarApr–Jun

The real-world bug this caused

On Analog Devices (FYE = October), Opus consistently put every period one month too late — FY2026 became gp_2025-12-01_2026-11-30 instead of the correct gp_2025-11-01_2026-10-31. Because the date is part of the ID, every fact got the wrong barcode, so re-runs couldn't dedup. Reproduced twice. This single bug is why Opus was rejected for production.

Opus said: Dec 2025 → Nov 2026 (WRONG — one month late, every period) Correct: Nov 2025 → Oct 2026

Rule: in earnings calls and filings, all periods are fiscal unless the text literally says "calendar". The math lives in fiscal_math.py + guidance_ids.py. (A buggy old helper, fiscal_resolve.py, was deleted in March 2026.)

Guidance · 11 of 24 · the hard parts

Every number is forced onto ONE clean scale.

The AI copies the number exactly as printed ("$10.3 billion") plus the raw unit word. Code does all the scaling — so "$1.13B" and "1,130 million" both become 1130 in millions.

canonical units: usd · m_usd · percent · percent_yoy · percent_points · basis_points · x · count · unknown "$10.3 billion" revenue → 10300 m_usd (aggregate money → millions) "EPS of $1.50" → 1.50 usd (PER-SHARE money → dollars, not millions) "4.94 billion shares" → 4.94e9 count (a share COUNT → absolute count) "gross margin 46–47%" → 46/47 percent "growth of 5–7% YoY" → 5/7 percent_yoy

Three overrides keep money honest

  • Per-share labels (EPS, DPS) → usd, never millions.
  • Share-count labels (shares outstanding) → count.
  • Everything else aggregate → m_usd (millions of dollars).

No alias match → canonical_unit = unknown, and the raw text is kept in unit_raw so nothing is lost. (This "unknown" bucket is bigger than it should be — see the gaps slide 53.)

Guidance · 12 of 24 · the hard parts

The newer unit judge uses evidence, not just the word.

"$2.50" could be EPS (dollars) or a $2.50B aggregate (millions). V2 resolution decides the kind of thing from the strongest evidence available, in 3 axes.

1 · kind

money / ratio / count / multiplier / unknown

2 · money_mode

if money: aggregate (millions) or price_like (per-share dollars)

3 · ratio_subtype

if ratio: percent / percent_yoy / percent_points / basis_points

The evidence ladder (strongest wins)

Hard surface marks (a literal %, $, "per share", "bps") beat the XBRL concept, which beats label hints, which beat the AI's own unit_kind_hint. The AI's hint is the last tiebreaker, not the boss. This is why the bot must always send unit_kind_hint + (for money) money_mode_hint + the verbatim unit_raw.

The bookkeeping (safe to skip)

V2-written rows carry resolved_kind / resolved_money_mode / resolved_ratio_subtype + resolution_version="v2". Older rows (~97% of the live graph) predate V2 and don't. There's also a "shadow" mode that writes the V1 answer but logs what V2 would have said, for safe comparison before flipping over.

Guidance · 14 of 24 · the hard parts

The implicit-switch trap

A CFO can flip basis mid-paragraph: "On a GAAP basis, we expect EPS of $3.20. And revenue should be about $95 billion." → EPS is gaap, but Revenue is unknown (no qualifier attached to it). Guidance · 15 of 24 · the hard parts

"iPhone Revenue" splits into a metric + a segment.

When a metric is qualified by a business dimension (a product, geography, or unit), we split it: the base metric is the label, the qualifier is the segment (default "Total").

"iPhone Revenue" → label="Revenue", segment="iPhone" (decompose) "North America Op. Income" → label="Operating Income", segment="North America" "Cost of Revenue" → label="Cost of Revenue", segment="Total" (do NOT split) "Adjusted EBITDA" → label="Adjusted EBITDA", segment="Total" (do NOT split)

The one test

"Could I have this metric for iPhone AND for Total?" If yes → the prefix is a segment, split it. If the prefix changes what is measured ("Cost of", "Adjusted", "Pro Forma") → keep it whole. Because segment is in the ID, iPhone revenue and total revenue never collide — that's why 5 revenue facts for one period is correct, not a duplicate.

The bot just writes the segment text and sets member_u_ids: []. Linking that text to a real segment node is the code's job (next slide but one).

Guidance · 16 of 24 · wiring to the books

Link each fact to the official accounting concept (XBRL).

Companies file numbers under standardized XBRL concepts (e.g. us-gaap:Revenues). Linking guidance to the concept lets you later compare a guide against the actual reported number. Code does this, deterministically. concept_resolver.py

How concept_resolver.py decides

  1. Look up the metric in a 65-entry reviewed registry (e.g. capex → 3 candidate concept names, tried in order).
  2. Match a candidate against the concepts the company actually uses, ranked by usage.
  3. Fail-closed: if the top two candidates tie, or nothing matches → xbrl_qname = null (no guess).
  4. Force null for derived metrics that have no clean concept (EBITDA, FCF, margins, anything ending _growth/_change/_yoy).

Two outputs

xbrl_qname (a stable property that survives across years) + a MAPS_TO_CONCEPT link to the actual Concept node.

Plus a family

A 19-entry concept_family_qname table maps derived metrics to their closest anchor (e.g. adjusted revenue → us-gaap:Revenues).

Live coverage: ~49% of facts have a concept link. Two gaps cap it: unreviewed labels (like adjusted_eps), and the concept cache being built only from 10-K/10-Q numeric facts — so guidance for a metric that never appears in those filings (some 8-K cases) can't link, no matter the registry (slide 53).

Guidance · 20 of 24 · per asset

Transcripts: the only asset read twice.

Pass 1 reads the scripted prepared remarks. Pass 2 (enrichment) mines the unscripted Q&A, where segment-level and conditional guidance often first appears.

Speaker priority

CFO remarks = formal guidance (highest). Then CFO Q&A, then CEO. Operator = skip. Tag prepared-remark quotes [PR], Q&A quotes [Q&A].

The analyst-framing rule

A number in an analyst's question is NOT guidance — only management's answer counts, and only if it clearly affirms the number.

Confirm vs not-confirm — the line

Analyst: "so ~12% operating margin?" Mgmt: "yes, that's reasonable" → EXTRACT (confirmed) Analyst: "double-digit margins?" Mgmt: "we see improving leverage" → NO GUIDANCE (dodged)

"We'll see", "it depends", "time will tell" = NOT confirmation. The proof quote must be management-answer text only.

Enrichment verdicts per Q&A: ENRICHES an existing fact · NEW ITEM · NO GUIDANCE. Guidance · 21 of 24 · per asset

8-Ks: the event filing — read the attachments.

An 8-K is a short "something happened" filing. The real content is usually an attached exhibit (the press release, EX-99.1), not the filing body.

Plain version: an 8-K reports one event, and its item number says which kind — 2.02 = earnings, 7.01 = a voluntary disclosure (Reg-FD), 8.01 = other, 9.01 = "exhibits are attached." EX-99.1 = the attached press release.

Which 8-Ks, and where to look

  • Only items 2.02 (earnings), 7.01 (Reg-FD / pre-announcements), 8.01 (other) are scanned for guidance. ~99.9% recall, and the extraction AI is the precision gate.
  • Item 2.02 guidance is 94% in the exhibit; 7.01 is 85% in the exhibit → exhibit-first.
  • But always fetch both sections and ALL EX-99.x exhibits — guidance can be split across the press release (99.1) and the slide deck (99.2).

The multi-item trap

An 8-K can carry items 2.02 + 8.01 at once: the EX-99.1 is the earnings release (item 2.02), but a buyback sits in the "Other Events" section (item 8.01). Read only the exhibit and you lose the section's event. Always read both. (Item 9.01 is just "exhibits attached" — pure noise.)

Dirty exhibit numbers exist (EX-99.01, EX-99.-1) — the query uses "starts with EX-99" to catch them. Quote prefix: [8-K].

Guidance · 22 of 24 · per asset

10-Q & 10-K: mostly backward-looking — and that's fine.

Quarterly (10-Q) and annual (10-K) filings are dominated by past results. Guidance, if any, sits in the MD&A (management's discussion). For 10-Ks, watch for multi-year targets.

Zero is a valid answer

10-Qs almost never carry guidance — in the live graph only one GuidanceUpdate ever came from a 10-Q. Don't lower the bar to force a fact. Apply the forward-looking filter strictly.

The apostrophe trap

The 10-K MD&A section name uses a curly apostrophe (Management’sDiscussion…, U+2019); the 10-Q version has no apostrophe. Match the right one per form, or query by "starts with Management".

What to skip, what to fetch

Skip Risk Factors, Legal, Controls, Signatures as guidance targets. Fetch the MD&A first (via the Bash warmup, since it's ~56–72KB), then other narrative sections, then financial-statement notes, with raw filing text only as a last bounded fallback. Quote prefix: [10-Q] / [10-K].

Cross-asset rule: never suppress a 10-Q/10-K fact just because similar guidance appeared in an 8-K or transcript — the barcode ID + value-hash sort out genuine duplicates vs real updates.

Guidance · 23 of 24 · per asset

News: the messiest source — attribution is everything.

A news article mixes company statements, analyst opinions, and reporter narration in one blob. The rule: extract only what the article attributes to the company itself.

Take

"Apple expects Q2 revenue $94–98B." Reaffirmations ("reaffirms FY guidance" → conditions=reaffirmed). Withdrawals ("pulls guidance" → qualitative=withdrawn).

Skip

Analyst targets, "the Street expects", price targets, rating changes, unnamed third parties. When attribution is unclear → skip (precision over recall here).

A tuned upstream filter narrows the haul first

Before the AI even reads, a filter picks news worth looking at: the Benzinga "Guidance" channel as anchor + a 17-keyword title whitelist ("sees", "guidance", "outlook"…) + a $[0-9] regex. Result: ~97% precision, ~98.7% recall. Embeddings and tags were tested and rejected — they cluster by company, not by "is this guidance."

Prior-period values: "Raised to $3.50 from prior $3.20" → extract $3.50 only. Storage gotcha: channels/tags are JSON strings, not lists. News runs on the cheaper Haiku model. Quote prefix: [News].

One Redis list is the conveyor belt.

All jobs — every type, every asset — share a single queue, extract:pipeline. Triggers push on one end (LPUSH); workers pull from the other (BRPOP). Simple, FIFO, no per-type queues to manage.

// a job payload on the queue:

Why one shared queue?

Isolation comes from the prompt stack + job granularity, not from separate queues. The worker reads each job's type field and routes itself. Different types/assets naturally run in parallel across pods.

The dead-letter list

Three outcomes: a job that fails processing 3 times moves to extract:pipeline:dead; an invalid payload is dead-lettered immediately; un-parseable junk is just logged and dropped. Rate-limit "failures" never count toward the 3 (next slide).

Production · 6 of 12

Four safety nets keep it from burning money or looping.

Running an LLM in a loop is dangerous without guards. The worker has four.

1 · Rate-limit re-queue

If a message contains "hit your limit", the job is put back without a retry penalty + the worker sleeps 5 min. A rate-limit isn't a failure.

2 · The write sidecar

The writer drops a sidecar (a small companion file, /tmp/gu_written_{source_id}.json) so the worker can count exactly what was written. It's deleted at job start to avoid stale counts.

3 · Usage throttle

Every 5 min it checks the Claude usage stats; if over the reserved daily quota it pauses 300s before the next job — so batch work never starves your interactive sessions. It guards two buckets: a 7-day all-model quota and a separate 7-day Sonnet quota.

4 · Graceful shutdown

On SIGTERM it stops pulling new jobs but lets the current one finish — K8s gives it a 300s grace period (covers one full extraction).

One footgun in here

The rate-limit re-queue has no cap — a persistently rate-limited job could loop forever — and detection is a fragile substring match on "hit your limit". Worth hardening (slide 53).

Production · 7 of 12

KEDA grows the worker pool when the queue fills.

KEDA is a Kubernetes autoscaler that watches the Redis list length and adds/removes worker copies to match the load.

Min pods
1 in the manifest, so KEDA can't kill a pod mid-extraction. (The live cluster was hand-patched to 0 + a KEDA pause — see slide 48.)
Max pods
7 — safe parallelism across many tickers
Scale trigger
Redis list extract:pipeline, length ≥ 1 adds a pod
Poll / cooldown
checks every 30s; waits 300s before scaling down (no churn)

Why min = 1 and a long cooldown?

Each extraction takes minutes. Scaling to 0 or shrinking too fast could terminate a pod mid-job. min=1 keeps one warm; the 300s cooldown + 300s termination grace mean an in-flight job always finishes. The trade-off is one idle pod when there's no work.

Pods are pinned to one node (minisforum), mount the project + the Claude CLI, and copy a writable .claude.json at startup.

Production · 8 of 12

The AI reads Neo4j through a read-only HTTP bridge.

The reader bots query the graph via an MCP server (MCP = a small adapter the AI talks to instead of the database directly) — an HTTP service that wraps Neo4j and exposes only a read tool. There is no write tool anywhere in the AI's reach.

reader bot ──(MCP over HTTP)──→ mcp-neo4j-cypher-http ──(Bolt)──→ Neo4j read_neo4j_cypher only pinned ClusterIP, /health probe

Why HTTP (not direct)?

In-cluster routing with a stable address. A readiness probe runs RETURN 1 against Neo4j; if Neo4j is down, the bridge stops taking traffic but the pod stays alive (no cascade).

The safety this enforces

Because the bridge has no write capability, an AI cannot mutate the graph even if it tried. All writes are forced through the audited Python writer.

The fiddly in-cluster bits (skip unless deploying)

The MCP call must send a Host: localhost:8000 header so in-cluster routing works. Readiness is an HTTP /health check that runs RETURN 1 against Neo4j (over the Bolt protocol); liveness is a plain TCP check, so a brief Neo4j blip stops traffic without killing the pod. Separately, the worker must chdir into the project before importing the SDK, because the SDK resolves .claude/ relative to the current directory.

Production · 9 of 12

Billing safety: the worker rides the subscription, not the API.

Running thousands of extractions on a metered API key would be ruinous. So the worker authenticates through the Claude OAuth subscription (the CLI's saved credentials), not a pay-per-token key.

How it's enforced — and where

The Kubernetes manifest blanks out ANTHROPIC_API_KEY (and the OAuth token env) to empty strings, forcing the CLI to fall back to its subscription credentials file. The worker calls the CLI at ~/.local/bin/claude.

Important nuance (verified in code)

This guard lives only in the YAML manifest — the worker's Python code never references ANTHROPIC_API_KEY at all. So the protection is the deployment config, not the code. ⚠️ If you ran the worker outside Kubernetes with a real key present in the environment, it could bill the metered API. Inside K8s it's safe.

Per-job ceilings as a second guard: max_turns=80, max_budget_usd=$15. (The reference doc says $5 — the code says $15; the code wins.)

The bigger picture

There's a grander 8-K plan — designed, not built.

A 4,767-line strategy doc designs a 24-tag event taxonomy for all 8-K events (not just guidance) — to feed a stock-move predictor. It is shelved: guidance is the only piece actually built.

The idea

Tag every 8-K with events (EARNINGS, M&A, BUYBACK, LITIGATION, RESTRUCTURING, EXEC_CHANGE…). Part deterministic (from the item code), part AI-judged. Then a predictor reasons over the event timeline.

Why it stalled the way it did

A Haiku "classifier-first" router was tested and retired (only 59% precision binary, 33% on the 24-way). The rule became: no classifier gate — the extraction LLM is the precision gate. That same rule is why guidance routing is just "items 2.02/7.01/8.01, then let Sonnet read."

The most useful finding hiding in there: the amplifier effect

Earnings alone move a stock ~6.8% on average. But earnings + restructuring ≈ 9.5% (and negative); earnings + an exec change ≈ +31% bigger. Co-occurring events matter more than any single one. Also: the more 8-Ks a company files between earnings, the smaller the eventual earnings surprise (the market already knows).

Treat this as a roadmap, not current behavior. Today: guidance extraction only.

Appendix · code 2 of 6

guidance_writer.py — the only thing that writes to Neo4j.

~524 lines. One atomic MERGE per fact, plus the concept/member edges. The single doorway to the graph.

per item: MERGE Guidance + MERGE GuidancePeriod + MERGE GuidanceUpdate + 4 core edges (UPDATES, FROM_SOURCE, FOR_COMPANY, HAS_PERIOD) + best-effort MAPS_TO_CONCEPT + best-effort MAPS_TO_MEMBER (UNWIND)

Write semantics

~25 properties SET on every write (latest-wins). source_refs & aliases are append-only. given_date is re-derived from the source node (not the AI's value). GuidancePeriod dates are ON CREATE only.

The gate & guards

Refuses to write unless ENABLE_GUIDANCE_WRITES is true. 8 validation guards reject corrupt rows (e.g. a per-share value mislabeled as millions). Also creates constraints + the 4 sentinel periods.

Appendix · code 3 of 6

guidance_write_cli.py — the orchestrating CLI.

~656 lines. The bot shells out to this (via guidance_write.sh). It is the sole authority that turns a flat JSON payload into graph writes.

What it does, in order

  1. Parse the JSON envelope; classify payload_origin (extract_v2 / readback / legacy).
  2. Inject source_id, run the 4-step period cascade (slide 21), validate V2 hints.
  3. Run concept_resolver to set/repair xbrl_qname + family.
  4. Recompute ALL IDs via build_guidance_ids() — overwriting anything the AI sent.
  5. Resolve members from the /tmp map (live fallback in write mode).
  6. Dry-run validate, or write a batch; emit the /tmp/gu_written_*.json sidecar.

Key behaviors: dry_run = (mode != '--write') (the footgun). It is the authority for IDs — the AI's IDs are never trusted as-is. Degrades gracefully: Neo4j/Redis lookups are best-effort, falling back to FYE math.

Appendix · code 4 of 6

concept_resolver.py + fiscal_math.py

The two small, sharp helpers behind XBRL linking and date math.

concept_resolver.py 442 ln

A 65-entry reviewed registry (label → ordered concept candidates), a 19-entry family table, force-null lists for derived metrics. Usage-ranked, fail-closed on ties → null. Overwrites the AI's concept pick (slide 28).

fiscal_math.py 140 ln

Calendar ↔ fiscal date math. The date-builder _compute_fiscal_dates() (what guidance calls) maps (FYE, year, quarter) → whole-month start/end dates with no 52/53-week awareness — so a wrong raw FYE produces ~28-day offsets (the Opus bug, slide 22). A sibling, period_to_fiscal(), does handle week-based calendars.

A buggy predecessor, fiscal_resolve.py, was deleted March 2026; its job now lives in guidance_write_cli.py (orchestration) + these two files (the math).
