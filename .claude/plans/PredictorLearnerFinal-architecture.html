Step 1 · Orientation

A self-improving loop, in free-form text

Every earnings event runs four phases. What carries forward between events is past learner reports written in plain prose, each with a one-line takeaway and a link.

A
Build
Builders fetch data; the orchestrator stitches a context bundle and attaches prior_reports_context.
B
Predict
One Claude skill reads the bundle, always opens the most-recent 4 own-ticker reports plus any others it judges relevant, emits a directional call.
C
Outcome
After the event horizon the realized return becomes knowable.
D
Learn
A learner explains why the stock moved versus the prediction and writes a free-form causal report.
Operating principle: free-form prose, well-organized, read on demand. The workflow steps in each skill keep the predictor and learner focused.
Step 2 · The memory model

Memory is past reports the predictor can read

Each completed event leaves a learner report. The next predictor sees a short summary line per report and may open the full prose on demand.

Unit of memory
A learner report — free-form causal prose, one per event
What the predictor sees first
A summary line: Date · Ticker · Quarter · predicted vs actual · one-sentence key_takeaway · Link
How depth is reached
The most-recent 4 own-ticker reports are always opened in full; the rest are opened only when the summary signals relevance (Step 5)
Source of truth
Files on disk — no index, no database, no state to keep in sync
Two consumers, two surfaces. The orchestrator (machine) reads result.json and never parses markdown — that is the canonical structured artifact. The predictor (LLM) reasons off the rendered bundle, and when it opens a prior report it opens that report's rendered result.md. JSON = canonical / numeric-exactness; rendered = the reasoning surface.

Step 5 · Attention budget

Tiered reading — price today's surprise

All 12 reports are visible as summary lines. Which ones get fully opened follows a fixed tier rule that protects attention quality.

GroupRule
Most-recent 4 own-tickerForced open — highest signal-by-recency for this stock
Older own-ticker (5–8)Summary-first; opened only if the takeaway suggests relevance
Industry peers (4 visible)Up to 3 opened based on summary relevance
Every opened reportListed in opened_prior_reports[] in the prediction JSON
Recent own-ticker is the core signal. Older own-ticker is available but stale-risk. Peers matter most during the current earnings season.
Step 6 · The predictor

One skill, two modes

The same skill produces the call in both modes. The mode travels in a single field, pit_mode, stamped into the bundle and echoed in the output.

historical

Point-in-time safe. Seeds the corpus and measures effectiveness on already-released prints. Soft time budget.

live

Real-time. Speed matters — earliest orders get the better bargain. Tiered SLA enforced.

The predictor's job: price the surprise correctly and on time. It first names the few metrics this name's market actually grades on (take-rate / SSS / NRR / copper-realized / backlog …) and judges the print through those, not a generic rev/EPS/margin lens — rejecting any signal (own-metric or peer read) that doesn't map to that mechanism. Two-pass: it always fully opens the most-recent 4 own-ticker reports, then selects additional older own-ticker / peer reports under the Step 5 caps. It cites grounded evidence, **states the one thing that would make this call wrong** (its falsifier), and may answer no_call as a first-class outcome.

Step 7 · Tool gating

The orchestrator gates tools per mode

One skill file. At each SDK invocation the orchestrator passes a mode-specific allowed_tools=[...].

historical

  • Read
  • Bash
  • DataSubAgents (Task) — point-in-time safe

live

  • DataSubAgents
  • WebFetch
  • Neo4j read
  • IBKR

Dedicated sector / macro fetcher sub-agents are a deferred follow-up (not a blocker for restructuring) — built later on the DataSubAgents pattern: live = IBKR, historical = Polygon (+ possibly Yahoo / other free sources).

Step 8 · Live timing

Tiered SLA — live mode

Median live runtime targets ~90–120s. 300s is the exceptional safety bound. Historical mode runs on a softer budget.

300s hard-kill fallback flow
  1. A valid prediction/result.json exists and passes the validator → keep it; stamp timeout_after_valid_write=true.
  2. Missing or invalid → orchestrator writes a complete valid live no_call record: direction:"no_call", confidence_score:0, expected_move_range_pct:[0,0], pit_mode:"live", data_gaps:["live_timeout"], opened_prior_reports:[…logged before kill], analysis:"hard timeout…" — plus timeout=true run metadata.
  3. Run closes as completed-timeout (not ordinary success). The learner still fires on the normal next-event trigger.
Step 9 · The validator

Shape, plus two content rules

The validator checks schema, metadata, range, and evidence. Two semantic rules are hard-blocks — output that violates them never lands on disk.

RuleContract
Anti-lazydirection in {long, short} requires ≥2 grounded source IDs across key drivers, with ≥1 from the current bundle. Applies to every directional call, at any confidence.
no_call (two paths)Balanced: ≥2 key_drivers with both a long and a short driver, each with evidence. Or missing-data: ≥1 data_gaps[] item plus analysis explaining the gap; drivers may be empty, opposing drivers must not be invented.
Evidence catalogStrict. Every cited fact resolves to a known SRC: ID. Live mode also accepts runtime SRC:LIVE:WEB:* / SRC:LIVE:IBKR:* IDs.
The missing-data no_call path lets the predictor say "not enough" honestly instead of fabricating opposing drivers.

Step 11 · The learner

Causal attribution

Once the outcome is known, the learner explains whether the prediction was off, why, and which drivers produced the realized move.

  • Whole-move attribution: account for the entire realized move — classify it durable-fundamental vs one-time/cosmetic vs positioning, name offsetting forces (not just the biggest driver), and explicitly flag any unexplained residual. The residual becomes the next checklist question.
  • Tests the predictor's falsifier: checks whether the one thing the predictor said would break the call actually fired — turning post-mortems into objective hypothesis-checks.
  • Mines its information edge: the analyst Q&A / dodged answers in the transcript (which the predictor never sees) are distilled into checklist questions — re-expressed so they're answerable from the next predictor's bundle (the predictor-observability fence).
  • Data gaps it reports are limited to data it could access that the predictor did not use — a concrete pointer, not speculation.
  • Just-in-time (live): the learner runs immediately before the next prediction for that ticker, whenever the prior outcome has become knowable.
  • Context-aware: it may pull sector / macro context so its causal account is complete for the next predictor.

Illustrative. "Actual returns" may span multiple windows; exact field list + types is a locked pre-implementation prerequisite.

Step 13 · The feedback loop

The standing checklist — learner⇄predictor loop

One rolling own-ticker checklist of questions to inspect, living only inside the latest learner result. It is the only memory beyond prior reports, and it rewrites itself every cycle.

1
LEARNER (after outcome)Reviews the recent PIT-visible own-ticker full reports (already surfaced by the prior-report plumbing; window N parked) + this prediction + outcome → consolidates one new future_checklist: keep durable recurring conditionals even when dormant; otherwise drop freely (refuted/generic/stale/redundant/one-off/weaker); sharpen; add.
2
ORCHESTRATOR (dumb pipe)Carries the latest allowed own-ticker list into the next bundle. No new store. No source link — the source report is already force-opened in prior_reports_context.
3
PREDICTOR (pre-outcome)Engages relevant items in its analysis prose; dismisses the rest in one line. Never used as evidence.
4
Outcome known → loopThe next learner rewrites the list again. The correction lives in the learner's reasoning — nothing else is stored.
RuleWhy
Questions, not answersA persisted answer rots into a wrong rule; a question only directs attention.
Own-ticker onlyLearner reviews recent PIT-visible own-ticker full reports — surfaced by the existing prior-report plumbing, no new discovery path. Peer learning stays in prior_reports_context — never merged.
Never evidenceChecklist text is never a source ID. Directional calls still need ≥2 grounded IDs, ≥1 from the current bundle (V2).
Anti-generic by disqualifierThe learner prompt bans lines true for any company/quarter — not by good/bad examples (LLMs over-anchor; this is a rewrite loop).
Prediction-time actionableLearner thinks with hindsight (it saw this quarter's transcript + outcome); predictor sees only the 8-K + bundle. So every line must be answerable from the 8-K/bundle alone. Insight from prior transcripts → translate into the available 8-K proxy, outcome-neutral wording. No 8-K proxy → omit or a company-history-grounded blind-spot caution; never a blanket "no transcript ⇒ less confident".
Shape gate onlyList of short single-sentence strings. No semantic validator, no IDs, no labels, no checklist_review[].

Cross-report review is ephemeral — questions only, tentative when history is thin, grounded in named past reports in the learner's reasoning, prediction-time actionable (the learner may think with hindsight but may only ask about what the predictor can observe at 8-K time), no stored company profile. Only future_checklist persists.

Not an alpha source — a cheap, asymmetric, compounding error-suppressor. Realized value is gated by the learner distillation prompt, not the architecture.

⏸ PARKED: any cap on item count, a minimum-evidence floor, and the review-window size N. No number is decided — a bot must not hard-code one.

Step 14 · How it's judged

A deterministic eval — scripts compute whether, the LLM only why

The loop is only trustworthy if its scoring is. Every number is authored by code, never by an LLM grading its own homework.

1
Scripts author all numbersdirection-correct, signed-midpoint MSE, Brier/calibration, tie-outs — computed deterministically from the prediction + actual return. The learner authors prose-only (why); it never emits a metric it could be graded on.
2
Frozen replay setRepresentative historical events + the edge cases (no-history, no-peer, missing-data, contradictory drivers, live-timeout), predeclared and frozen so neither a bot nor a human can cherry-pick easy wins.
3
Systematic-bias reportAggregate mean signed error + hit-rate by predicted direction + calibration-by-confidence-bucket — surfaces chronic long/short tilt or over/under-confidence the per-ticker loop structurally can't see.
4
Fail semanticsHard-exit only on objective integrity failure (malformed/missing/impossible math). A single wrong prediction is signal, not failure; CI fails only if a predeclared replay-suite threshold is breached.
This is a script/build, not prompt text — it never touches the predictor/learner prompts. It creates a trustworthy, un-gameable eval where today there is none.
Step 15 · Canonical vs rendered

JSON is canonical; rendered text is what the predictor reads

The learner writes JSON. A renderer produces the markdown. The orchestrator scans the JSON (machine, never the markdown) for the next predictor's summary line — while the predictor reasons off the rendered bundle and opens prior reports as rendered result.md.

Section 10 of the rendered bundle is a bulleted "Prior Reports" list — one multi-line bullet per report: Ticker · Quarter · Date / predicted vs actual / key_takeaway in quotes / Link. The predictor reads this rendered section for reasoning; the JSON sidecar is only the canonical/exactness layer the orchestrator parses. The Link a predictor follows resolves to the prior report's rendered result.md, never its JSON.

Step 16 · Execution

The K8s pipeline

Both predictor and learner run through one production pattern: trigger daemon → Redis queue → KEDA worker → orchestrator subprocess → SDK.

1
earnings-trigger daemonDedicated, always-on; enqueues jobs. Historical backfill starts at the warm-start anchor: earliest 8-K with usable PIT-visible prior 10-Q/10-K. Pre-anchor quarters do not block; no anchor means SKIP no_warm_start_anchor, not failure.
2
Redis queue earnings:pipelineDedicated queue with per-job leases for retry safety.
3
earnings-worker (KEDA)Thin shell — delivers the job to the orchestrator.
4
subprocess: earnings_orchestrator.pyOwns PIT, quarter identity, bundle assembly, learner recovery, and the SDK call with mode-specific allowed_tools.
5
complete.json sentinelsExplicit for both prediction and learner. result.json alone is not safe completion truth under retries/re-entry — the complete.json sentinel is. The Redis lease (row 2) is a separate concern: retry safety, not completion truth.
Step 17 · The whole picture

One screen, end to end

The architecture in a single trace.

trigger enqueues → earnings:pipeline → worker spawns → orchestrator builds the bundle with prior_reports_context (12 visible, glob-discovered, PIT-safe, warm-start bounded) → predictor (mode-gated tools, tiered reading, live SLA) emits prediction_result.v1 through a hard-block validator (anti-lazy + no_call two-path + strict catalog) → the outcome becomes knowable → learner writes learner_result.v1 (causal report + key_takeaway + future_checklist) → that JSON is the next predictor's memory, and its rolling standing checklist of questions is carried forward into the next prediction.

Free-form prose, well-organized, read on demand — a maximally minimal self-improving loop, scored on direction accuracy, signed-midpoint MSE, and Brier calibration.
