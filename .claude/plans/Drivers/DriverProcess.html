A1 · 30-second picture

The whole driver system in 100 words.

Doc status (post-v10 + Tier-6 fold): This HTML is the narrative orientation for the driver naming system. Authoritative spec for v10 lives in: CombinedPlan.md §5.7 (Tier-6 fold of E26-E29 self-heal levers + E30 producer-scope), DriverOntology_Implementation.md §F.10 / §J / §K (Live Token Stores + writer contract + audit schemas), and Neo4jXBRLDesign.md §C3.x (Driver / DriverChange / VocabToken / EquivalenceToken / 5 audit labels). Key v10 facts: Phase-1 producer is LEARNER only per E30 (predictor consumes via registry catalog + prior driver_tags, never writes); PIT visibility is MIN-backdate across all 3 stores (Driver / VocabToken / EquivalenceToken per v9-1 + v10-1 + v10-2); shortcut Drivers land directly with is_shortcut=true per Pattern B (no parallel store, v5-5 + v8-1 + v7-2 ≥2-token gate). The D1/D2 worked examples below use the learner (the Phase-1 producer) end-to-end — read those as illustrative of the algorithm. Reminder: the predictor is consumer-only per E30 (it reads the registry, never writes).

The problem (one paragraph)

Multiple LLMs (or one LLM across multiple events) reading "Apple iPhone sales in China decelerated" emit different names for the same cause. The Neo4j graph ends up with unrelated rows. You can't count, can't query, can't compare. Across thousands of earnings calls this is fatal: there's no way to ask "how often did china_iphone_sales drive a move?" — half the rows say something different.

The fix (one paragraph)

Force the Phase-1 producer (learner; predictor consumes the registry, doesn't write) and future producers (news in Phase 2, fiscal.ai in Phase 3) to consult one shared registry of canonical driver names before emitting. Force the writer to fold every variant through one Python function (canonicalize()) that produces the same canonical form for every spelling/order variant. Result: many LLM emissions → 1 Driver row → many DriverChange rows all pointing to the same Driver. Queryable. Three self-heal levers (learner self-correct + EquivalenceToken N=2 promotion + isolated judge; Lever #1 writer-side auto-repair is now DEMOTED and Lever #3 is now learner self-correct) close the residual ~10-12pp gap toward the ~96-98% clean-landing target.

Three things to notice from this picture

  • The producer LLM never canonicalizes its own output. Python does that, post-hoc. This is why we hit ~96-98% naming accuracy projection even though raw LLM consistency caps near 85% — combined with learner self-correct + EquivalenceToken N=2 promotion + isolated judge (Lever #1 auto-repair is now DEMOTED; Lever #3 is now learner self-correct).

Two terms used above, defined once here: orchestrator = the conductor script (scripts/earnings/earnings_orchestrator.py) that builds the bundle, calls the LLM, validates the output, and dispatches the writer. writer = the Python CLI (driver_write_cli.py) that runs canonicalize + validators + MERGE — NOT a worker pod.

B1 · The progressive layer

A driver tag — the 4 fields you emit.

This is the smallest unit. Every other piece in the system exists to serve, validate, store, or reuse this tag.

{
  "driver_name":  "oil_price",
  "driver_state": "accelerated",
  "direction":    "short",
  "evidence":     ["SRC:NEWS:bz-12345"]
}

Each field, defined here

FieldTypeWhat it carriesExample
driver_namestringThe reusable causal NOUN. Must exist in the registry (or in same emission's propose_new_drivers).oil_price
driver_statestringPast-tense VERB. Drawn from this driver's allowed_states (one of the 7 STATES_VOCAB classes per DriverOntology_Implementation.md §F.5).accelerated

Three things that are not in the tag:

  • The ticker. The Company lives on the FOR_COMPANY edge between the DriverChange node and the Company node.
  • The date / quarter. Lives on the DriverChange's pit_cutoff property.
  • The source ID. Lives on the DriverChange's source_id property.
B2 · Add concept: noun vs verb separation

The name is a noun. The state is a verb. They live in different fields.

3 examples — bad vs good

EvidenceBAD (state baked into name)GOOD (name = noun, state = verb)
"OPEC announced a supply cut" driver_name = "opec_supply_cut" driver_name = "opec_supply"
driver_state = "cut"
"Revenue dropped versus prior quarter" driver_name = "revenue_decline" driver_name = "revenue_trend"
driver_state = "declined"
"Apple lowered iPhone guidance" driver_name = "guidance_lowered" driver_name = "forward_guidance"
driver_state = "lowered"

Why this matters

If the verb is in the name, every state change creates a different driver. opec_supply_cut, opec_supply_raised, opec_supply_inline would all be separate Drivers — defeating the whole "one reusable name" goal. With the verb in driver_state, one opec_supply Driver hosts every event regardless of which direction supply moved.

B3 · Add concept: slot grammar

A name is made of tokens in 6 slot types, in fixed order.

  theme   →   object   →   customer   →   geography   →   institution   →   metric

  THEMES         {ai, ev, ip, semiconductor, cloud, autonomous, cybersecurity, biotech}
  OBJECTS        {iphone, mac, ipad, vision_pro, datacenter, gpu, cpu, cloud_service, advertising}
  CUSTOMERS      {hyperscaler, enterprise, consumer, government, smb}
  GEOGRAPHIES    {china, us, japan, india, eu, emea, apac, latam, mexico, canada}
  INSTITUTIONS   {fed, ecb, boj, opec, fda, sec, doj, ftc}
  METRICS        {revenue, sales, units, deliveries, capex, opex, supply, price, rate,
                  demand, inventory, backlog, bookings, subscribers, arpu, churn}

Real names decomposed into slots

Namethemeobjectcustomergeographyinstitutionmetric
iphone_china_salesiphonechinasales

Rules attached to slots

  • At most one token per slot. china_japan_sales would fail — both china and japan are geography.
  • Effective slot count ≤ 4 (compound metrics like gross_margin count as one). 5+ slots = automatic rejection.
  • Order is fixed. Same set of tokens in any other order = the same name (folded by canonicalize()).
  • Some names are "standalone shortcuts" (next section) and skip slot grammar entirely.
B4 · Add concept: canonical form

5 different LLM spellings → 1 canonical name.

A Python function called canonicalize() takes any raw slug and either folds it to the registry's canonical form or returns a structured rejection reason. Same input → same output, every time, on every machine.

The fold in action

  INPUT (5 different LLM emissions):           CANONICAL FORM:

  "china_iphone_sales"             ─┐
  "iphone_sales_china"             ─┤
  "iphone china sales"             ─┤───→    "iphone_china_sales"
  "iPhone-China-Sales"             ─┤
  "iphone_china_sale"   (plural)   ─┘

  All five resolve to the SAME Driver row. No duplicates created.

How canonicalize() does the folding (high-level — the 12 steps come in C2.2)

  1. Lowercase + replace non-alnum with _ (slug shape).
  2. Apply synonym map (topline → revenue).
  3. Apply plural map (sale → sales).
  4. Apply acronym map (gm → gross_margin).
  5. Strip stopwords (the, of, in).
  6. Check for "standalone shortcut" (yield_curve, opec_supply, fda_approval) — if match, return as-is.
  7. Classify each token into a slot.
  8. Reorder tokens by the fixed slot order.
  9. Emit the joined string.

Standalone shortcuts (R5 renamed from "macro shortcut" per E23 / OQ3 — applied in v10): some names are one whole canonical phrase that doesn't fit the slot grammar. E.g. yield_curve, fda_approval, share_buyback, opec_supply, chip_shortage. Per Pattern B (v5-5 + v8-1 + v7-2 ≥2-token gate), shortcut Drivers register directly with the is_shortcut: bool DEFAULT false schema property — the registry IS the shortcut store; no parallel :EquivalenceToken record. Bootstrap loader populates VocabSnapshot.shortcuts from the SHORTCUTS_VOCAB markdown seed PLUS MATCH (d:Driver) WHERE d.is_shortcut = true (PIT-filtered by registry_visible_at).

Required Driver fields (each new term defined here)

  • name — the canonical lowercase form. Immutable once registered.
  • id = "driver:" + name (derived; UNIQUE constraint enforces no duplicates).
  • label — human-readable display form. May evolve; name stays locked.
  • aliases[] — (alias defined) alternate spellings or token orders that mean the same driver. e.g. china_iphone_sales is an alias of iphone_china_sales. The writer uses these as fast-path lookup hits before invoking canonicalize() (covered in B6). Additive on re-run via Cypher reduce.
  • allowed_states[] — the closed list of verbs valid for this driver. Drawn from one state class (e.g. trend_motion: accelerated, decelerated, stable, declined).
  • segment — (segment defined) the sub-dimension this driver represents. "Total" means whole-company; otherwise it's a specific sub-dim like "iPhone China". Used downstream by the financial-sliver writer to MERGE :MAPS_TO_MEMBER edges to :Member nodes. Required, never null.
  • definition — one-sentence description.
  • base_label — (base_label defined) optional XBRL anchor for the financial-sliver of drivers. Drawn from a closed list (Sales, Revenue, GrossMargin, EPS, CapEx, FreeCashFlow, ...). NULL for non-financial drivers (macro, news, sentiment). Powers :MAPS_TO_CONCEPT resolution. (XBRL = eXtensible Business Reporting Language, the SEC's structured-data format for financial filings; XBRL "concepts" like us-gaap:Revenues live as :Concept nodes in Neo4j and link to financial drivers via the xbrl_qname field.)
  • registry_visible_at — the earliest PIT this Driver is visible from (covered in B10).
B6 · Add concept: reuse-first lookup

Before proposing a new driver, the writer tries 4 ways to reuse an existing one.

slug defined here: a slug is a lower-case ASCII string with non-alphanumeric characters replaced by underscores. The function slug("iPhone China Sales!") returns "iphone_china_sales". All driver names are slugs.

                                  │
        ┌─────────────────────────┘
        ▼
  ┌──────────────────────────────────────────────────────────────────────┐
  │ Try 1: exact match on candidate_slug                                 │
  │   MATCH (d:Driver {name: "china_iphone_sales"})                      │
  │   →  zero results.                                                   │

        │ no hit

  │   MATCH (d:Driver) WHERE "china_iphone_sales" IN d.aliases           │
  │   →  HIT.  Driver.name = "iphone_china_sales".                       │

  └──────────────────────────────────────────────────────────────────────┘

  (If Try 2 had also missed:)

  ┌──────────────────────────────────────────────────────────────────────┐
  │ Try 3: canonicalize, then exact match on canonical form              │
  │   canonical = canonicalize("china_iphone_sales") = "iphone_china_sales"

  │   MATCH (d:Driver) WHERE "iphone_china_sales" IN d.aliases           │
  │   →  zero results.                                                   │
  └──────────────────────────────────────────────────────────────────────┘
        │ still no hit
        ▼

These 4 tries plus the sorted-token + grammar steps + new-driver gate together form the formal B1-B10 algorithm covered in detail in §C2.1. The labels Try 1-4 here map to B3, B4, B6, B7 respectively in that formal numbering.

Why 4 tries

Each try is cheap (exact-string lookup against an indexed field). The 4-step ladder catches the common patterns: identical spelling, alias hit, word-order variation, alias-of-word-order-variation. Less than 5% of well-formed candidates need to reach B8 (the more expensive sorted-token check).

Key takeaway: Reuse paths are exact-string lookups (fast). Canonicalization (B5) only runs if the cheap lookups all miss. Most LLM emissions exit at B3 or B4 — the registry catalog rendered into the bundle makes this extremely common.
B7 · Add concept: propose new

If no reuse works, the LLM can propose a brand-new driver in the same emission.

What a propose-new entry looks like

"propose_new_drivers": [
  {
    "name":           "ai_datacenter_us_capex",
    "label":          "US AI Datacenter Capex",
    "base_label":     "CapEx",
    "segment":        "US AI Datacenter",
    "definition":     "Capital expenditure on AI datacenter buildouts in the US.",
    "allowed_states": ["expanded", "contracted", "accelerated", "stable"],
    "aliases":        ["us_ai_datacenter_capex"]
  }
]

The R11 new-driver gate — ALL must hold

  1. No existing Driver name OR alias matches under canonical form.
  2. Name passes every shape + grammar + banned-content rule.
  3. Every token is in the runtime vocab/registry, OR the new-token gate passes (slot determined by position, not in banned content, appears in evidence text).
  4. Same emission attaches this driver to at least one tag with non-empty evidence.
  5. The driver is reusable beyond one event (no aapl_q3_2025_china_keynote single-use names).
  6. Companion fields all consistent (aliases canonicalize to name, label tokens = name tokens, segment matches name's sub-dim, allowed_states from one class, definition is one sentence not a tautology).

What happens if R11 fails

The proposal is rejected (logged to :DriverProposalRejection audit row). Per E1 PARTIAL policy, other valid proposals in the same emission still land. The single bad proposal is dropped, not the whole run.

B8 · Add concept: DriverChange

Driver = the reusable noun. DriverChange = one event mentioning that driver.

A Driver row is created ONCE, ever. DriverChange rows are created on EVERY mention — every 8-K, every transcript, every news article that names the driver.

One Driver, many DriverChanges over time

  :Driver {name: "iphone_china_sales"}        ← created 2018-Q3 (first 8-K mention)
     ▲      ▲       ▲       ▲       ▲
     │      │       │       │       │
     │      │       │       │       │            (one :DriverChange per source mention)
   2018Q3  2018Q4  2019Q1  2019Q2  ...  +47 more DCs over 5 years

  Each DriverChange carries:
    - source_id        e.g. "learner:AAPL:Q2_FY2026"
    - source_type      learner_result | news | fiscal_kpi
                       (predictor's "prediction_result" formula preserved forensic-only
                        in §source_id FORMULA — not in active enum)
    - driver_state     e.g. "decelerated"
    - pit_cutoff       ISO timestamp — when this mention became visible
    - evidence_refs[]  SRC:* IDs that grounded the claim
    - + supersession triplet (covered in B11)
    - + edge to :Company carrying direction + exposure_role?

Why split Driver from DriverChange

  • Aggregate queries: "show me every event involving iphone_china_sales" = follow :UPDATES from all DCs back to the one Driver row.
  • Per-event facts: different states, different directions per company, different evidence — all live on the DC, not on the Driver.
  • PIT semantics: the Driver carries the EARLIEST visibility timestamp; each DC carries its OWN per-mention pit_cutoff.

:MAPS_TO_MEMBERDC → Member—Only for segment-specific financial drivers

The exposure_role field — why it exists

Most causes affect every company the same way. But some affect different companies in opposite directions. Classic example: OPEC cuts supply → producers (XOM, CVX) go long, consumers (airlines) go short. ONE DriverChange for opec_supply + state=cut can carry MULTIPLE :FOR_COMPANY edges, each with its own direction + exposure_role (producer/consumer/supplier/competitor/neutral). This is populated only by the news pipeline (Mode 2) when one event has cross-company asymmetric impact.

B10 · Add concept: PIT visibility

A Driver is only "visible" to runs whose pit_cutoff is at or after its registry_visible_at.

This is the time-travel guard rail. Without it, a backfill simulating a 2019 prediction would see Drivers coined in 2024 and fake-inflate accuracy.

How registry_visible_at is computed

  registry_visible_at = MIN(all DriverChange.pit_cutoff for this Driver)

  Example timeline for Driver "iphone_china_sales":
  ┌────────────────────────────────────────────────────────────────────┐
  │  2018-09-12   first DC (8-K mention)   pit_cutoff = 2018-09-12    │
  │  2018-12-15   second DC                pit_cutoff = 2018-12-15    │
  │  2019-03-10   third DC                 pit_cutoff = 2019-03-10    │
  │  ...                                                                │
  │  2024-08-20   80th DC                  pit_cutoff = 2024-08-20    │
  └────────────────────────────────────────────────────────────────────┘
  →  Driver.registry_visible_at = MIN(...) = 2018-09-12

  A 2017-Q3 historical backfill (pit_cutoff = 2017-09-30):
    bundle catalog WHERE registry_visible_at <= 2017-09-30
    →  iphone_china_sales is HIDDEN (its 2018-09-12 visibility is after 2017-09-30)
    →  LLM cannot reuse a name that "didn't exist yet"

Cold-start seed exception

Seed drivers (cold-start ~32 anchors like oil_price, fed_rate, fda_approval) have no real DC at bootstrap time. They carry registry_visible_at = 1970-01-01T00:00:00Z (the "epoch sentinel") — always visible, predate any conceivable real pit_cutoff. Era-bound seed candidates like iphone or ai_datacenter_us_capex are excluded from the seed (would leak into 1990 backfills).

L6 MIN-backdate now applies to ALL 3 stores (post v9-1 + v10-1 + v10-2)

The L6 source-pit-anchored visibility rule (analog of Driver.registry_visible_at = MIN(DC.pit_cutoff)) applies symmetrically across all three live token stores:

  • Driver (E5 + original L6): registry_visible_at = MIN(DC.pit_cutoff), MIN-backdate on every new DC.
  • :VocabToken (E10 amended + v9-1 + v10-1): vocab_visible_at = source_driver.registry_visible_at on CREATE; ON MATCH MIN-backdate via MIN(existing, $source_pit). Closes the L6 leak via slot-vocab path AND the out-of-order under-visibility scenario (the v10 reversal of the prior X3 deferral premise — E30 doesn't lock chronological backfill order, so backdate is required for L6 strict alignment).
  • :EquivalenceToken (v4-7 + v10-2): equivalence_visible_at = sort(observation_pit_cutoffs)[N-1] set at promotion; ON every subsequent observation re-evaluated — if sort(new_pit_cutoffs)[N-1] < et.equivalence_visible_at, BACKDATES. Closes the post-promotion under-visibility scenario when out-of-order observations arrive.

promoted_at on :EquivalenceToken is wall-clock audit ONLY (v5-12 — NEVER used for PIT filter). All three stores filter historical-run reads by their respective *_visible_at field. No new mechanism — each is a single CASE expression mirroring the original Driver pattern.

B11 · Add concept: supersession

When a re-run drops a driver, we MARK the old one superseded — we don't delete it.

Re-run scenario

  Run #1  (run_id="R1", date=2024-01-10) on AAPL Q1 8-K:
    emits: [iphone_china_sales, services_revenue, gross_margin]
    Three DCs created, all with source_id = "learner:AAPL:2024Q1", run_id = "R1"

  Run #2  (run_id="R2", date=2024-02-20) on the SAME 8-K (we improved the prompt):
    emits: [iphone_china_sales, gross_margin, services_arpu]
                              ↑ services_revenue is no longer emitted; services_arpu is new

  Writer reconciles by comparing source_id current-set vs prior-set:
    ─ iphone_china_sales  →  no change.    DC properties refreshed.
    ─ gross_margin        →  no change.    DC properties refreshed.
    ─ services_revenue    →  DROPPED in R2. Old DC gets:
                                superseded_at         = 2024-02-20
                                superseded_pit_cutoff = R2.pit_cutoff
                                superseded_by_run_id  = R2
    ─ services_arpu       →  NEW in R2. Fresh DC inserted with run_id="R2".

Why this matters (E14 — the source_id formula)

Supersession only works because source_id is stable across re-runs: learner:AAPL:2024Q1 is the SAME string in R1 and R2. run_id is a separate field that changes per run. If source_id were run-scoped (e.g. some UUID per run), the writer would have no way to detect "R1 had services_revenue, R2 doesn't" — the audit trail breaks.

The PIT query (after supersession is in the picture)

  Visible to a query at time T:
    dc.pit_cutoff <= T
      AND (dc.superseded_pit_cutoff IS NULL
           OR dc.superseded_pit_cutoff > T)

  Plain-English: "this DC was already visible at T, AND it wasn't yet
                  superseded as of T." So queries at T_old (before
                  supersession) still see the dropped driver; queries
                  at T_new (after) don't.

RulePlain English
R1Reuse first. Check the registry before proposing.

C2.1

The B1–B10 algorithm — "should I reuse or propose?"

For every causal variable the LLM extracts from evidence, the writer runs these 10 steps in order. Most candidates exit early (B3-B7 are reuse paths). Only fresh, never-seen drivers reach B10.

  B1  EXTRACT          evidence → (noun_phrase, state_verb, direction, evidence_refs)
                       Reject if any field empty.
                       │
                       ▼
  B2  SLUGIFY          slug(noun_phrase). Reject if shape regex fails.
                       │
                       ▼
  B3  EXACT NAME       Does any Driver.name == candidate_slug?
                          YES → REUSE. STOP.
                       │ NO
                       ▼
  B4  EXACT ALIAS      Is candidate_slug in any Driver's aliases[]?
                          YES → REUSE. Append candidate to aliases (optional). STOP.
                       │ NO
                       ▼
  B5  CANONICALIZE     canonical = canonicalize(candidate_slug)
                       The 12-step pure function — see C2.2.
                       │
                       ▼
  B6  CANONICAL NAME   Does any Driver.name == canonical?
                          YES → REUSE. STOP.
                       │ NO
                       ▼
  B7  CANONICAL ALIAS  Is canonical in any Driver's aliases[]?
                          YES → REUSE. STOP.
                       │ NO
                       ▼
  B8  SORTED-TOKEN     IF every token is known to vocab/registry,
                       compare sorted(canonical tokens) vs sorted(Driver name tokens).
                          Unique match → REUSE. STOP.
                          Zero/multiple matches → proceed.
                       │
                       ▼
  B9  GRAMMAR + GATE   Run §D grammar + new-token gate + §E validators on name.
                       Any fail → STOP with named rejection reason.
                       │ all pass
                       ▼
  B10 NEW-DRIVER GATE  Verify R11 in full (covered in C2.5).
                          Pass → emit propose_new_drivers entry + tag.
                          Fail → STOP with failing condition as rejection reason.
C2.2

canonicalize() — the 12-step deterministic folder, in full detail.

This single Python function is the heart of the entire driver system. It takes any raw slug an LLM emits and either folds it to the canonical form OR returns a structured rejection reason. Pure function = no side effects (reads nothing, writes nothing) + same input ALWAYS produces same output, on any machine, today or in 5 years. Runs AFTER the LLM, never inside it (per L3).

The function in one signature

def canonicalize(candidate: str) -> str | REJECTION:
    """
    Input:  any lower_snake_case slug ("china_iphones_topline")
    Output: canonical slug ("iphone_china_revenue")
            OR structured rejection (REJECTION_BANNED_TOKEN('cut'), ...)

    Pure. Deterministic. No I/O. No network. No randomness.
    """

Why this specific order — 5 key dependencies

  1. Step 2 (compound) BEFORE step 3 (tokenize). Compound substitution operates on substrings; once split into individual tokens, the function can't re-recognize that gross + margin were a compound metric. The substitution has to happen on the joined string.
  2. Step 4 (stopwords) BEFORE step 5 (normalize). Stopwords aren't in any normalize map anyway — cheaper to strip them first so step 5 has less to process.
  3. Step 5 (normalize) BEFORE step 6 (dedup). Normalize is what CREATES the dupes (two different spellings collapse to the same token). Dedup has to come after.
  4. Step 7 (banned) BEFORE step 8 (shortcut). Shortcuts must still respect R7. If a slug looked like a shortcut but contained a banned word (e.g. someone tried opec_supply_cut), the cut token would still get caught at step 7 — the shortcut lookup at step 8 can't bypass it.
  5. Step 11 (metric + length) at the END. Earlier steps (compound substitution at step 2, dedup at step 6) can change the effective slot count. The check has to happen after all transformations.

Key takeaway: canonicalize() is a 12-step pipeline where 6 steps TRANSFORM the data, 4 can REJECT (steps 1, 4, 7, 11), 1 can EARLY-RETURN (step 8), and 1 EMITS (step 12). The ordering is non-negotiable — every step depends on the work the previous steps did. The function is pure: same slug in always gives same output out — 100% reproducible. But be precise about what that buys: the pure fold guarantees LEXICAL determinism only (spelling/word-order plus known synonym/plural/acronym variants collapse to one name). That is the FLOOR, not the whole story. The ~95-98% naming consistency is a PROJECTION from canonicalize() plus the self-heal levers (N=2 promotion) plus the isolated judge — not the pure function alone. And the pure fold does NOT converge SEMANTIC word-choices: iphone_china_sales vs iphone_china_demand mint DIFFERENT names until the judge / Pass-2 synonym layer folds them (eventual consistency).

Implementation invariant: a reused Driver is malformed if canonicalize(driver.name) != driver.name — that would mean a Driver was registered under a non-canonical form, which would be a constraint violation. V15 catches this at the registry level.

Failed proposals — what gets dropped vs what breaks the run (E1 PARTIAL policy)

  Per-proposal outcomes:
   ─ Proposal canonicalizes + validates              →  ACCEPTED.  Driver + DC written.
   ─ Proposal fails canonicalize OR a validator      →  REJECTED.  Logged to
                                                          :DriverProposalRejection
                                                          (audit row).  This single
                                                          proposal is dropped.

  Run-level sentinel decision (E1 exit codes):
   ─ ≥1 driver wrote successfully, no system failure
        → exit 0.  Orchestrator writes complete.json sentinel.
        → Sidecar reports accepted_count, rejected_count, per-driver reasons.

   ─ ALL proposals failed (zero drivers wrote)
        → exit 1.  Sentinel BLOCKED.  Run marked failed in run_ledger.

   ─ Writer / system failure (Neo4j down, file I/O error, schema violation)
        → exit 2.  Sentinel BLOCKED.  Run marked failed in run_ledger.

  proposal) was rejected during plan review — too brittle.  One sloppy
  "collapse" token in a driver_name should not throw away an otherwise-
  good prediction.

C2.5

R11 new-driver gate in detail

Reaching B10 means none of the reuse paths matched. R11 is the final check before a brand-new Driver is allowed into the registry.

Gate conditions — ALL must hold

  1. No registry name or alias matches the candidate under canonical form. (Already checked by B3-B7; revalidated here.)
  2. Every rule R1-R10 passes on the proposed name.
  3. Every token is either in vocab/registry/aliases, OR it's a new token whose slot is unambiguously determined by its position among known tokens.
  4. The new token is not in any banned category.
  5. The new token does not exactly equal any existing name/alias/vocab entry.
  6. The new token (or its synonym/plural/acronym pre-image) appears in the supporting evidence text. Prevents "made-up" tokens.
  7. The same emission attaches this driver to at least one tag with non-empty evidence.
  8. The driver must not be tied to a single specific event, date, filing, company-quarter, headline, or source row. One-off concepts are rejected.
  9. All companion fields satisfy R10 (aliases canonicalize back to name, label tokens = name tokens, segment matches sub-dim, allowed_states from one class, definition one sentence).

Any single failed condition = the proposal is rejected (audit row written), the run continues with other proposals per the PARTIAL policy.

Key takeaway: R11 is a 9-condition AND-gate. Condition #8 ("reusable beyond one event") is the most rejected — LLMs love to encode quarter-specific or company-specific details. Condition #6 ("token appears in evidence text") is the anti-hallucination floor.

C3.3

The 6 edges and what each carries

  :DriverChange edges:

   (dc)-[:UPDATES]->(:Driver)
        ALWAYS. Always exactly one.

   (dc)-[:FROM_SOURCE]->(:Report | :Transcript | :News | :FiscalKPI)
        BEST-EFFORT. Created only when the source maps to an existing Neo4j node.
        For source_type ∈ {learner_result}: NO edge.
        Learner result is a file-backed artifact; source_id + result_path stay
        as DC properties. (Historical: prediction_result was also file-backed
        before being removed from the active enum.)

   (dc)-[:FOR_COMPANY {direction, exposure_role?}]->(:Company)
        ALWAYS at least one. One edge per affected company.
        direction is REQUIRED on the edge ({long, short}).
        exposure_role is OPTIONAL ({producer, consumer, supplier, competitor, neutral}) —
          populated only when one DC affects multiple companies with non-uniform signs
          (e.g., OPEC cut: producers long, consumers short — populated by news pipeline).
        ON MATCH: direction MAY change on legitimate revision, but every change is
          LOGGED to :DriverDriftAudit (audit trail without per-revision DC bloat).

   (dc)-[:CITES_EVIDENCE]->(:Report | :Transcript | :News)
        BEST-EFFORT. Created when an SRC:* ID in evidence_refs parses to a graph node.
        Multiple per DC if evidence_refs has multiple resolvable IDs.

   (dc)-[:MAPS_TO_CONCEPT]->(:Concept)
        OPTIONAL. Only for financial drivers with a resolved xbrl_qname
        (inherited via Driver.xbrl_qname). LIMIT 1 picks one for multi-taxonomy
        Concepts that exist across multiple year nodes.

   (dc)-[:MAPS_TO_MEMBER]->(:Member)
        OPTIONAL. Only for segment-specific financial drivers, resolved via
        Driver.segment + per-ticker member_map. Can be 0..N — multi-dimensional
        segments produce multiple member edges.
C3.4

The atomic MERGE Cypher template

One per item. Mirrors the guidance writer pattern (guidance_writer.py:191-259).

// 1. MATCH preexisting source + company
MATCH (company:Company {ticker: $ticker})
OPTIONAL MATCH (source) WHERE labels(source)[0] IN ['Report','Transcript','News','FiscalKPI']
                          AND source.id = $source_node_id

// 2. Snapshot existing DriverChange for was_created flag
OPTIONAL MATCH (existing:DriverChange {id: $dc_id})

// 3. MERGE Driver node (registry)
MERGE (d:Driver {id: $driver_id})
  ON CREATE SET d.name                = $name,
                d.label               = $label,
                d.aliases             = $aliases,
                d.allowed_states      = $allowed_states,
                d.segment             = $segment,
                d.definition          = $definition,
                d.base_label          = $base_label,
                d.registry_visible_at = $pit_cutoff,
                d.created_at          = $now_iso
  ON MATCH SET d.aliases = reduce(acc=[], a IN (coalesce(d.aliases, []) + coalesce($aliases, []))
                          | CASE WHEN a IS NULL OR a IN acc THEN acc ELSE acc + a END),
               d.registry_visible_at = CASE
                                          WHEN d.registry_visible_at IS NULL THEN $pit_cutoff
                                          WHEN datetime($pit_cutoff) < datetime(d.registry_visible_at)
                                            THEN $pit_cutoff
                                          ELSE d.registry_visible_at
                                       END

// 4. MERGE DriverChange (occurrence)
MERGE (dc:DriverChange {id: $dc_id})
  ON CREATE SET dc.source_id    = $source_id,
                dc.source_type  = $source_type,
                dc.result_path  = $result_path,
                dc.driver_id    = d.id,
                dc.driver_state = $driver_state,
                dc.evidence_refs = $evidence_refs,
                dc.pit_cutoff   = $pit_cutoff,       // NEVER updates
                dc.authored_at  = $authored_at,      // NEVER updates
                dc.written_at   = $now_iso
  ON MATCH SET  dc.evidence_refs = dc.evidence_refs
                                   + [x IN $evidence_refs WHERE NOT x IN dc.evidence_refs],
                dc.written_at = $now_iso
                // pit_cutoff, source_id, driver_id, driver_state, result_path NEVER update on match

// 5. Core 3 edges
MERGE (dc)-[:UPDATES]->(d)
WITH dc, d, source, company, existing
FOREACH (s IN CASE WHEN source IS NOT NULL THEN [source] ELSE [] END |
   MERGE (dc)-[:FROM_SOURCE]->(s)
)
MERGE (dc)-[fc:FOR_COMPANY]->(company)
  ON CREATE SET fc.direction = $direction, fc.exposure_role = $exposure_role
  ON MATCH SET fc.direction = $direction, fc.exposure_role = $exposure_role
  // (driver-drift audit row written separately when fc.direction actually changes)

RETURN dc.id AS id, existing IS NULL AS was_created

The CITES_EVIDENCE, MAPS_TO_CONCEPT, MAPS_TO_MEMBER edges run as separate per-item queries after the atomic block (best-effort, won't fail the main write).

C3.5

Supersession triplet — when set, what clears

Trigger

After the writer MERGEs all currently-emitted DCs for a given source_id, it runs a reconciliation pass: which DCs exist in Neo4j for this source_id but are NOT in the current emission? Those are the dropped drivers.

  After writing R2's drivers:
   ─ Compute current_set = {dc_id for each emitted tag in R2}
   ─ Query prior_set = MATCH (dc:DriverChange {source_id: $source_id})
                       WHERE dc.superseded_at IS NULL
                       RETURN dc.id
   ─ to_supersede = prior_set - current_set

  For each dc in to_supersede:
   ─ SET dc.superseded_at         = $now_iso
   ─ SET dc.superseded_pit_cutoff = $run.pit_cutoff
   ─ SET dc.superseded_by_run_id  = $run.run_id
   ─ Optionally: write :DriverDriftAudit edge for the un-emission event

Un-supersession (gap-then-return case)

If a previously-superseded DC reappears in a later run's current set (we briefly dropped a driver, then re-emitted it), ALL THREE supersession fields are cleared back to NULL. The DC becomes authoritative again. The drift_audit log captures the un-supersession event.

  R3 (later run) re-emits services_revenue:
   ─ MATCH (dc:DriverChange {id: "dc:learner_AAPL_Q2_FY2026:services_revenue:declined"})
   ─ WHERE dc.superseded_at IS NOT NULL    // was superseded by R2
   ─ SET dc.superseded_at        = null,
        dc.superseded_pit_cutoff = null,
        dc.superseded_by_run_id  = null
   ─ + log un-supersession to :DriverDriftAudit
Key takeaway: Supersession is a SET operation, never DELETE. The triplet tells PIT queries "this was true then, isn't now." Un-supersession (clear the triplet on re-emission) is symmetric — gap-then-return scenarios stay clean. Every flip is audited.
C4 · Component

PIT + re-run — the time-travel discipline.

5 sub-pieces: how registry_visible_at is computed (C4.1) → the bundle PIT filter (C4.2) → the cold-start two-tier seed policy (C4.3) → supersession behavior on re-run (C4.4) → the live vs historical PIT predicate (C4.5).

C4.1

registry_visible_at = MIN(DC.pit_cutoff)

Why MIN? Because a Driver becomes "knowable to the world" the moment its FIRST mention exists. After that, every backfill from a later date should see it.

  Scenario:  Driver "iphone_china_sales" has DCs at PITs:
                2018-09-12, 2018-12-15, 2019-03-10, 2024-08-20

  WRONG (using MAX):  registry_visible_at = 2024-08-20
    →  A 2020 backfill (pit_cutoff=2020-03-01) would HIDE iphone_china_sales
       (2024-08-20 <= 2020-03-01 is FALSE).
    →  But the driver clearly existed in 2018.  Backfill mis-hides it.
    →  LLM re-proposes a near-duplicate → registry pollution.

  RIGHT (using MIN):  registry_visible_at = 2018-09-12
    →  A 2020 backfill sees the driver (2018-09-12 <= 2020-03-01 is TRUE).
    →  Reuse works.  No pollution.

  GENUINE future-vocab leak still blocked (Mode A):
    →  Driver first observed Jan 2026 → registry_visible_at = Jan 2026.
    →  2024 backfill (pit_cutoff=2024-Q1): Jan 2026 <= 2024-Q1 is FALSE.
    →  HIDDEN.  Future drivers correctly invisible to past backfills.

The MIN update happens ON MATCH SET in the writer Cypher: registry_visible_at = MIN(existing, new dc.pit_cutoff). Backward updates allowed — a later backfill discovering an earlier mention pushes the visibility timestamp back, which is correct PIT logic.

C4.2

Bundle PIT filter — the WHERE clause that prevents future leaks

  When the bundle renderer queries the registry for the catalog block:

  // For historical runs (run.pit_cutoff IS NOT NULL):
  MATCH (d:Driver)
  WHERE d.registry_visible_at <= datetime($run_pit_cutoff)
  RETURN d.name, d.aliases, d.allowed_states, d.segment, d.definition
  ORDER BY d.name

  // For live runs (run.pit_cutoff IS NULL — predictor running right now):
  MATCH (d:Driver)
  RETURN d.name, d.aliases, d.allowed_states, d.segment, d.definition
  ORDER BY d.name

The catalog excerpt that lands in the LLM prompt only contains drivers whose registry_visible_at predates the run's pit_cutoff. The LLM physically cannot reuse a name it doesn't see.

C4.3

Cold-start seed — the two-tier policy

Before the very first production run, the registry is empty. Empty registry → LLM has no slot-classification anchors → accuracy drops to ~80%. So we pre-seed ~32 anchor Drivers.

TIER 1 — timeless tokens (~32 anchors, all carry epoch_sentinel = 1970-01-01)

  macros (~11):           yield_curve, fed_rate, ecb_rate, boj_rate, oil_price, oil_supply,
                          vix, credit_spread, treasury_yield, usd_index, inflation_rate
  compound metrics (~6):  gross_margin, operating_margin, net_margin, free_cash_flow,
                          operating_cash_flow, capital_expenditure
  timeless geographies:   china, us, japan, eu, india                              (~5)
  timeless institutions:  fed, opec, ecb, fda, sec                                  (~5)
  timeless metrics:       revenue, sales, capex, opex, eps                          (~5)

  Total: ~32 anchors covering metric, geography, institution, and macro slots.
  All carry registry_visible_at = 1970-01-01T00:00:00Z — always PIT-visible.

TIER 2 — modern / product / company-specific (EXCLUDED from seed by default)

  Examples that would BE TIER 2 (NOT seeded):
    iphone, mac, ipad, datacenter, gpu, ai_capex, ev_demand, hyperscaler_bookings

  Why excluded:  these are era-bound.  Seeding "iphone" with epoch_sentinel
                 would mean a 1990 backfill sees "iphone_china_sales" as a
                 valid existing name — a clear PIT leak (the iPhone didn't exist).
  Path forward:  Tier 2 drivers grow organically through the first 3-5 successful
                 propose_new_drivers emissions on real evidence.

Slot-coverage caveat

Tier 1 anchors don't cover OBJECTS, CUSTOMERS, or THEMES slot types. Until ~3-5 propose_new_drivers entries land for those slots, early-run classification for those slot types may drift more than for the slots Tier 1 covers. Plan accordingly: first month after launch is the noisy period.

Where the seed lives in code (OQ4)

A Python constant named COLD_START_SEED_DRIVERS inside driver_writer.py. NOT an LLM bootstrap call (would violate L4 — no runtime curator). The list is editable in source control, code-time engineering.

C4.5

Live vs historical PIT predicate

  HISTORICAL run (predictor.pit_cutoff IS NOT NULL):

   WHERE dc.pit_cutoff <= predictor.pit_cutoff
     AND (dc.superseded_pit_cutoff IS NULL
          OR dc.superseded_pit_cutoff > predictor.pit_cutoff)

   ─ First clause:   DC was visible by time T
   ─ Second clause:  DC was not yet superseded as of time T
   ─ Together:       returns the AUTHORITATIVE driver set as-of T

  LIVE run (predictor.pit_cutoff IS NULL — happening right now):

   WHERE dc.superseded_pit_cutoff IS NULL

   ─ Skip the time filter (live has no PIT restriction)
   ─ Keep the supersession check (only see currently-authoritative DCs)

What's preserved across re-runs

  • The ORIGINAL run's DCs are never modified at the identity/PIT level.
  • NEW DCs added in a revision have their own pit_cutoff = revision pit; only visible to predictors at T ≥ revision pit.
  • DROPPED DCs carry a superseded_pit_cutoff; visible to predictors at T < revision pit (they were authoritative then), hidden at T ≥ revision pit.
  • Direction flips on FOR_COMPANY edges show the latest call; the drift_audit log preserves the history.

F · Edge case

F1 — PIT leak (what happens if the filter is removed).

  WITHOUT the bundle PIT filter:

  Scenario: 2017-Q1 historical backfill for AAPL.
   run.pit_cutoff = 2017-03-31

  Bundle renderer query (NO filter):
   MATCH (d:Driver) RETURN d.name, d.aliases, ...
   →  returns ALL 47 Drivers, including ones with
      registry_visible_at = 2024-08-20  (coined 7 years AFTER the simulated PIT).

  LLM sees "ai_datacenter_us_capex" in the catalog.
   →  emits a tag referencing it.
   →  "the predictor in 2017 used a name that didn't exist until 2024."
   →  Accuracy metric is FAKE-INFLATED — the model couldn't really have used that name.

  WITH the filter (E5):
  WHERE d.registry_visible_at <= datetime("2017-03-31")
   →  returns only Drivers visible at or before 2017-Q1.
   →  LLM physically cannot reuse future-coined names.
   →  PIT honest.
Key takeaway: Without the WHERE clause, every backfill becomes a fake-accuracy benchmark. The filter is one line; the cost of forgetting it is silent and irreversible.
F · Edge case

F2 — Partial vs strict failure (why E1 is PARTIAL).

  STRICT policy (rejected during plan review):
   - On ANY failed proposal, the whole run fails.
   - One sloppy state-in-name ("collapse" inside a driver_name) → entire prediction
     thrown away.  result.json is preserved but no sentinel fires.
   - Operationally too brittle: a 4-tag emission with 1 bad tag loses 3 good drivers.

  PARTIAL policy (locked):
   - On failed proposal, log to :DriverProposalRejection.  Drop THIS proposal.
   - Continue with remaining valid proposals.
   - Sentinel fires IFF ≥1 driver wrote successfully AND no system failure.
   - Run-level outcomes:
       exit 0:  ≥1 wrote, all OK             → complete.json sentinel
       exit 1:  ALL failed (nothing wrote)   → no sentinel, run marked failed
       exit 2:  system failure (Neo4j down)  → no sentinel, run marked failed
Key takeaway: One bad tag never kills the run. The sentinel fires whenever at least one driver wrote. Audit-log every rejection (per-tag) so you can chase quality without losing throughput. (Mechanism + exit-code semantics → §C2.4.)
F · Edge case

F3 — Concurrent writers (Phase 1: learner re-runs of the same event · Phase 2: learner + news).

  Two WRITERS can run for the same earnings event at different times — Phase 1: re-runs
  of the learner on the same event; Phase 2+: learner + news. (The predictor is
  consumer-only per E30 — it NEVER writes, so it is never a concurrent writer.)
  Two writers may emit the SAME proposed new Driver simultaneously (rare but possible).

  Without protection:
   - Both writers MERGE (d:Driver {id: "driver:foo"})
   - Neo4j MERGE is atomic per node — only one CREATE succeeds; the other becomes a MATCH.
   - BUT: between writer A's lookup and MERGE, writer B may MERGE — leading to
     potential ON CREATE vs ON MATCH timing differences.

  E11 protection:
   1. UNIQUE constraint on Driver.id  (Neo4j enforces).
   2. Wrap canonical lookup + MERGE in a single transaction.
   3. On Neo.ClientError.Schema.ConstraintValidationFailed:
        retry the lookup → existing Driver is found → REUSE.

  Net effect: at most one Driver row per canonical name, ever, regardless of
  how many writers race.
Key takeaway: UNIQUE constraint + transactional lookup + retry-on-collision. Three lines of code. Without this, two writers (Phase 1: learner re-runs of the same event; Phase 2+: learner + news) running back-to-back can both create a Driver row, breaking the "one row per canonical name" invariant silently. (The predictor never writes — consumer-only per E30.)

F4 — Cold-start period (the first ~30 days).

  Before first production run:  registry has 0 Drivers.
   →  LLM has no slot anchors  →  classification of novel tokens is guesswork
   →  accuracy ~80% (vs target ~95%)

  E9 mitigation: cold-start seed of ~32 Tier-1 anchors loaded at deploy time.
   →  All 6 slot types except OBJECTS, CUSTOMERS, THEMES are now covered.
   →  LLM can pattern-match new emissions against existing slot examples.

  But Tier 1 doesn't cover era-bound slots.  First 3-5 propose_new_drivers
  for OBJECTS / CUSTOMERS / THEMES carry classification risk.

F5 — Slot-vocab pollution (low-quality accepted proposal).

  Scenario: a propose_new_drivers entry SQUEEZES through R11 with a
            questionable token (e.g., "synergy" — vague, sentiment-adjacent).

   →  Future canonicalize() calls classify "synergy" into that slot.
   →  More LLM emissions use it.  Pollution compounds.

  Mitigations:
   1. R7 banned content is broad: vague descriptors {update, performance, quality,

Key takeaway: Banned-content list is the front-line defense; zero-anchor rejection is the second; code-time vocab pruning (engineer-edited constant, not runtime curator) is the cleanup. Three layers, no runtime human-in-loop required.

  CORRECT (locked formula):
