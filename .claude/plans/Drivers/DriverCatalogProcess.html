The goal

Think of it as one master ingredient list — for the whole stock market.

We want one list of every cause that moves a stock, where the same cause always has the same name. So when "beef price" jumps, we instantly know every company it touches → a trade signal.

The restaurant analogy

Imagine every restaurant in the country writes its own menu, separately:

  • Restaurant A writes guest count. Restaurant B writes customer transactions.
  • Those are the same thing → they must collapse to one name.
  • But delivery mix at a restaurant ≠ delivery mix at a shipping company → must stay two names.

The whole project is: collect every "menu," then carefully decide what is truly the same and what isn't — without ever guessing.

Why same-name matters

If two companies use one shared name for a cause, software can watch that one name and detect a change across all of them at once. That's the payoff: learn once, predict everywhere.

Why it must be tiny

Keep the list as small as possible while still being specific. No bloat. But "small" never wins over "correct" — we never merge two different causes just to shrink the list.

The one promise that makes it tradeable: one name = one meaning

A single driver name must never stand for two different real-world causes, anywhere, at any level. If it ever does, software would read a fake connection and place a bad trade. So the entire build is bent toward protecting that one promise — and when two names truly mean the same, they're linked with a reversible link (never a destructive delete), so the originals can always be recovered.

The scars

Two earlier tries failed. Every rule here exists to avoid their deaths.

This isn't over-engineering — it's two specific funerals we refuse to repeat.

AttemptHow it diedThe fix baked in now
v1 Used a fixed, pre-written word-list. Real names didn't fit it → it rejected 82% of them. Open vocabulary — names come straight from the source text, never a fixed list.
v2 Merged too eagerly. Many different ideas collapsed into one mushy revenue_demand. Fail-closed merging — only merge on exact same meaning; a skeptic attacks every merge; when unsure, keep separate.

The one mistake we fear most

A wrong merge. If we wrongly say two companies share a driver, software reads a false connection between them and places a bad trade. So merging is the dangerous direction — and the entire design is built to make wrong merges hard.

The two judgment moments that caused both failures
  • "Is this already named?" → get it wrong and you fragment one idea into many names.
  • "Is this a real, specific-enough driver?" → get it wrong and you drift into over-broad mush.

Rules can narrow these two judgments but never remove them. So the design keeps the judgment with AI, wraps it in skeptics, and lets code catch every structural mistake.

The spine

One rule explains the entire system.

AI decides meaning. Code does structure. Every merge must survive a skeptic — or it doesn't happen.

🧠 AI (the agents)

Anything that needs understanding:

  • "Is this a real driver?"
  • "Do these two mean the same?"
  • "Try to break this merge."

⚙️ Code (Python / JS)

Anything mechanical:

  • Pull data, split text, count.
  • Write the final file.
  • Check everything & hard-stop on any flaw.

Why this split is the whole trick

Because code writes the catalog, an AI cannot sneak in a merge the skeptic never approved. The old design let an AI write the file — so it could quietly re-create the v2 mush. Now that's structurally impossible. The AI only hands code a short list of approved decisions.

fail-closed When in doubt, don't merge. A missing or uncertain answer is treated as "no." The safe direction is always keep separate.

The model · part 1 of 5

Three things, not one: a label, a fact, and a verdict.

Before any code, picture the end result. The whole system stores exactly three kinds of thing. Get these three and everything else clicks.

1 · Driver node

The reusable label (a "class"). Just a clean name + its permanent kind. same_store_sales. Built once.

2 · DriverUpdate node

One fact on one event (an "instance"): "this driver did X on this day." Many pile up under one label over time.

3 · Verdict edge

A judgement that the fact caused the stock move. It is not a box — it's a labelled link (EXPLAINED_BY). A fact can exist with no verdict.

( Driver ) the LABEL (class) e.g. same_store_sales ^ | OF_DRIVER (every fact belongs to exactly 1 driver) ( DriverUpdate ) one FACT on one event "rose 3% in Q1" | ^ FROM_SOURCE | EXPLAINED_BY { the VERDICT } ← only if it moved the stock v | ( Event )--+ a real 8-K / transcript / news item | | INFLUENCES / PRIMARY_FILER { the real stock move } v ( Company ) reached ONLY through the event

The one analogy

A Driver is a blank thermometer (oil_price). A DriverUpdate is one reading taken at a real event ("oil rose to $80"). The verdict is a separate note clipped onto the event: "and that reading is why the stock fell." Three separate things — never mashed into one.

Why is the verdict an edge and not its own box?

Because a verdict only means something against the real return — and the return lives on the Event → Company link. A company has hundreds of thousands of returns, so it can't say which move a verdict is about; the event can. So the verdict hangs off the event, one hop from its grade. (It also means a market-wide driver like oil_price needs no special case — no company is stored on the driver layer at all.)

The model · part 2 of 5

Five links wire it together — and two of them point opposite ways.

Every connection in the graph is one of just five links. The trick to remember: a fact and its event are joined by two separate one-way links, never one.

LinkFrom → ToMeansWhen
OF_DRIVERDriverUpdate → Driverthis fact is an instance of this driveralways (exactly 1)
FROM_SOURCEDriverUpdate → Eventthis fact came from this event (its evidence)always
EXPLAINED_BYEvent → DriverUpdatethis event's move is explained by this fact the verdict0, or 1 per producer
SAME_ASDriver → Drivertwo names are the same cause (reversible)only on exact-same meaning
INFLUENCES / PRIMARY_FILEREvent → Companythe real stock move already existsexisting pipeline

The key subtlety: two opposite links between a fact and its event

FROM_SOURCE points down (fact → event): "I came from here." EXPLAINED_BY points up (event → fact): "you explain my move." They are separate because a fact can be reported in an event without explaining its move. Never collapse them into one link.

Three more rules baked into the shape
  • No company on the driver layer. The company is reached only via Event → Company (verified: every event has exactly one company). A market-wide driver takes the exact same shape.
  • No BASED_ON link. Trends are queried (each fact's state + numbers carry the value), never stored as links between facts.
  • Same fact in two different events = two DriverUpdates (keyed by event). They're only combined in a read-time view, never merged in storage.
  • If two producers ever judge one event → two parallel EXPLAINED_BY links, one per producer (they can't collide).

This mirrors the production Guidance graph (a class node + per-event instances), already live at 548 Guidance + 8,432 GuidanceUpdate nodes — so the shape is battle-tested.

The model · part 3 of 5 · the contract (LOCKED)

The build makes labels. A producer makes facts. Nothing else.

This is the single most important rule of the whole design. Two jobs, two owners — and they never trade places.

🏭 Catalog build → the CLASS

In one run: read non-news sources + the company's own reported metrics (KPIs) → coin/reuse driver names → as final steps stamp each driver's fact_type + optional links.

It never creates a DriverUpdate.

🤖 Producer → the FACTS

An AI reading a real event (earnings-learner or news-driver) decides the event-level fact and writes the DriverUpdate (+ the verdict, if it moved the stock).

Producers are the ONLY makers of DriverUpdates — live or backfill.

Mention vs fact — the line that splits the two jobs

build sees: "...comparable restaurant sales trends..." → a MENTION → justifies a NAME producer sees: "comp sales ROSE 3% in Q1 vs +1% guided" → a FACT → writes a DriverUpdate

A bare mention never becomes a DriverUpdate. The producer's gate drops it. (Generic risk boilerplate — "litigation could harm us" — is a mention, so it's dropped too.)

Three things that are deliberately forbidden

  • No build-time "seeder." A second thing that invents facts would be a rival source of truth (it lacks the producer's prior-period memory + consensus) → silent errors. Forbidden.
  • KPI-only evidence makes a NAME only — never a DriverUpdate (a KPI has no real event, date, or move).
  • No "needs >2 events" rule. A single event is a valid fact (a CEO resignation, an FDA approval, a guidance cut). "Seen often" is a read-time view, never a write gate.

Want early history? Run the producer over the past (a "backfill")

That's the only correct way to "seed" — same producer, same judgement, one source of truth, zero duplicate logic. A backfilled fact is just flagged produced_mode = backfill so we know it could have used hindsight.

The 8 locked points + the tensions we already settled (so they're never reopened)
  • 1. Build creates the Driver CLASS (name + fact_type + optional links) in one run, never a DriverUpdate.
  • 2. A DriverUpdate = a real event-level FACT (state / change / surprise / guidance / action) — not a mention. (Supersedes the older "= a change" wording: an action like a CEO exit has no X→Y change but is still a fact.)
  • 3. Producers are the only creators of DriverUpdates (live + backfill).
  • 4. KPI-only evidence → a name only.
  • 5. No separate build-time seeder / change-detector (it would be a rival source of truth).
  • 6. No ">2 events" write rule — a single event can be a fact; recurrence is read-time only.
  • 7. Early history = run the producer as a backfill.
  • 8. The verdict (EXPLAINED_BY) is separate: a fact may exist with no verdict.

Tensions logged & closed: "the build already sees the quote → seed a fact for free" → No (a mention isn't a fact-judgement). "Only seed drivers seen >2 times" → dropped. This section is the authority: if any older wording conflicts, this wins.

The model · part 4 of 5

The Driver: a tiny, permanent label.

The class node stays small forever — just enough to name the cause and route everything else. Five fields, two kinds of link.

id
code-built stable key, e.g. driver:same_store_sales (code builds it, the AI never does).
name
the lower_snake cause, e.g. same_store_sales. A SAME_AS head is the canonical name.
created
when the driver first appeared.
definition
a plain one-line meaning. optional
fact_type
the driver's permanent KIND — one of 4 (next slide). Routes everything downstream.

SAME_AS is a link, not an alias list

When two names mean the exact same cause, they stay as two separate nodes joined by a reversible SAME_AS link (a "star": each variant points to one head; the head itself has no link out — it is its own canonical; no chains; the head never moves; nothing is ever deleted).

customer_transactions ──SAME_AS──→ guest_count (the head) comp_sales ──SAME_AS──→ same_store_sales

Why not a flat aliases array? Because each variant keeps its own evidence and its own facts. An array would flatten that away. Trading code just follows SAME_AS to the head and unions the whole star (every variant's evidence + DriverUpdates).

Optional links — the driver's "wiring to the books" best-effort

Three optional links on the class (never per-fact): MAPS_TO_CONCEPT → an XBRL accounting concept · MAPS_TO_MEMBER → a segment/brand · MAPS_TO_GUIDANCE → a company-guidance metric. They're attached only on a confident match, never block the catalog, and self-heal on re-run.

Fine print: what's deliberately NOT on the class
  • No evhash16 (value-hash) on the class — a label is never re-extracted, so there's nothing to change-detect. (The fact and verdict carry one; the label doesn't.)
  • MAPS_TO_GUIDANCE is one-to-many — one driver can union several fragmented guidance synonyms (e.g. revenue_guidance → both guidance:revenue and guidance:net_sales). More on this in slide 39.
  • Links are MATCH-only (a bad guess makes no link, never a new node) and use set-replace on re-run (stale links are deleted, not left behind).
  • The state, numbers, quote, and scope live on the DriverUpdate (the instance), never on the class. ("scope" = which version of the fact — a period/segment; defined on slide 36.)
The model · part 5 of 5 · LOCKED 2026-06-15

Every driver has exactly ONE permanent kind.

fact_type is the driver's KIND, set once at build time. It does one powerful job: it routes which "what happened" words are even legal for that driver (slide 32).

fact_typethe driver is about…examples
metricany standing variable you can re-read over time — a number, cost, rate, ratio, count, or a qualitative condition (weather, sentiment, policy in force)same_store_sales · commodity_cost · consumer_sentiment · tariff_policy
guidancethe company's own forward outlook / target / forecasteps_guidance · revenue_guidance
surprisean actual result vs an expectation (consensus or the company's own prior guide) — not vs last periodeps_surprise · revenue_surprise
action_eventa discrete thing that happened — a decision, deal, incident, approval, one-off chargedividend · restaurant_closure · ceo_succession

The one test that splits metric from action_event

Between two events, is there a standing level you could re-read ("what is it now vs before?")?

YES → metric ("what is the cost now?") e.g. commodity_cost NO → action_event ("did it happen / get undone?") e.g. asset_impairment

A _surprise or _guidance framing overrides this to those kinds.

Dual framing is allowed

The same topic can be two drivers of different kind, decided by the name: dividend (action_event) vs dividend_per_share (metric).

Who sets it & when

A strong model (Opus), as a final build step, once per driver (never per event). Cheap, because it runs once. The blind reader never sets it.

Why a routed kind, and why we won't add more rules
  • Why not one flat up/down list? ~1 in 6 real facts have no up/down (risk talk, plain snapshots, "stabilized", guidance ranges). Routing by kind handles them cleanly.
  • Why distinct kinds matter for trading: each is its own query signal — "show me all withdrawn guidance" = bearish — so we deliberately do not collapse them.
  • Validated on all 1,282 real driver names — 0 fit none; every "merge to 3" or "add a 5th" attempt failed for a concrete reason.
  • Names that read both ways → default to the standing bucket: litigation, convertible_notes, dividend_policy, restructuring_costs → metric; corporate_restructuring, asset_impairment → action_event.
  • Keep the 4 definitions exactly as written — add no extra clauses or examples. A tested extra clause overfit (fixed two cases, broke one). Opus was stable on 28/28 keyed cases; a cheap model wobbled.
  • It's a precondition: a driver with no fact_type yet → any DriverUpdate write is rejected.
Big picture

Build from the bottom up, like merging folders.

Start at one company. Merge companies into an industry. Merge industries into a sector. Merge sectors into the global list. The same merge engine runs at every level.

GLOBAL catalog <-- merge all SECTORS ▲ ┌---------------┼---------------┐ SECTOR SECTOR SECTOR <-- merge each sector's INDUSTRIES ▲ ┌------┼------┐ INDUSTRY INDUSTRY INDUSTRY <-- "the leaf": blind bots + reconcile ▲ ┌-----┼-----┐ co. co. co. <-- one company's filings & transcripts

New names: only at the bottom

The only place names are invented is the leaf (one company at a time). Every level above just consolidates — it never invents new drivers (one rare exception: splitting a true homonym).

Keep each step small

Each box must be small enough for an AI to judge carefully. If a box is too big, code splits it into batches. Levels are just batches of batches.

Real shape of the full tree: 11 sectors → 115 industries → ~796 companies.

The unit

Everything flows as one simple shape: a "record."

One record = one driver, plus the real quotes that prove it. This is the only shape in the whole system.

{
  "driver_name":     "same_store_sales",     // the cause, as a plain lowercase noun
  "canonical_name":  "same_store_sales",     // itself, OR the name it merged into
  "companies":       ["CAKE"],               // who has this driver
  "evidence_refs": [                          // WHY it exists — real quotes
    { "company":"CAKE", "source_type":"10-K",
      "source_id":"0001104659-26-018643", "date":"2026-02-23",
      "quote":"...comparable restaurant sales trends." }
  ],
  "same_as_variants": [],                     // other names that mean the same thing
  "optional_links":   { "xbrl_concept": null, "xbrl_member": null, "guidance_ref": null }  // the 3 links from slide 8, flattened into the record
}

The atom: 5 fields that ID one quote

Each piece of evidence is identified by five fields together: company + source_type + source_id + date + quote. Code dedups, sorts, splits, and verifies everything by these five fields. If you remember one thing, remember this is the unit of truth.

This is the build artifact — the catalog file. In the live graph it becomes the Driver node (slide 8): same_as_variants turn into SAME_AS links, and canonical_name just marks the link's head. Notice what's not in the name — no direction, number, or date; those live on the DriverUpdate (slide 31).

Step 2 · Fetch

Pull every company's real words — and clip nothing.

For each ticker, grab all the non-news sources with full text. fetch_company_sources.py

What gets pulled

Earnings call transcripts (prepared remarks + every Q&A) · 8-K press releases · 10-K / 10-Q business, risk, and management-discussion sections · the company's reported KPI names.

The big change: stop dropping text

The old version had size caps that quietly clipped ~62% of the text (and 100% of transcripts). That's a huge recall loss. Now it pulls everything. One real company (CAKE) = 2.24M characters of source text.

"News" is deliberately not in this seed — news drivers get added live later (slide 38). The seed is built only from what the company itself reports.

Step 3 · Chunk

An AI can't read 2 million characters. So split — never clip.

Cut the text into ~40,000-character pieces, always at a clean seam. chunk_company_sources.py

The fallback ladder (it always finds a cut point)

(a) pack whole small events together (b) too big? → split at natural seams (a section, or one Q&A exchange) (c) still big? → split by paragraph (d) still big? → split by sentence (e) still big? → split by raw character count ← always works

The promise: not one byte lost or changed

Glue the pieces back together and you get the original text, exactly — checked by a hash. The chunker only slices; it never rewrites. So splitting big filings costs us nothing.

Step 4 · Menus

One blind AI bot per piece — the only place names are born.

Each bot sees only its one chunk — no other company, no shared list. It reads the real text and coins specific driver names.

Real example (a CAKE bot)

raw KPI label: "North Italia Comparable Restaurant Sales" | bot applies the naming rules v driver_name: north_italia_same_store_sales evidence_quote: "North Italia Comparable Restaurant Sales" source_type: fiscal.ai-kpi

Naming rules (plain)

  • Name the cause only, as a noun: oil_price, restaurant_traffic.
  • NO ticker, date, number, or up/down word in the name.
  • Vague text → skip (never invent).

Why "blind"?

If a bot saw the shared list first, it would lazily reuse a near-match. Blind naming + careful merging later is what makes two independent bots actually agree — instead of being nudged into fake agreement.

The full naming rulebook (the fine print)
  • Banned inside a name: state words (beat, cut, rose), direction (long/short), dates/quarters, numbers/units (bps, %, usd), any company ticker or person's name, source labels, accounting prefixes, metaphors, and bare category words (demand, macro, sentiment alone).
  • Earnings convention: name the metric + its mechanism — eps_surprise (reported vs expected) or revenue_guidance (forward outlook). "beat / raised" is a state, not part of the name.
  • Brand ≠ company-wide: taco_bell_same_store_sales is its OWN driver, separate from same_store_sales. They are never merged.
  • Reusable class, even if seen once: government_shutdown is fine. A name glued to one single event (q1_2026_shutdown_effect) is rejected.
  • Two causes in one quote? Emit two driver names, never one bundle.
Anti-cheat: the bot writes its menu to a file too

Each bot both returns its menu and writes it to disk. A later count cross-check fails the run if the two disagree — so a bot can't quietly return something different from what it saved.

Step 1 · Guard & slice

Don't get billed wrong, and don't overload the AI.

Billing guard (step 0 of every workflow)

Check: is an API key sitting in the environment? If yes → refuse to run. Everything must go through the subscription, never the metered pay-per-call API. (A hard rule from your setup.)

Slice into batches slice_seed.py

594 records is over the safe limit (400 records / 300k chars). So code cuts it into name-sorted batches so each AI judge reasons over a small, careful pile.

Trade-off: two synonyms with no shared words might land in different batches and miss each other. That's an under-merge (the safe direction) — and the repair pass (slide 29) catches it later.

Step 2 · Two judges

Two independent AI agents, working in parallel.

┌-- DEDUP → "which names mean EXACTLY the same? → propose merge links" seed batch --┤ └-- GATE → "is each name a valid, reusable driver? → admit / rewrite / skip"

Dedup — the matcher

Proposes "these two are the same" links and picks one survivor name. Different brand, scope, or metric → rejected, kept separate.

Gate — the bouncer

An independent judge (doesn't trust the bot that coined it):

  • admit — valid driver
  • rewrite — fix wording only
  • skip — vague or one-off

Both judge from the evidence quotes, not the bare name string.

The "3-check" every merge must pass

To call two names the same, all three must be true: same object (the thing) · same scope (brand vs company-wide, geography) · same mechanism (how it acts). If any one is false or unclear → keep separate. A "rewrite" may only fix wording — never change the meaning.

Step 3 · The skeptic

A third agent whose only job is to break the merges.

This is what stops the v2 collapse. For every proposed merge, the default answer is NO. It flips to YES only if the skeptic cannot refute that they're the exact same object + scope + mechanism.

Then code — not AI — filters

Refuted links are dropped. Refuted rewrites are parked. A missing answer counts as refused. Fail-closed, always.

High-blast: a second skeptic for big merges

A wrong merge across 2 companies is small; across hundreds it's a disaster (exactly how v2 died). So any surviving merge touching ≥ 8 companies gets a second, independent skeptic that must separately pass object · scope · mechanism, each backed by a quote. Either one says no → the merge dies.

Why the company count is computed by code

Seven rules keep the numbers honest — four examples show them.

Code enforces these at write time: the producer proposes the numbers, code decides if they're legal and rejects anything off.

The 7 hard rules (plain)

  1. Open bounds null the other end. floor → no level_high; ceiling → no level_low.
  2. A stated delta's sign must match the state. + for increased/raised/beat; − for decreased/lowered/missed. (states with no direction — reported, persists, unchanged — impose no sign rule.)
  3. Store a baseline only if the source says it (a frozen fact of this event) — never compute it from another node. Set it null for "vs peers", a fixed anchor year, or a streak ("third straight quarter"); if a stated baseline fits no menu value, keep the number but set the baseline null.
  4. Per-share / per-week lives in the NAME (eps, sales_per_square_foot); the unit stays the base (usd / count).
  5. One primary baseline only. If a fact cites several, keep the headline one (tiebreak: prior_year > sequential_period); the rest stay in the quote. "Exceeded expectations" reads by kind — a surprise → consensus; a metric/guidance "beat our guide" → previous_guidance.
  6. Store change_value only if the source states the move. If it's derivable from level + baseline, leave it null (no drifting third copy); a ranged move stays in the quote (there is no change_high).
  7. Rate-vs-level routing. A % that is the metric goes in the level; a % move on a $-metric goes in change.

The verdict: did this fact actually move the stock?

A fact can exist with no verdict. But when the producer judges that a fact drove the move, it clips a verdict onto the Event → DriverUpdate link (EXPLAINED_BY).

(Event 8-K) ─[:EXPLAINED_BY]─► (DriverUpdate: same_store_sales increased 3%)

Three independent dials — none derivable from another

stock_impact = direction
long or short. Which way this fact pushed — it can push opposite to where the stock actually went.
weightage = force
how strong this one driver was, 0.1…1.0. A standalone estimate — not a share of the move.
confidence = certainty
how sure the producer is the attribution is true, 0…100. Separate from force.

Why weightage is independent (this is the subtle part)

Forces are never summed to 1 and adding a new driver never rewrites another's weightage. A move can be partly unexplained (market / sector / noise) — so a sole driver is never auto-set to max. Unsure of the size? Set weightage = null (abstain) — keep the direction.

Shares are computed only when you read: share = weightage ÷ (sum of weightages on that event); signed_force = weightage × direction. Neither is ever stored.

Point-in-time, and graded fairly

The real share / actual return is never stored on the verdict and never shown to the predictor. Reality gives one net return, so a verdict set is graded on net sign/size + ranking — never a per-driver "true share." A live verdict (judged before the move) always beats a backfill one at the same key.

The verdict's identity & validator
  • Key = event + driver + fact_scope + producer. (fact_scope = which fact within the event; slide 36.) The producer is in the key, so two producers can disagree without colliding. No company (it's reached via the event).
  • evhash16 = a hash of exactly (stock_impact, weightage, confidence). produced_mode is excluded — a live↔backfill swap isn't a judgement change. So the hash flips only on a real change to this edge.
  • Validator (hard-fail): stock_impact ∈ {long, short}; confidence ∈ {0,10,…100}; weightage ∈ {0.1,…1.0} or null. Both sit on a 10-step ladder — no false-precision decimals, so the hash stays stable across re-runs.
  • The name: we call it the attribution (chosen over "impact"); there is no separate verdict box — it's purely the edge.

(a) exact slug (deterministic), then (b) an evidence judge that reasons from each anchor's unit + sample values, never its name, with a Refute step — fail-closed on any doubt. No curated synonym list (proven unsafe on live data).

Measured, honestly

Offline test: ~98–100% precision (0 wrong links on 87 hard pairs), ~85% coverage. There is no fully-deterministic 100% shortcut — synonym-matching is part-judgement.

The safety asymmetry (why fail-closed)

A missed link is a safe gap (just a short series; fixable later). A wrong link is the dangerous, hard-to-undo case. So the resolver always errs toward leaving things unlinked.

The fuller idea (resolver, set-replace, born-linked pointer, open decisions)
  • Why it's even possible: a Guidance node is itself a thin, cross-company, value-free metric tag (a class) — so Driver↔Guidance-anchor is class↔class, and our catalog supplies the canonicalization guidance never enforced.
  • Why no auto alias / auto XBRL-code match: both are unsafe on live data (227 alias→wrong-anchor collisions; one XBRL code spans 9–11 anchors incl. GAAP + non-GAAP). The evidence-judge reproduces every true synonym from data, so a list is redundant.
  • Set-replace every run: delete links no longer matched, then write the current set (additive-only would leave stale links).
  • Optional back-pointer: a derived canonical_driver on the Guidance anchor (class-level, never relabels guidance) so a guidance-only tool can find the driver fast. The edge alone is required + sufficient; the pointer is pure convenience and must be re-derived from the edge each run.
  • Instance-level link (DriverUpdate ↔ GuidanceUpdate) is producer/backfill work, deferred.
  • 4 open owner decisions remain (keep the judge vs slug-only; one-to-many edge; defer instance link; adopt the back-pointer or stay strictly read-only).

⚙️ Tool 1 of 8

resolve_driver_scope.py

List the companies. Given an industry name, it asks the database which tickers belong to it.

Takes in
--industry "Restaurants" (or --sector, or --list to see valid names)
Gives back
a list of tickers, and writes them to a small scope file

Step by step

  1. Connect to the company database (read-only — it never changes anything).
  2. Ask: "which companies have industry = Restaurants?"
  3. Return the distinct tickers, sorted: BLMN, CAKE, CMG, DRI …
  4. Write that exact list to a file the next tool reads directly.

Its two guards

0 companies found → stop the run (you typed the name wrong). Only 1 company → warn (you can't measure agreement with one). Writing the list to a file means an AI can never quietly drop a company.

⚙️ Tool 2 of 8

fetch_company_sources.py

Pull the text. For each company, download all of its real words from filings and earnings calls.

Takes in
the ticker list + a run folder
Gives back
sources/<TICKER>.json (full text) + a list of fingerprints

Step by step

  1. For each company, query the database for its 10-K / 10-Q / 8-K filings, earnings-call transcripts (prepared remarks + every Q&A), and its reported KPI names.
  2. Keep the full text — the old version's size caps clipped ~62% of it.
  3. Tag each item with its date, type, and whether the stock moved hard that day.
  4. Save each company's bundle, plus a fingerprint (sha256) to prove it wasn't tampered with.

Why it matters

This is the raw material for the whole catalog. Read-only, nothing dropped. A driver missed here can never be recovered, so it errs on the side of grabbing everything.

One small bug it fixes

Q&A exchanges are numbered "0","1",…,"10" as text. Sorting text puts "10" right after "1" → scrambled order. This tool sorts them as real numbers, so the conversation reads in the right order.

⚙️ Tool 3 of 8

chunk_company_sources.py

Split the text. One company can be 2 million characters — too big for an AI. So cut it into ~40,000-char pieces, always at a clean seam.

Takes in
sources/<TICKER>.json
Gives back
chunks/<TICKER>__chunk_NNN.json — one bot's reading load each

Step by step

  1. Build each event's full text.
  2. Small enough? Keep it whole. Pack several small events into one chunk.
  3. Too big? Walk the ladder until it fits: natural seam → paragraph → sentence → raw characters (this last one always works).
  4. Run --verify: glue the pieces back and confirm they equal the original, byte for byte.

Why it matters

It lets a bot read everything without losing or changing a single byte. It only slices; it never rewrites. So even a giant 10-K costs us no text.

⚙️ Tool 4 of 8

build_seed.py

Group into the seed. After the blind bots name drivers, this code gathers all their menus into one raw draft — no AI, just bookkeeping.

Takes in
menus/*.json (one file per bot)
Gives back
seed.json — the deduped driver records

Step by step

  1. Read every bot's menu file.
  2. Group candidates by their lowercased name (exact matches only).
  3. Merge their evidence quotes; drop exact-duplicate quotes; list the distinct companies.
  4. Cross-check: do the counts match what the bots reported? If not → fail.
  5. Check every chunk produced a menu — no chunk silently skipped.

Real example

2,246 raw candidate names → 594 distinct drivers

It only merges exact spelling matches here. Real synonyms (guest_count vs customer_transactions) are left for the careful judges in Stage 2.

⚙️ Tool 5 of 8

slice_seed.py

Batch a big seed. 594 drivers is too many for one AI to weigh carefully. So cut the seed into smaller piles.

Takes in
seed.json
Gives back
seed_batch_001.json, _002, …

Step by step

  1. Walk the records in name order (they're already sorted alphabetically).
  2. Start a new batch when the current one hits 400 records or 300,000 characters.
  3. A single record that's bigger than the limit goes alone — a record is never split in half.

Why it matters (and the small cost)

Each AI judge then only ever sees a small, careful pile. The trade-off: two synonyms with no shared words can land in different batches and miss each other — an under-merge (the safe direction). The repair tool (slide 55) sweeps those up later. If the seed is already small, it just makes one batch.

⚙️ Tool 6 of 8

assemble_catalog.py

Write the catalog. This is "the writer." The AI only produced short decision lists; this code turns them into the actual file.

Takes in
seed.json + decisions.json (the AI's approved lists)
Gives back
catalog.json + approved.json

Step by step — the fixed 5-way rule, per driver

1. gate said skip? → skips list 2. approved merge? → point it at the survivor name 3. approved rewrite? → point it at the fixed name 4. parked rewrite? → unresolved list 5. otherwise → keep as its own driver

Then it fills each survivor's same_as_variants (so trading reads the whole group in one hop).

Why it matters most

Because code writes the file from an approved list, the AI cannot invent a merge the skeptic never approved. This is what makes the old v2 mush structurally impossible. It also hash-checks that the bytes the AI typed match what the workflow intended.

⚙️ Tool 7 of 8

validate_catalog.py

The hard gate. The last check. It re-inspects the structure and stops the run if anything is wrong. Zero judgment — pure checks.

Takes in
seed.json, catalog.json, approved.json
Gives back
PASS / FAIL + a signed receipt (validation_exit.json)

What it checks

  • Every merge traces back to an approved link (it can't justify one → fail).
  • Nothing lost or duplicated — every seed name accounted for exactly once.
  • No invented names; the catalog's quotes match the seed's quotes exactly.
  • It recomputes the "≥8 companies" list itself and demands the second-skeptic proof exists.

The receipt

It writes a signed note: "passed, and here's the catalog's fingerprint." Later steps refuse to use any catalog unless its receipt says passed and the file hasn't changed since. This catches "the check never actually ran" and "someone edited it afterward."

⚙️ Tool 8 of 8

fold_catalogs.py

The merge math. The no-AI engine behind merging levels up. It has three small commands.

Its three jobs

part-a
tidy each child catalog, then group names across children. A name in 1 child → keep. A name in 2+ children → queue it for the AI (same spelling ≠ same meaning).
draw
build a fair sample of quotes for each queued name — smallest side first — so the AI sees the disagreement.
part-b
apply the AI's verdicts: SAME → merge, DIFFERENT → split, UNCLEAR → park. Write the parent seed.

Why it matters

It does the heavy, error-prone bookkeeping (union evidence, partition splits, sort, count) so the AI only makes the meaning calls. It refuses to fold a child whose validation receipt isn't passing-and-unchanged, and records every kept/dropped count so nothing vanishes silently.

🧠 Conductor 1 of 7

menu_build.js

Runs the leaf. The conductor for Stage 1: turn one industry into a raw seed. It calls the worker tools in order and runs the blind bots.

resolve → fetch → chunk → [blind bots name drivers] → build_seed → record

Step by step

  1. Billing guard: refuse to run if a pay-per-call API key is present.
  2. Call resolve → tickers, fetch → text, chunk → pieces.
  3. Spawn one blind AI bot per chunk to coin candidate driver names (this is the only AI work it does directly).
  4. Call build_seed (code) to group them, then save a manifest of proof.

Run it like this

menu_build.js with { industry: "Restaurants" }. If a long run dies, restart with resume_run_id → it re-runs only the missing bots, never re-pays for finished ones.

🧠 Conductor 2 of 7

reconcile.js

The merge engine. The conductor for Stage 2: clean a messy seed into a real catalog. The same engine runs at every level later.

guard+slice → (dedup + gate, in parallel) → skeptic → assemble → validate

Step by step

  1. Guard (billing) + slice the seed into safe batches.
  2. Per batch: run two judges in parallel — dedup (find same-meaning pairs) and gate (admit / rewrite / skip).
  3. Skeptic tries to break every proposed merge; a second skeptic checks merges touching ≥8 companies.
  4. Hand the short approved lists to assemble_catalog.py (code writes the file).
  5. Run validate_catalog.py — fail the run if anything is off.

Why it matters

This is the brain that decides what merges — but it never writes the file itself. It only produces a list of approved decisions for the code to apply. That separation is the whole safety story.

🧠 Conductor 3 of 7

fold_catalogs.js

Merges levels up. The conductor that wraps the merge-math tool with the AI's same-name review — to fold several child catalogs into one parent.

part-a (code) → draw (code) → review (AI) → refute (AI) → part-b (code)

Step by step

  1. Run part-a + draw (code): find the names that appear in 2+ children and build their evidence views.
  2. For each clash, one AI reviewer says SAME / DIFFERENT / UNCLEAR.
  3. A skeptic attacks every SAME; big ones get a second skeptic.
  4. Run part-b (code): apply the verdicts and write the parent seed.

The output

A parent seed.json — which then goes through the exact same reconcile.js engine, one level up. That's how the tree climbs.

🧠 Conductor 4 of 7

build_tree.js

The master conductor. Ties everything together and climbs the whole tree: company → industry → sector → global.

Its three modes

{ list: true }
Read-only: print the Sector → Industry → Ticker tree and fail loudly if any company has two parents. Never merges.
{ fold: {…} }
Do one merge: fold these children → reconcile → fold-check → repair → fold-check.
{ walk: {…} }
The full climb: fold every sector, then the global. Every step fail-closed.

The one rule it enforces

For each level it runs fold → reconcile → fold-check → repair → fold-check, and one failure anywhere kills the whole run. No level can be skipped, no "latest" guessing — you pass exact run IDs.

🧠 Conductor 5 of 7

repair_duplicates.js

Catches missed merges. A second sweep for duplicates the batching split apart. Runs after each level.

suggest (code) → review (AI) → apply (code) → validate

Step by step

  1. Suggest (code): find look-alike pairs by shared rare words, or by cheap similarity scores (embeddings). Suggestions only.
  2. Review (AI): judge each pair SAME / DIFFERENT — same skeptic discipline as everywhere else.
  3. Apply (code): add the approved links through the same writer, then validate.

Why it's safe

It only ever adds a link between two kept drivers — it never breaks an existing link and never touches the parked items. So it can only ever tidy up, never damage.

"Embeddings" = a cheap math score of how similar two names feel. Here it only points the AI at pairs to look at; it never decides a merge.

🧠 Conductor 6 of 7 · live use

gate.js

The admission gate (called "G2"). The single bouncer for any new name — used both inside reconcile and, later, live.

Takes in
candidate names (with their evidence) + the existing catalog
Gives back
one verdict per name: reuse / admit / rewrite / skip

The one test it applies

"Is this a valid, reusable driver?" — judged from the evidence, not the bare name.

  • reuse — it's the exact same cause as one already in the catalog
  • admit — a new valid driver
  • rewrite — right driver, just fix the wording
  • skip — vague, or tied to one single event

It's deliberately "producer-agnostic" — it never tags a driver as "news vs fundamental." That's a job for whoever uses the catalog, not a property of the driver.

🧠 Conductor 7 of 7 · live use

catalog_first.js

Reuse-first (called "G1"). The live mechanism for handling a brand-new event: name it yourself first, then reuse only if it's truly the same.

coin your OWN name (blind) → see the related catalog names → reuse if EXACT, else create / skip

Why "name it yourself first"?

If you saw the catalog up front, you'd lazily grab a near-match and slowly blur the meanings. Naming blind, then comparing, keeps everyone honest — the same anti-laziness trick as the blind leaf bots.

Status: not wired yet

This file still holds an older version of the flow (it shows the catalog up front). The new "name-it-first" rules — and exactly how related names are shown — are still being decided. The file header literally says: raise to the user before using. So it's the one piece here that is design-in-progress, not built.
