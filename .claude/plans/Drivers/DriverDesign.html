One index card per cause.
Everything else is a rule protecting it.

The whole Driver design, decision by decision — what we believed before, what we decided instead, and why. No wall of text: every rule is one card.

§1

The one law everything bends around

If you remember one thing, this is it. Every other rule exists to serve it.

MERGETwo different meanings glued into one driver. The evidence is now mixed. You cannot tell which quote meant which thing. Permanent.
SPLITOne meaning kept as two drivers. Ugly, findable, joinable later with a reversible link. Repairable.

The two mistakes are not equal, so the system is deliberately lopsided: when unsure, keep them separate. Nothing is ever deleted or re-keyed to make history look tidy.

§1
What a driver is
NOTa metric, a number, a company's business line
ISone reusable cause — a thing that can move a stock, written once, shared by every company

Reusable is the point. fed_rate means the same thing for a bank and an airline. If a company's own name leaked into it, it couldn't be shared.

§0§2
Driver vs DriverUpdate — the split that makes it work
DRIVERthe class. Name + permanent kind + family links + birth evidence. Never changes.
UPDATEthe fact. One real occurrence, in one source event, with values + a quote.

Think index card vs. sightings. The card revenue exists once. Every 8-K that mentions revenue adds a sighting, never a new card.

§1FS-26
Recovery is always possible — by design
BEFOREmerge two drivers → the losing name is gone forever
AFTERboth survive; a reversible SAME_AS link says "these are the same"

A link can be cut. A deletion can't. So the system links instead of deleting — that's what makes over-splitting cheap and keeps the law honest.

§1
What the writer actually promises
CLAIM"over-merge is impossible" — we never claim this
CLAIM"conflicting values never silently overwrite each other" — this is guaranteed

Honest scope. Judging whether two meanings match is a judgment call with a measured error rate, guarded by judges and audits — not a proof. The storage guarantee is absolute.

This is what makes the whole thing auditable. You can always ask a number "who said you?" and get a sentence back.

§3

Naming — 19 rules guarding one sentence

The sentence: the name contains the cause only. Everything the 19 rules do is push something else out of the name — state, size, time, company, unit, measurement — so the card stays reusable.

§4

Four kinds, four state lists, one birth

Every card is permanently one of four kinds. The kind decides which states are legal and which fields are forbidden. It's stamped once and can never change. The state lists below are the exact vocabularies (DU-08..DU-12) — driver_state lives on the fact, never in the driver's name, and code hard-fails a state outside its lane.

kindthe card is about…legal states
metric a standing variable you could read again later — number or condition (weather, sentiment, labor, brand) increased · decreased · unchanged · mixed · reported · persists · unknown
guidance the company's own forward outlook introduced · raised · lowered · reaffirmed · withdrawn · unknown
surprise a company value vs a cross-party expectation beat · in_line · missed · unknown
action_event a discrete happening at_risk · announced · occurred · continued · resolved · canceled · suspended · rumored · failed · unknown
DU-06
The persistence test — metric or action?
ASK"is it a number?" — wrong question
ASK"between two events, is there a standing level I could re-read?" Yes → metric. No → action.

"Is it a number" fails both ways: weather is a metric with no number; a $2B buyback has a number but is an action. Dual framing is fine — dividend is an action, dividend_per_share is a metric.

DU-05DU-07
The surprise definition — three near-misses it excludes
NOTvs a prior actual → that's a metric change
a new guide vs own prior guide → that's a guidance movement
ISa delivered actual or a promised guide, vs a cross-party expectation (Street, or own prior guide for an actual)

"Cross-party" is the whole idea — a surprise needs two parties disagreeing. You beating your own last quarter isn't a surprise; it's growth.

DU-07
The prompt is frozen, word for word
BEFOREimprove the classifier prompt by adding a helpful clause or example
AFTERcopy it verbatim. Add nothing.

Not superstition — a tested extra clause overfit. The prompt text is stored in the rulebook as law, not as a suggestion.

MF-01
The type is permanent
BEFOREre-type a driver later when catalog work learns better
AFTERstamped once at admission. A driver with no type cannot accept a fact. Fact-bearing → never re-typed.

Re-typing would silently rewrite the meaning of every fact already stored underneath it.

OD-1
Suffix admission — ask twice, both must say yes
BEFOREname ends in _guidance → it's guidance. Done.
AFTERstrip one terminal suffix, then ask the one meaning question twice, independently. Any NO / UNCLEAR / disagreement blocks.

Admits bookings_guidance. Rejects fda_guidance — that's a document, not a guided metric. Mid-name words never count (fda_guidance_issuance). Stacking is invalid (revenue_guidance_surprise). The verdict is frozen in a memo — refreshes reuse it and never re-roll the dice.

OD-2

BEFOREclassifier says "metric" → stamp metric
AFTERmetric must pass a challenge with a quoted phrase. Anything less → action_event + a counted warning.

Asymmetric on purpose: a wrong metric invents a series that was never measured; a wrong action_event is visible and blocks less. Owner's own worked examples: buyback → action · dividend → action · bookings → metric only if the quote proves a standing measure.

OD-2F2
The family gate — you can't guide an unproven thing
BEFOREbuyback_guidance → buyback. Fine, both exist.
AFTERa family may attach only to a base that is metric AND proven — deterministically, by memo, or as an OD-1 latent anchor

Otherwise a bad metric stamp propagates into a whole family of guidance and surprise cards hanging off a thing that was never a metric.

§4.2

Name-only seeding is rejected outright. One real fact in one event is enough — recurrence was never a gate. The only legal empty node is a latent base anchor.

§5

Identity — what makes two facts the same fact

Get this wrong and you either overwrite a real number or store the same number twice. The answer: identity is computed by code from three things, and never includes who produced it.

FS-01
The producer is never in the key
BEFOREthe fact's ID includes which model/producer made it
AFTERidentity = event + driver + fact_scope. Nothing else.

Otherwise swapping models duplicates your whole history. Two channels reading the same 8-K must land on the same ID — that's what makes them converge instead of collide.

FS-27
Never slice=total
BEFOREwhole-company → write slice=total
AFTERwhole-company → omit the slot entirely

"Total" is a word companies also use for real sub-populations — so it would collide with a genuine slice. Absence is unambiguous; a word isn't.

OD-21
The surprise type is composed by code, before fusion
GUESSperiod already ended → must be an actual
COMPOSEsurprise_basis_hint (actual|guidance) × comparison_baseline (consensus|previous_guidance)

Never infer basis from whether a period ended — companies guide on ended periods all the time. Required on surprise facts, forbidden everywhere else.

§5.1
Every surprise needs a home, in the same event
BEFORE"results beat expectations" → store a surprise
AFTERungrounded → park. A surprise must sit beside its own home fact.

Six things must match: family, period, period scope, slice, measurement, normalized value/unit. An actual surprise before its period ends is rejected outright. A missing home re-extracts the whole event — never an orphan-only replay.

FS-04

FS-20
Why the elimination guard is a list, not a regex
REGEXmatch /elimination/i → also eats GlobalPestElimination — a real business (~20% false positives)
LISTSthree frozen exact buckets — ~24 hard-exclude · ~241 provisional · ~3,000 keep

A missed elimination lands in keep — an over-split, which is safe. A regex's false positive deletes a real segment. Every exclusion is logged, and one that later shows real activity auto-demotes to provisional (FS-20).

FS-05
Judge an axis by its members — names lie
BEFOREread the axis name to decide its kind
AFTERjudge by its members — names lie ~20% of the time

The axis→kind table is frozen in code and refreshed only by a governed offline pass. At runtime, three outcomes: known slice axis → mapped · known non-slice → skip · unknown → PROVISIONAL, never silently dropped.

FS-18
Code dedupes exactly — or not at all
BEFOREnear-match snap: iPhone ≈ iPhones ≈ iphone_products
AFTERexact after normalization folds in. Never a fuzzy snap.

The split again: code does exact, the producer does meaning — it's shown existing values first and coins a new one when unsure. The model assigns; it never merges two existing identities.

FS-06

BEFOREoffer every slice value we know today
AFTERwrite menus cut at the event's public time — prior public 10-Q/10-K members only

Otherwise the past gets labelled with the future and every backtest is quietly poisoned. Naming has no menu at all; repair may see full history.

§5.4

Continuity — when a renamed thing is still the same thing

The hardest question in the system, answered by refusing to answer it unless the company said so.

OD-20
Only the company may declare continuity
BEFOREa detector notices the old label vanished and the new one appeared → link them
AFTERcreated only when company text explicitly says the old continues as the new, composition unchanged

There is no standing detector. "Recast", "recompose", "reclassify" trigger a code pre-refusal — those words mean the numbers changed shape, so the series must break. A dedicated blind judge (not the sameness judge) confirms, or it fails closed.

OD-20
Three determinism guards
fan-outone old → two new = refused fan-intwo old → one new = both quarantined cyclea loop-closing link = auto-refused

All three are state-based and reversible. A chain that forks isn't a rename — it's a split, and following it would silently fuse two histories.

OD-18
Flagged attach — same word, two industries
BEFOREexact name match → attach
AFTERrisky attaches get a blind three-check vs the head's frozen birth anchor: same cause · same causal scope · same mechanism

The trigger everyone misses: same-industry heterogeneity. Two software firms can both say "churn" and mean subscriber vs employee. REFUSE can never be overturned toward a write; UNSURE gets one blind escalation; only CONFIRM writes; a timeout parks.

OD-19
A rule that ships switched off
TEMPTINGlet the judge decide generic token-subset matches (rpo vs current_rpo)
INERTstays off until the K-pairs.v2 gate records zero wrong-same

Cross-flavor, terminal-suffix, per-X and OD-17 portion differences are permanent refusals — no gate ever unlocks those. A designed rule sitting deliberately dormant is what "prove it first" looks like.

§5.4
Recovery — mark, never erase
BEFOREfound a wrong link → delete it and move the facts
AFTERsignal → two blind strong graders → flip quarantined=true + a RecoveryEvent. A wrong fact is marked disputed.

Never delete, re-key, or move history. Disputed facts drop out of cross-company and history-weighted features until cleared — they stop influencing decisions without ever being erased.

§5.4
The deepest worry, stated out loud
XBRL-BACKEDa falsifier tripwire watches the boundary — a concept split or opposite-direction jump auto-quarantines it
NO XBRLno tripwire exists. Documented as the design's deepest worry — watched only by drift probes and audits.

This card exists because the design names its own worst residual risk instead of hiding it. A false continuity with no structural backstop is the one thing nothing can automatically catch.

OD-11

BEFOREstore plain percent and lose the basis

OD-11

Two names for one thing would split one series in half. Also pinned: a bare "up X%" on a static-percentage metric (like a margin) is unknown unless it says points/bps or "to X%" — because "margin up 2%" is genuinely ambiguous.

OD-12
The signed axis — loss is a negative number

The traps are pinned by golden fixtures: "up to a loss" is a floor, not a ceiling; "loss narrowed" is increased. Charges are positive, benefits/releases/reversals negative. And low ≤ high catches the flipped spoken order for free.

PER-18

§7.1

A stored derivation is a second copy of the truth that can drift from the first. Stored only when the source states a non-derivable delta and its sign is determinable — "beat by X" with unclear polarity stays null and the quote carries it.

The trigger is the producer detecting the comparison at extraction — never a read-back of the stored baseline enum. (Reading back would make the write depend on what was already written.)

§7.3

DU-22

§8

XBRL enrichment — attach the exact concept, or attach nothing

Linking a text fact to the company's own filed accounting concept. The entire design is built around one asymmetry.

WRONGSilent until detected. The link is revocable by design — but reads and decisions already made on it are not undone.
ABSENTSelf-heals on the next re-run, at zero risk.

Missing links are safer than wrong links. That single line explains why every layer below can only abstain, never create.

XC-04

§8

§9

Reading — collapse, point-in-time, and the one derived write

Facts go in append-only and messy. The read layer makes them a series — without ever rewriting them.

§9
Never group by the quote
BEFOREcollapse duplicates by comparing quote text
AFTERthe comparator is the numeric level_* signature — or normalized value_text for qualitative facts. Never the quote.

Same theme as the collision hash: two sentences, one number. The quote is the last display fallback, never the identity.

§9
Point-in-time is strict
LEAKYdate <= as_of, and "day" means whatever the server thinks
STRICTdate < as_of. "Day" = Eastern Time. Realized returns are never exposed.

A <= would let a fact published at the cutoff instant into a backtest — that's lookahead, and it's the bug that makes a strategy look brilliant and lose money.

OD-14
Guidance movement is derived, never written back
BEFOREcompute "raised" and store it on the fact
AFTERstated movement is stored as said; a bare guide stores unknown and the read layer derives effective_driver_state

Derived mechanically (the midpoint rule) from the prior collapsed value in the same series — never across years, quarters, series units, or slices. No prior → introduced. Not safely comparable → unknown. Self-healing: a late fact re-derives the whole timeline at the next read.

OD-14
A typo fix is not a raise
BEFOREan amendment corrects $1.5B → $1.6B → the derive reads it as "raised"
AFTERa correction with no business-change wording has effective_driver_state FORCED to unknown

Beautiful bug, deliberately killed: the mechanical derive can't tell a fix from a decision — so corrections are excluded by identification, not by cleverness.

§9
Withdrawal fan-out — the ONE derived write
LOOSE"we're suspending guidance" → withdraw everything that looks covered
BOUNDEDALL must hold: clearly STATES a withdrawal · EXACT scope · only OPEN guides · containment via the RESOLVED period, never loose text

"Suspending guidance practices" is a policy remark — not a withdrawal. Scope unclear → don't fan out. A guide the same source replaces is excluded per-guide, not a veto on the rest. Add-only, never deletes; a late covered guide gets its withdrawn fact under the same conditions.

§9
Every result says which kind it is
RAWwhat the sources literally said
RECONCILEDcontinuity chains applied — disableable, deterministic PIT edges only, per-hop cutoff

A reconciled view is an interpretation, so it's labelled and switchable. narrowed is read-derived from consecutive closed widths — never stored.
