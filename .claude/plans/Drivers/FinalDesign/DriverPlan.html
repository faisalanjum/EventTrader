The mental model

A driver is an index card. 01_Overview

Like one idea per slip in a Zettelkasten: each driver is one atomic cause on its own card — specific, uniquely named, reusable. One card = one meaning.

Driver = a CLASS

The reusable cause (same_store_sales, oil_price). Coined once. Backed by a real quote.

In code: a class definition.

Driver-update = an INSTANCE

One occurrence of that cause in one event (one earnings call, one 8-K). Written every time it happens.

In code: one stamped object of the class.

The promise: the same cause always gets the same name everywhere. Scattered mentions line up into one clean history per cause — across companies and over time. That clean history is the tradeable signal.

The edge over RavenPack / Bigdata: they list that an event happened. This grades each cause against what the stock actually did. Don't drift toward "just list events."

The two deaths that shaped every rule
  • v1 — the closed word-list. Validated names against a fixed vocabulary → rejected 82% of even-correct names → died. Lesson locked in NAME-03 (open vocabulary) + NAME-19 (principles, never examples).
  • v2 — the eager merger. Merged too aggressively, collapsed 3 demand stories into one generic name, failed a fresh exam → died. Lesson locked in NAME-04 (specific by default; breadth emerges).
  • v3 — this design. The AI judges meaning; code checks only structure; every ambiguous call leans specific.

The safety engine

Why the AI can be wrong and the graph is still correct. FS-04 · PIPE-04

The whole design hangs on one asymmetry: the AI may only propose; only code may merge. So every AI mistake becomes a recoverable over-split — never a permanent over-merge.

🤖 The AI judges MEANING

  • assign to an existing value
  • coin a new value
  • flag / mark provisional

Never merges two existing identities.

⚙️ Code checks STRUCTURE

  • merges only on exact / confident match
  • deletes only from frozen lists
  • hard-fails on anything malformed

Never guesses meaning.

The payoff (the proof, not a slogan): because the AI can only over-split, and the one law says over-split is a cheap fix, the graph is correct from day one no matter how wrong the model is. That single asymmetry is the spine everything else hangs on.

One advertised exception where over-split is NOT free: write-once period dates (ISS-42) — a bad first write persists.

The gate (NAME-18)

Coin a driver only if it's a genuinely reusable class, grounded in the source, unambiguous. Vague text → skip, never invent.

The name-vs-slice role rule (NAME-11) + the 13-item banned list

Use only the quote + source company: own measured part → slice; external object/actor/platform/policy/event/product causing the outcome → name; unclear role → name. Examples: Apple iPhone sales → sales + product:iphone; Qualcomm hurt by iPhone demand → iphone_demand.

The payoff

The verdict is an edge, with three independent axes. 07 · DU-21…24

Not a node — the property-laden EXPLAINED_BY edge (Event → fact), added only when a producer blames the move on that fact.

stock_impact

{long, short} — the driver's direction of push. Can oppose the net move.

weightage

{0.1…1.0} or null — an independent force. Never a share, never summed to 100%. A move may be partly unexplained.

confidence

{0…100} — how sure the attribution is TRUE. Orthogonal to weightage.

PIT / no look-ahead: the share-of-move is computed at read time, never stored; the realized return is never stored on the verdict and never shown to the predictor. Grading is aggregate (net sign/magnitude + ranking), never a per-driver "true share".

The macro/news path — DailyCompanyMoveEvent

A significant move with no filing event attaches the verdict to a thin DailyCompanyMoveEvent (dcm:<cik>:<trade_date>, its own label, not an :Event). Returns are read from the existing Date-HAS_PRICE-Company, never duplicated. On filing days the filing Event is the single attribution target. open: significance threshold · pure-macro source · shared-fact id (ISS-11)

The hardest rule · just locked

Beat / miss routing — a three-way split. 12 §10.5 · ISS-16

"Compared to an expectation" is the trigger — not a "beat" word. The state is derived from the two stated numbers.

The comparison is…Routes toWhy
Forward guide vs old guide ("now expect $X, was $Y")guidance lane (raised/lowered)the tense trap — it's a revision, not a surprise

The beat/miss arithmetic inverts for lower-is-better metrics — cost, capex, tax rate, churn. "Opex below guidance, beating expectations" → word says beat, number is below range → the directional-conflict check would hard-fail a correct fact. The north star · the real ambition

A living, self-driving Driver graph. 14 §7 · aspirational

The end goal isn't a catalog — it's a system where every new asset (report, transcript, filing, news) is automatically and near-instantly turned into the right Drivers + DriverUpdates. No human in the loop, by construction.

100%

recall — a miss is an unacceptable design failure

100%

precision — a false write is just as unacceptable

Lowest

practical cost — deterministic code first, cheap models, strong only when needed

How it should work

  • Live trigger → daemon/worker reuse → run ledger → reuse/create Driver → write fact → write verdict
  • Advisor model strategy: a cheap model escalates to a strong one only when needed
  • Staged backfill — upcoming-earnings companies first, then high-value, then full history

Hard constraints

  • No steady-state human review · parks/fails minimized to near-none, never silent
  • Claude Code subscription / workflow path — not the metered SDK (unless owner-approved)
  • Minimalist machinery · aim for 100% reliability — no complexity without a clear win

The graph should become intelligent through its structure — Drivers, DriverUpdates, sources, companies, events, periods, concepts, members, verdicts, and time connected so the system gets more useful with every asset that arrives. Powerful and trading-actionable before any predictor/learner loop even runs.

Why it's worth it

How a connected graph is tradeable — before any prediction. 14 §7

Even with no ML model, a well-connected Driver graph is directly actionable. The structure itself is the intelligence.

What the structure gives you

  • Clean per-cause history — group by the full series key (driver · fact_type · slice · period · measurement · unit · time_type) → one clean time-series per cause, per company.
  • Graded verdicts — EXPLAINED_BY says which causes actually moved the stock, direction + force, checked vs the realized return.
  • Cross-company breadth — the same driver name across companies → instant sector / theme exposure maps.

What you can do with it

  • Query, don't predict — "which drivers changed materially this week AND historically moved the stock the same way?" → a candidate buy/sell, no model.
  • Answer in one hop — "what's driving this move · who else is exposed" is a graph query, not a research project.

Honest gap: the change-flag scanner that turns "a cause moved" into an automatic buy/sell alert is still to design (ISS-8 · 90 §D). The graph enables it; the trigger layer is future work.
