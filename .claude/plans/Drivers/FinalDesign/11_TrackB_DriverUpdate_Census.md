## §1 Mission constraints & laws (what the structure must guarantee)

- **T1.1 One stated fact = one node.** Which cause (`driver`) · stated where (`event`) · which version (`fact_scope`) · what happened (`driver_state`) · the stated numbers/words · the proof (`quote`). Everything else = an edge or a read-time derivation. `[09 §1]`
- **T1.2 The one law.** Over-merge = permanent damage; over-split = cheap fix; unsure → separate. The LLM assigns / coins / flags — never merges two existing identities; code merges exact/confident only, deletes only from frozen lists, validates structure with hard-fails. `[01 · FS-04 · 99 §1]`
- **T1.3 Store-when-stated.** Numbers, baselines, comparisons: only what the source states — never computed, never fabricated, never derived from another node. The explicit code-derived exceptions: implied reported periods (PER-06) and the blanket-withdrawal fan-out (§11.7). `[DU-13/16 · 09 §4 bottom row]`
- **T1.4 Producer-free convergence.** The fact id excludes the producer, so ANY writer of the same fact converges to one node (re-runs MERGE in place, never fork); disagreement is isolated to verdict edges, whose key includes the producer. `[FS-01 · DU-19/22 · 99 §3.2]`
- **T1.5 PIT wherever a fact is written.** Write-time menus (slice + concept) cut at ≤ event/public-source time; the realized return is never stored on a verdict and never shown to the predictor. Name-creation is PIT-exempt; read/offline-repair sees all history. `[FS-14 · XC-09 · DU-23 · PIPE-34]`
- **T1.6 Enrichment is never identity.** XBRL concept/member links are recoverable bonuses on the FACT (never the Driver class); the text identity works without them; a WRONG link is the only real failure — the design abstains rather than risk one. `[FS-21 · XC-01 · PIPE-29]`
- **T1.7 Provenance ≠ attribution.** `FROM_SOURCE` (where the fact came from — always present) is never collapsed with `EXPLAINED_BY` (what the stock move is blamed on — optional). A fact can be reported without explaining the move. `[DU-20]`
- **T1.8 Class vs fact (the scoping law).** Everything company/value/time-specific — slice VALUES, measurement, periods, units, numbers, states, verdicts, XBRL links, PIT menus — lives on the DriverUpdate fact, written at fact time; the catalog build never writes a DriverUpdate. `[PIPE-03 · DU-02]`

### §2b Value-shape grammar — self-describing, no shape field `[09 §3 · approved 2026-07-03 · 95 #16]`

- **point** = `level_low == level_high` (BOTH set) · **range** = low < high · **floor** ("at least X") = low only · **ceiling** ("up to X") = high only · **numberless** = all null. **Hard writer rule: never write a point as low-only; low-only ALWAYS means floor, high-only ALWAYS means ceiling.**
- A shape is **closed** when both bands are present (point / closed range). Deltas, widths, and midpoints derive ONLY from closed shapes.
- `comparison_low/high` use the SAME grammar — a single-value baseline sets BOTH (the $1.20 consensus = 1.20/1.20); floor/ceiling baselines are expressible ("previously guided at least $85M").
- **Transient hints (never stored):** `level_shape_hint` accompanies level numbers, `comparison_shape_hint` accompanies comparison numbers, each ∈ {point, range, floor, ceiling, none}; the writer derives the shape from the slots, hard-fails on mismatch, discards the hint. **Hints are REQUIRED whenever the corresponding numbers are present — a MISSING hint hard-fails exactly like a mismatch; numberless facts OMIT the hints (§14.7, owner-approved 2026-07-03).** Catches both slip directions (forgotten-high point→floor AND forgotten-label floor→point). Production-proven encoding (points stored low=high for 3 years; 6 malformed rows in 8,432).

## §3 Identity — id + fact_scope grammar

- **T3.1** Fact id = **event + driver + fact_scope** — NO producer (two readers of one fact converge); code builds it. `[FS-01]`
- **T3.2** fact_scope serialization: named slots in fixed order — `period=<period_u_id>` · `slice=<kind:value[;kind:value…]>` · `measurement=<tok[,tok…]>` · (`quote_hash=<hash>` tie-break only). Absent parts OMIT their slot (never default slice to `total`, never pad with a hash). `[FS-02 · 09 §3]`
- **T3.3** Identity normalization = **FORMAT only** (case/whitespace/punctuation) on slice AND measurement values. Per-company alias files that unify drift ("First Aid" × 5 spellings) are READ-time views + member matching — **never inside the id** (ids are immutable; alias growth must not change them). **NOTE (owner 2026-07-03, §14.4): the ONLY approved drift-recovery is member-anchored read-time grouping (T12.9) — deterministic, no humans, ids never change; everything without a clean shared member pair stays split.** `[FS-02 · 09 §3]`
- **T3.4** `quote_hash`: only when two genuinely DIFFERENT same-event facts still collide after fusion (§11.3); deterministic, from normalized quote + normalized value signature; EVERY fact in the collision set gets one (none keeps the bare id); restatements (same value) MERGE instead — never split on quote wording alone. `[FS-03 · 99 §3.3]`
- **T3.5** measurement = multi-label, code-sorted set, comma-joined in its slot (`measurement=adjusted,constant_currency` for source-separated spans); the SPECIFIC stated word, no stemming, no meaning-merge at write; novel words kept verbatim (format-normalized); default EMPTY — never assume gaap; gaap/non_gaap = a read-time grouping view, never stored. **OD-9 (66 §0.R OD-9): producer copies exact source spans → transient `measurement_raw_spans`; code alone normalizes (lowercase → each non-alphanumeric run → `_` → trim → collapse); MAXIMAL contiguous spans → one token ("adjusted diluted" → `adjusted_diluted`, not two) — separate tokens only for non-contiguous spans; measurement = never-drop sink (a qualifier leaves only on lossless capture elsewhere; unsure → keep); no write-time aliasing.** `[FS-25 · OD-9]`
- **T3.6** slice = `kind:value`; kind ∈ 6, each defined by an operate/sell test — **segment** (operates AS it, incl. brands: `taco_bell`) · **product** (SELLS it: `iphone`, `aws`) · **geography** (operates IN it: `china`) · **customer** (SELLS TO it: `major_customer_a`) · **channel** (HOW it sells/runs: `franchised`; **store_type folds into channel**, never its own kind — 95 #11) · **entity_ownership** (a stake it OWNS, JV/equity-method: `jv_x`) — plus `unknown` (safe 7th); value free-text, lowercase, punctuation-stripped; multi-part = code-sorted, ';'-joined, never drop a part; whole-company / consolidated / total-company / no stated segment → OMIT slice (display "Total" for metric/guidance/surprise only; for action_event, omitted means no slice applies or no narrower business part is stated); period is NOT a slice; "brand" is not a kind (a brand's kind comes from its axis or the producer pick). **The slice test:** a value is a slice ONLY if "revenue/earnings from ___" names a real business population — accounting labels (Level 2 fair value, Term Loan, RSUs, reconciling items) are NOT slices. `[FS-05…10 · 95 #11/#25]`
- **T3.7** The period appears BOTH as the `HAS_PERIOD` edge AND inside fact_scope — the SAME `period_u_id`, validated in both directions; never raw Q1/FY2025 as the identity token. `[PER-12 · 99 §3.9]`
- **T3.8** A slice value, once written, is IMMUTABLE (first-written wins; everything attaches to it). `[FS-17]`

## §6 Lanes — fact_type × driver_state × field matrix

- **T6.1** `fact_type` lives on the Driver (stamped once at catalog finalization, permanent): **metric** (standing variable or condition, numeric OR qualitative, re-readable) · **guidance** (company's own forward outlook) · **surprise** (actual vs EXPECTATION — consensus or own prior guide; never vs prior-period actual) · **action_event** (a discrete happening). Persistence test decides metric vs action; suffix framing overrides; outlook verbs → guidance; dual framing legal (`dividend` action vs `dividend_per_share` metric); bare-root defaults (applied at the Track A stamp, carried for zero-loss): litigation / convertible_notes / dividend_policy / restructuring_costs → metric · corporate_restructuring / asset_impairment → action_event. The 4 definitions are kept EXACTLY — add NO extra clauses or examples (a tested extra clause overfit: fixed 2, broke 1). `[DU-05/06/07 · MF-01]`
- **T6.2** State lanes (exact vocab; the quote carries precision) `[DU-09/10/11 · 99 §3.12]`:
  - **metric**: `increased, decreased, unchanged, mixed, reported, persists, unknown` — pick-FIRST-match ladder: direction stated → increased/decreased ("weather worsened" = increased, MORE of it) · same driver up+down in different parts → mixed (two DIFFERENT drivers opposite = split, not mixed) · explicit flat → unchanged · ongoing no direction → persists · bare value, no comparison → reported (a prior value present alongside it → increased/decreased) · else unknown. Good/bad NEVER decides state (that's the verdict).
  - **guidance**: `introduced, raised, lowered, reaffirmed, withdrawn, unknown` (vs the company's own prior guide).
  - **surprise**: `beat, in_line, missed, unknown` (vs the expectation).
  - **action_event**: `at_risk, announced, occurred, continued, resolved, canceled, suspended, rumored, failed, unknown` — decision ladder: **Step 0** classify the LATEST stage of one action; **Step 1** finality gate — TERMINAL: failed = ended involuntarily (outside party blocked / external condition unmet; **incl. declining an offer never committed to**) · canceled = own free-choice withdrawal · resolved = two-sided dispute settled · occurred = completed; NOT-TERMINAL: rumored = third-party-reported, company-unconfirmed (a denial keeps it rumored) · at_risk = a specific CURRENT source-flagged adverse not-yet-happened event that is **NOT the company's own plan** — STRICT (generic boilerplate gate-dropped) · suspended = paused/resumable · announced = the company's OWN action stated pre-completion · continued = still ongoing. Conventions: shelve/postpone → suspended · scrap/abandon/withdraw → canceled · threat → at_risk until executed → failed.

## §7 Sub-system: DriverPeriod (the time window)

- **T7.1** DriverPeriod = ONLY "the calendar window this fact is about" — not the event/filing date, not free-text "Q1", not forecast-vs-actual (fact_type's job). One generic node for all 4 fact-types. Node label `DriverPeriod` (not Period — XBRL collision; not GuidancePeriod); ids keep the `gp_` prefix (proven code returns it; no migration). `[PER-01/02/03]`
- **T7.2** ONE edge `HAS_PERIOD`; meaning comes from fact_type (guidance → future target window · metric → reported window · surprise → actual-vs-expectation window · action_event → rare stated action window). `[PER-04]`
- **T7.3** By fact-type: guidance REQUIRED (both company_confirmed values); metric/surprise when stated, source-implied, or code-derivable; action_event rare/optional. NEVER force a period — no real window, no edge. **Owner pin (2026-07-03, 12 §10.7): guidance's requirement = a real resolved period OR an explicit sentinel; non-guidance facts with truly no period stay edge-less.** `[PER-05]`
- **T7.4** Implied reported periods: derive from event/report metadata ONLY when the source gives exactly ONE clear period (a Q1 10-Q's "revenue grew 12%" → that Q1 window); never guess from vague wording; ambiguous market/news → no period unless dates stated; prefer XBRL/SEC exact dates for 10-Q/10-K. `[PER-06]`
