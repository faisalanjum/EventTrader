## §4 `driver_writer.py` + `driver_write_cli.py` — the write path

- **FACT-14 — Node write semantics.** `MERGE (du:DriverUpdate {id: $id}) ON CREATE SET du.created = $ts`, then a per-field SET of the 23-field payload — with TWO changes from the substrate: **(a) change detection** — before writing, fetch the existing node's producer-written fields and diff (the 09 §7 no-op rule); a no-op re-run skips the SET entirely (log `unchanged`); a real change logs the changed-field list. **(b) NO-NULL-CLOBBER merge (✅ APPROVED owner 2026-07-03, §10.8).** The substrate's `SET x = null` REMOVES properties (verified footgun) and full-SET last-writer-wins means a WEAKER re-run of the same event (extracts `level_low`, misses `comparison_low`) silently ERASES a richer prior fact — colliding with ISS-62's "under-extraction accepted at write." Recommended rule: on a normal write the SET clause is built **per field** — a field is written when the new value is non-null OR the stored value is already null; a **non-null stored field is NEVER overwritten with null**. So: fill-empty-slots + update-changed-non-nulls always apply; a real correction (value genuinely changed, non-null→non-null) always writes; only a null-clobber is blocked. Why safe: within ONE id the event is fixed (same `source_id`), so a field going null on a re-read is ~always under-extraction, not a real retraction (a genuine cross-event restatement lands on a DIFFERENT id and collapses at read-time, not write). **Escape hatch:** an explicit `--repair` mode (or an item flag) permits deliberate clears — the only path that may null a non-null field. This makes the dual-producer probe's "presence disagreement" benign (the richer extraction wins by construction) and closes erase-on-rewrite without a completeness heuristic. The lane matrix still runs BEFORE any SET, so a lane-legal null on a FIRST write (numberless fact) is written normally; the no-clobber rule only governs re-writes over existing non-nulls.
- **FACT-15 — Edges.** In the atomic query: `MERGE (du)-[:OF_DRIVER]->(d:Driver {name})` (target MUST have fact_type — no fact_type ⇒ REJECT before write, DU-12) · `MERGE (du)-[:FROM_SOURCE]->(source)` via the existing `SOURCE_LABEL_MAP` {8k/10q/10k→Report, transcript→Transcript, news→News} · `MERGE (du)-[:HAS_PERIOD]->(dp:DriverPeriod)` only when a period resolved. **DROPPED vs substrate:** `FOR_COMPANY` (company via Event → Company, 09 §3) and `UPDATES` (replaced by OF_DRIVER). **KEPT patterns:** concept edge (`MATCH… WITH…LIMIT 1 MERGE MAPS_TO_CONCEPT`, best-effort/non-blocking/self-healing, un-gated on re-runs) · member edge UNWIND — **extended with the part anchor**: `MERGE (du)-[r:MAPS_TO_MEMBER]->(m) SET r.slice_part = $part` (census §14.4 build note; the part = the `kind:value` token the link anchors). `date` stays writer-authoritative from the source node (the substrate's given_date Cypher CASE — transcript `conference_datetime`, else `created` — reused verbatim; full ISO timestamp).
