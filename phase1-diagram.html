Recovery (on daemon restart)

One rule: Recovery uses the same canonical handlers as normal monitoring (Sections 17 & 18). No separate recovery decision tree. Same broker reads, same branching, same terminal-write rules.

plan.json status = entering

pending_terminal_reason set + position → flatten & close

pending_terminal_reason set + no position → close

Position + stop active → promote to open

Position + no stop → FLATTEN (stop_repair_failed)

No position + deadline expired → entry_timeout / entry_unresolved

No position + deadline pending → keep entering

plan.json status = open

pending_terminal_reason set + position → flatten & close

No position → stop_fired

Position + stop active → resume monitoring

Position + no stop → FLATTEN (stop_repair_failed)

Safety invariant: There is never a live position without a broker-native stop. If stop is missing → flatten immediately. Broker stop protects even when daemon is down.

Write-Once Rule

Terminal files are write-once

trade/result.json and trade/skipped.json — if the file already exists, the daemon must NOT rewrite it.

All terminal branches use write_result_if_missing() or write_skipped_if_missing().

This prevents crash-recovery from overwriting a result that already captured real fill prices with a recovery result that has null fill prices.

Retry paths remain idempotent: they may cancel lingering orders, they may update plan.json to closed, but they must not create a second terminal record.

Signal-Agnostic Architecture

Canonical Signal Contract

TES reads only these fields from prediction/result.json:

Required: signal_id, source_type, symbol event_id, event_time, issued_at direction, confidence, expected_move_range Optional: thesis_summary, context_ref, session_id

Producer emits canonical fields directly.
TES never translates field names.

Matching by con_id

Before entry: qualify contract → persist con_id

All position/order matching uses con_id.
symbol is for logging only.

Position match: Position.con_id == plan.con_id Stop match: plan.stop_order_id == order.order_id Fallback: order.con_id + order_type == "STP"

Broker-state invariant: unmatched positions/orders block new entries.
